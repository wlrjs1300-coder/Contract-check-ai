## 프로젝트 안전 원칙

- 실제 개인정보를 사용하지 않는다.
- 실제 계약서 원문을 사용하지 않는다.
- 외부 AI API를 호출하지 않는다.
- 비용이 발생하는 기능을 만들지 않는다.
- raw input과 raw external response는 Git에 저장하지 않는다.
- 생성된 대량 합성 데이터는 Git에 저장하지 않는다.
- 스파이크 결과는 제품 품질 보장이 아니라 관찰 결과로만 사용한다.

## 합성 데이터 정책

- Git 추적 fixture는 `data/fixtures/*.sample.txt`에 둔다.
- 생성 데이터는 `data/generated/`에 생성하며 Git에서 제외한다.
- fixture와 generated data는 모두 완전 합성 데이터여야 한다.
- 민감 식별번호 형식은 이번 PR에서 사용하지 않는다.

## 관찰 지표와 A/B/C/D 판정 원칙

| 등급 | 의미 | 후속 판단 |
|---|---|---|
| A | MVP 설계에 활용 가능한 수준 | 상세 설계 진행 |
| B | 제한 사항을 기록하면 활용 가능 | 제약 포함 설계 |
| C | 불안정하거나 보완 필요 | 범위 축소 또는 보완 |
| D | MVP 적용 부적합 | 접근 재검토 |

A/B/C/D는 스파이크 의사결정 보조 기준이며 법률 안전성, 제품 품질, 사용자 위험도 점수가 아니다.
