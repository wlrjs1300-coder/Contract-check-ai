## 확정할 Git 운영 흐름

- 개별 작업 브랜치는 `develop`에서 생성한다.
- 작업 브랜치에서 커밋 후 원격 브랜치로 push한다.
- 작업 브랜치는 `develop`으로 PR을 생성한다.
- `develop`에서 여러 작업을 누적하고 검토한다.
- 큰 단계가 안정적으로 마무리되면 `develop`에서 `main`으로 PR을 생성한다.
- `main` 병합 후에만 태그를 생성한다.
- `main`은 안정 버전 기록용으로 유지한다.

## 결과

### 변경 파일

- `docs/02-git-rules.md`
- `docs/checklists/v0.0/v0.0.2-repository-rules.md`

### 확정된 브랜치 전략

- `main`: 안정 버전 기록용 브랜치
- `develop`: 작업 누적 및 통합 검토용 브랜치
- `docs/*`: 문서 작업 브랜치
- `feature/*`: 기능 구현 브랜치
- `fix/*`: 버그 수정 브랜치
- `chore/*`: 설정/환경 작업 브랜치
- `security/*`: 보안 관련 작업 브랜치
- `release/*`: 릴리스 준비 브랜치

개별 작업 브랜치는 `develop`에서 생성하고, 작업 완료 후 `develop`으로 PR을 생성한다.

### 확정된 PR 흐름

1. `develop` 최신화
2. `develop`에서 작업 브랜치 생성
3. 작업 브랜치에서 변경 작업 및 커밋
4. 원격 작업 브랜치로 push
5. 작업 브랜치에서 `develop`으로 PR 생성
6. PR 검토 후 `develop` 병합
7. 로컬 `develop` 최신화
8. 큰 단계가 안정적으로 마무리되면 `develop`에서 `main`으로 PR 생성
9. PR 검토 후 `main` 병합

### 확정된 태그 기준

- 태그는 `main`에 병합된 안정 버전에만 생성한다.
- 작업 브랜치나 `develop` 브랜치 기준으로 태그를 생성하지 않는다.
- 태그 형식은 `v<major>.<minor>.<patch>`를 사용한다.
- 태그는 annotated tag를 사용한다.

### 다음 단계 이동 가능 여부

- PR 병합 후 가능
- 이유: 저장소/Git 운영 기준, develop 중심 작업 흐름, PR 흐름, 태그 생성 기준은 정리되었으며, 커밋과 PR 병합 후 v0.0.3 단계로 이동할 수 있다.

### 검증 결과

- 기준 문서와 PR 템플릿을 확인했다.
- `docs/02-git-rules.md`를 develop 중심 운영 흐름에 맞게 수정했다.
- 체크리스트와 실제 Git 운영 기준 문서의 불일치를 해소했다.
- 공개 저장소에 남겨도 되는 운영 기준만 포함했다.
- 민감정보, 원본 계약서, 작업 중간 산출물은 포함하지 않았다.

### 남은 이슈

- 커밋, PR, main 병합, 태그 생성은 아직 진행하지 않았다.
