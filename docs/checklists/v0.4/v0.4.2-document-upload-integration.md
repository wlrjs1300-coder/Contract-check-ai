## 실제 API 계약

- 요청: `POST /documents/upload`
- body: `multipart/form-data`, field name `file`
- `Content-Type`은 브라우저가 boundary와 함께 설정하며 프론트에서 직접 지정하지 않는다.
- API base URL은 `VITE_API_BASE_URL`, 기본값은 `http://localhost:8000`이다.
- 성공 응답은 문서 식별자·파일 메타데이터·상태·조항·미분류 영역·warning을 포함한다.
- 조항 `title`은 `string | null`, 미분류 영역은 현재 splitter와 DB 모델 기준 `string[]`이다.
- `source_hash`는 응답 타입에는 포함하지만 사용자 화면에는 표시하지 않는다.
- 숫자 필드는 정수와 범위를 확인하고 `clause_count`는 조항 배열 길이와 일치해야 한다.

## 클라이언트 사전 검증

- [x] 파일 미선택 차단
- [x] `.txt` 확장자를 대소문자 구분 없이 허용
- [x] 빈 파일 차단
- [x] `1 * 1024 * 1024` bytes 초과 차단
- [x] 정확히 1 MiB인 파일 허용

사전 검증은 사용자 경험을 위한 보조 검증이며 UTF-8을 포함한 최종 입력 검증은 백엔드 응답을 기준으로 한다.

## 업로드 상태

- 초기, 파일 선택, 업로드 중, 성공, 실패 상태를 구분한다.
- 업로드 중 input과 버튼을 비활성화하고 `aria-busy`, live region과 텍스트 상태를 제공한다.
- 새 파일 선택 시 이전 성공 결과와 오류를 초기화한다.
- 실패 후 같은 파일로 재시도할 수 있다.
- `fetch`에서 실제 진행률을 받지 않으므로 퍼센트를 표시하지 않는다.

## 오류 매핑

| 서버 detail 또는 상황 | 사용자 메시지 |
|---|---|
| `Only .txt files are allowed.` | TXT 파일만 업로드할 수 있습니다. |
| `The uploaded file is empty.` | 내용이 없는 파일은 업로드할 수 없습니다. |
| `The uploaded file exceeds the 1 MB limit.` | 파일 크기는 1MB 이하여야 합니다. |
| `The uploaded file must be UTF-8 encoded.` | UTF-8 형식의 TXT 파일을 선택해 주세요. |
| 네트워크 실패 | 서버에 연결하지 못했습니다. 잠시 후 다시 시도해 주세요. |
| 그 밖의 HTTP·응답 형식 오류 | 업로드를 완료하지 못했습니다. 파일을 확인한 뒤 다시 시도해 주세요. |

내부 예외나 응답 전체를 사용자에게 노출하거나 console에 출력하지 않는다.

## 결과 표시

### 문서 메타데이터

- 파일명, 파일 크기, 문자 수, 문서 처리 상태와 분리된 조항 수를 표시한다.
- 백엔드 status를 근거 없이 법적 또는 품질 의미로 재해석하지 않는다.

### 조항 목록

- `ordinal` 순으로 marker, title, body와 clause type을 표시한다.
- `reference_id`를 안정적인 React key로 사용한다.
- 빈 title과 body를 안전하게 처리하고 본문 줄바꿈을 유지한다.
- 조항 0개는 성공 응답의 별도 빈 상태로 표시한다.

### warning과 미분류 영역

- 문서 warning은 문서 요약에서 처리 확인 정보로 표시한다.
- 조항 warning은 해당 조항 안에서만 표시한다.
- warning이 없으면 빈 경고 영역을 렌더링하지 않는다.
- 미분류 문자열은 별도 영역에 원문 줄바꿈을 유지해 표시한다.
- 미분류 영역을 위험 또는 오류로 표현하지 않는다.

## 알려진 한계

- 현재 기술 검증 범위는 UTF-8 TXT 한 파일이며 PDF와 OCR을 지원하지 않는다.
- 브라우저에서는 복잡한 UTF-8 사전 판정을 하지 않으며 백엔드 검증을 최종 기준으로 사용한다.
- 현재 화면은 업로드와 조항 분리 결과까지만 제공하며 분석 결과를 제공하지 않는다.
- warning 코드는 알려진 값만 한국어로 설명하고 알 수 없는 값은 정보 손실 없이 표시한다.
- 업로드한 문서는 현재 백엔드 SQLite에 저장되며 사용자별 접근 제어가 없다.
