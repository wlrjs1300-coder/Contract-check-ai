## 결과 조회 API 계약

- 요청: `GET /documents/{document_id}/analysis-results`
- 요청 body, `Content-Type`, Authorization header: 없음
- `document_id`는 URL path segment로 인코딩한다.
- 최상위 필드: `document_id`, `job_id`, `status`, `items`
- item 필드: `clause_id`, `reference_id`, `display_label`, `summary`, `expert_review_recommended`
- 허용 상태: `queued`, `processing`, `completed`, `failed`
- 허용 라벨: `안전`, `주의`, `위험`, `추가 확인`
- 문서가 없으면 `404 Document not found.`이다.
- 최신 분석 작업이나 결과가 없으면 `404 Analysis result not found.`이다.

API는 최신 분석 작업을 반환하며 completed가 아닌 작업도 같은 구조로 반환할 수 있다. 프론트는 현재 completed 작업과 응답의 문서·작업 ID가 모두 일치하고 응답 status도 completed일 때만 결과를 정상 표시한다.

## 조항·결과 연결 정책

- 업로드 조항과 결과의 `clause_id`와 `reference_id`를 모두 비교한다.
- 응답의 중복 `clause_id` 또는 `reference_id`를 거부한다.
- 결과 개수와 업로드 조항 개수가 다르면 전체 연결 오류로 처리한다.
- 모든 조항이 정확히 한 결과와 연결되어야 한다.
- 표시 순서는 API 순서가 아니라 조항 `ordinal` 순서를 따른다.
- 식별자를 새로 만들거나 사용자 화면에 노출하지 않는다.

파이프라인은 조항 처리 중 오류가 발생하면 transaction을 rollback하고 부분 결과를 남기지 않는다. v0.4.3은 조항이 없는 문서의 분석 시작을 차단하므로, 현재 정상 completed 흐름에서 빈 items나 일부 조항 누락은 잘못된 응답으로 처리한다.

## 결과 표시 계약

- 라벨 원문을 바꾸거나 재분류하지 않는다.
- 색상과 함께 라벨 텍스트를 항상 표시한다.
- 위험 점수, 퍼센트와 법적 확정 표현을 만들지 않는다.
- summary는 React 텍스트로 렌더링하고 HTML로 해석하지 않는다.
- `true`는 `전문가 검토를 권고합니다.`로 표시한다.
- `false`는 `별도 권고가 표시되지 않았습니다.`로 표시하며 안전 보장으로 표현하지 않는다.
- 조항 순서, marker, title, clause type을 함께 표시하고 본문은 중복하지 않는다.

## 조회 시점과 상태

- 분석 작업이 completed일 때만 `분석 결과 보기` 버튼을 제공한다.
- 조회 전, 조회 중, 성공, 오류 상태를 구분한다.
- 조회 중에는 결과 버튼, 파일 선택, 업로드, 분석 시작과 상태 조회를 비활성화한다.
- 자동 조회나 자동 폴링을 사용하지 않는다.
- 새 파일 선택, 새 업로드, 새 분석 시작, queued·processing·failed 전환 시 이전 결과와 오류를 초기화한다.

## 오류 처리

- 네트워크: `서버에 연결하지 못했습니다. 잠시 후 다시 시도해 주세요.`
- HTTP·응답 형식: `분석 결과를 불러오지 못했습니다. 잠시 후 다시 시도해 주세요.`
- 결과 없음 404: `저장된 분석 결과를 찾지 못했습니다.`
- 조항 연결 불일치: `분석 결과와 문서 조항을 연결하지 못했습니다. 다시 분석해 주세요.`
- 서버 detail, stack trace, 내부 식별자와 hash를 사용자에게 노출하지 않는다.

## 합성 Provider 및 법률 경계

현재 기본 Provider는 `SyntheticAnalysisProvider`다. 결과 화면은 화면·API 흐름 검증용이며 실제 외부 분석 품질, 정확도, 법률 검토 완료나 계약서 안전을 보장하지 않는다. `안전` 라벨과 전문가 권고가 없다는 사실도 법적 안전이나 검토 불필요를 의미하지 않는다.
