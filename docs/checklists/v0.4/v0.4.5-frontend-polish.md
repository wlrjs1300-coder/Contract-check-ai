## 오류 메시지와 재시도 UX

- 업로드 오류는 파일 형식·크기 확인 또는 잠시 후 재시도를 안내한다.
- 분석 요청 오류는 같은 문서에서 `분석 시작`을 다시 누를 수 있음을 안내한다.
- 상태 조회 오류는 기존 작업 상태가 유지됨을 알리고 `상태 다시 확인` 재시도를 허용한다.
- 결과 조회 오류는 `분석 결과 보기` 재시도를 안내한다.
- 네트워크 오류 문구는 `서버에 연결하지 못했습니다. 잠시 후 다시 시도해 주세요.`를 기본으로 한다.
- 서버 내부 오류, 문서·작업 식별자, source hash와 Provider 내부 정보를 노출하지 않는다.
- 오류 후 기존 동작 버튼을 다시 활성화하며 별도 중복 재시도 버튼은 만들지 않는다.

## 빈 상태

- 파일 미선택 상태에는 빈 카드나 상태 alert를 표시하지 않는다.
- warning과 미분류 영역이 없으면 해당 목록을 렌더링하지 않는다.
- 분석 전과 결과 조회 전은 오류로 표시하지 않는다.
- 조항 0개는 문서 형식 확인 안내와 분석 시작 비활성화로 표현한다.
- completed 작업의 결과 0개는 기존 all-or-nothing 계약에 따라 정상 빈 결과가 아닌 연결 오류로 유지한다.

## 접근성

- file input과 label의 명시적 연결을 유지한다.
- 주요 동작은 모두 native button을 사용하며 진행 중 실제 `disabled`를 적용한다.
- 업로드 진행 상태는 `role="status"`, 오류는 `role="alert"`로 전달한다.
- 분석과 결과 panel의 작업에만 `aria-busy`를 적용하고 전체 main에는 잘못된 busy 상태를 두지 않는다.
- 분석 상태 진행 영역과 결과 진행 영역의 `aria-live="polite"`를 유지한다.
- 라벨과 전문가 권고는 색상이나 아이콘만이 아니라 텍스트로 전달한다.
- 포커스 outline을 제거하지 않는다.

코드와 DOM 테스트 기준 검토를 완료했다. 실제 키보드 Tab 순서와 스크린 리더 읽기는 브라우저에서 추가 확인해야 한다.

## 색상 대비와 버튼 일관성

- 결과 라벨은 Bootstrap `text-bg-success`, `text-bg-warning`, `text-bg-danger`, `text-bg-secondary`와 원문 텍스트를 사용한다.
- 위험 점수나 퍼센트 형태를 사용하지 않는다.
- 주요 행동은 primary, 상태 확인과 결과 조회는 outline primary를 유지한다.
- 진행 문구는 `업로드 중…`, `분석 요청 중…`, `상태 확인 중…`, `결과 불러오는 중…` 형식을 유지한다.
- 576px 미만에서는 작업 버튼을 한 열과 전체 너비로 표시한다.

Bootstrap 클래스 기준 검토를 완료했다. 실제 대비 측정과 브라우저별 렌더링은 수동 확인이 필요하다.

## 모바일 반응형과 긴 텍스트

- 최소 body 너비는 320px이며 콘텐츠에 추가 고정 폭을 두지 않는다.
- 576px 미만에서 문서 메타데이터를 한 열로 배치한다.
- 버튼 묶음을 한 열로 전환한다.
- 카드와 flex item에 `min-width: 0`을 적용한다.
- 파일명, marker, 조항 유형, 제목과 warning에 `overflow-wrap: anywhere`를 적용한다.
- 조항 본문, 미분류 텍스트와 summary는 줄바꿈을 보존한다.
- 사용자 텍스트를 자르거나 HTML로 해석하지 않는다.

CSS 규칙과 build 기준 검토를 완료했다. 320·375·576·768px 실제 뷰포트는 브라우저에서 수동 확인해야 한다.

## 포트폴리오 공개 경계

- 상단에서 UTF-8 TXT 한 개, 최대 1MB의 기술 검증 범위를 알린다.
- 상단과 결과 영역에서 합성 Provider 사용과 법률 자문·최종 판단이 아님을 알린다.
- 같은 경계를 분석 상태 카드에서 반복하지 않아 실행 안내와 책임 경계를 구분한다.
- `안전` 라벨과 전문가 권고 없음은 계약서 안전을 보장하지 않는다.
