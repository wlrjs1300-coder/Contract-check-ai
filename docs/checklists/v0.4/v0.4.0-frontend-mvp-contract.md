## 3. 현재 백엔드 기준

1차 프론트엔드 MVP는 장기 기획이 아니라 현재 v0.3.x 백엔드가 실제 제공하는 범위를 기준으로 연결한다.

| 항목 | 현재 계약 |
|---|---|
| 업로드 형식 | `.txt` 파일만 지원 |
| 파일 크기 | 최대 1 MiB (`1 * 1024 * 1024` bytes) |
| 문자 인코딩 | UTF-8 지원, UTF-8 BOM은 `utf-8-sig`로 처리 |
| 업로드 개수 | 한 번에 한 파일 |
| PDF | 미지원 |
| OCR | 미지원 |
| 인증·인가 | 미지원 |
| 사용자별 분석 이력 | 미지원 |
| 사용자별 결과 조회 | 미지원 |
| 기본 Provider | 외부 AI가 아닌 합성 `SyntheticAnalysisProvider` |

프론트엔드는 위 범위를 넘어선 기능이 현재 동작한다고 표시하지 않는다. 파일 선택 단계에서 확장자와 크기를 사전 확인하되, 서버 검증을 대체하지 않는다. 현재 UTF-8 TXT 지원은 파이프라인과 API 연결을 검증하기 위한 기술 검증 MVP 범위이며 최종 제품의 파일 형식을 TXT로 확정한 것이 아니다. 장기 서비스 기획의 텍스트 레이어 PDF 방향은 폐기하지 않고 PDF, OCR 및 계약서 유형 선택을 후속 단계로 둔다.

## 4. 실제 API 계약

| 메서드 | 경로 | 목적 | 프론트엔드 사용 시점 |
|---|---|---|---|
| `GET` | `/health` | 백엔드 연결 상태 확인 | 서비스 진입 시 또는 사용자가 다시 연결을 시도할 때 |
| `POST` | `/documents/upload` | TXT 파일 업로드, 조항 분리 및 문서 저장 | 파일 사전 확인 후 업로드할 때 |
| `GET` | `/documents/{document_id}` | 저장된 문서와 분리된 조항 조회 | 업로드 결과 재확인 또는 화면 상태 복구 시 |
| `POST` | `/documents/{document_id}/analysis-jobs` | 분석 작업 생성 및 현재 파이프라인 실행 | 조항 확인 후 사용자가 분석을 시작할 때 |
| `GET` | `/analysis-jobs/{job_id}` | 분석 작업 상태 조회 | 분석 요청 응답 후 상태를 확인할 때 |
| `GET` | `/documents/{document_id}/analysis-results` | 해당 문서의 최신 분석 작업과 결과 조회 | 분석 완료 확인 후 결과를 표시할 때 |

### 4.1 문서 응답 필드

- `document_id`
- `filename`
- `content_type`
- `size_bytes`
- `character_count`
- `status`
- `clause_count`
- `clauses`
- `unclassified_sections`
- `document_warnings`

`clauses`의 각 조항에는 다음 필드가 포함된다.

- `clause_id`
- `reference_id`
- `source_hash`
- `ordinal`
- `marker`
- `clause_type`
- `title`
- `body`
- `warnings`

조항 표시에는 `clause_id`, `reference_id`, `ordinal`, `marker`, `clause_type`, `title`, `body`, `warnings`를 사용할 수 있다. `source_hash`는 응답에 포함되지만 내부 검증용 성격이 있으므로 사용자 화면의 필수 노출 값으로 두지 않는다.

### 4.2 분석 작업 응답에서 사용할 필드

- `job_id`
- `document_id`
- `status`

### 4.3 분석 결과 응답 필드

분석 결과 최상위 응답에는 다음 필드가 포함된다.

- `document_id`
- `job_id`
- `status`
- `items`

`items`의 각 결과 항목에는 다음 필드가 포함된다.

- `clause_id`
- `reference_id`
- `display_label`
- `summary`
- `expert_review_recommended`

`clause_id`와 `reference_id`는 업로드 결과의 조항과 분석 결과를 연결하는 식별자로 사용한다. 프론트엔드는 식별자를 임의 생성하거나 변경하지 않는다.

## 5. 분석 작업 상태 계약

허용 상태는 다음 네 가지다.

- `queued`: 분석 요청이 생성되어 대기하는 상태
- `processing`: 분석 파이프라인이 실행 중인 상태
- `completed`: 모든 조항 결과가 검증·저장된 상태
- `failed`: 처리 중 오류가 발생해 실패 상태가 저장된 상태

### 현재 동기 실행과 실패 응답 한계

`create_analysis_job`은 작업을 `queued`로 저장한 직후 `run_analysis_pipeline`을 같은 요청 안에서 동기 호출한다. 파이프라인은 `processing`을 거쳐 성공하면 `completed`로 저장한 다음 응답한다. 따라서 성공 시 create API 응답에서 `completed` 상태를 받을 수 있으며, `queued`나 `processing` 응답은 실제로 관찰되지 않을 수 있다. HTTP 요청이 끝날 때까지 사용자에게 요청 진행 중 로딩 상태는 표시하되, 이를 서버의 `processing` 상태와 동일하다고 단정하지 않는다.

실패 경로에서는 파이프라인이 rollback 후 DB의 작업 상태를 `failed`로 저장하지만 예외를 다시 발생시킨다. 현재 코드만 기준으로는 분석 생성 요청이 `{ "job_id": "...", "document_id": "...", "status": "failed" }` 형태의 JSON을 반환한다고 보장할 수 없고 FastAPI 500 계열 응답이나 클라이언트에서 네트워크 오류처럼 보이는 요청 실패로 끝날 수 있다. 또한 실패 응답에서 `job_id`를 전달하는 별도 오류 계약도 없다.

프론트엔드는 HTTP 요청 실패와 API에서 관찰한 `failed` 상태를 별도 분기로 처리한다. create API의 정상 응답 등으로 `job_id`를 확보한 경우에만 `GET /analysis-jobs/{job_id}`로 상태를 재조회한다. `job_id`를 확보하지 못한 실패에서는 상태 조회 경로가 제한되므로 존재하지 않거나 알 수 없는 작업 ID로 재조회하지 않는다. 실패 상세 사유와 안정적인 작업 식별자를 API에 노출하는 것은 후속 백엔드 개선 사항이다.

상태 조회 API는 연결한다. 다만 현재 동기 처리 구조에서는 장시간 폴링을 기본 동작으로 구현하지 않는다. 정상 응답이 `completed`이면 즉시 결과를 조회하고, `job_id`를 확보한 응답에서 `queued` 또는 `processing`이 관찰될 때만 제한적으로 상태를 다시 확인할 수 있다. 무제한 폴링, 고정 장시간 대기, 작업 복구는 이번 계약 범위가 아니다. 향후 비동기 작업 큐를 도입한 뒤 폴링 간격, 최대 횟수, 종료 조건, timeout 및 복구 정책을 별도로 확정한다.

## 6. 결과 라벨 계약

백엔드 schema가 허용하는 라벨은 다음 네 가지다.

- `안전`
- `주의`
- `위험`
- `추가 확인`

표시 원칙:

- 프론트엔드는 `display_label`을 임의 재분류하거나 다른 라벨로 치환하지 않는다.
- 정렬이나 필터링을 위해 별도 위험 점수 또는 퍼센트를 만들지 않는다.
- 색상만으로 의미를 전달하지 않고 라벨 텍스트를 항상 함께 표시한다.
- 색상, 아이콘, 배지는 보조 수단으로만 사용하며 텍스트 대비와 접근성을 확인한다.
- 알 수 없는 라벨을 임의로 `안전` 처리하지 않는다. 계약 외 값은 표시 오류로 다룬다.
- 현재 기본 합성 Provider는 `추가 확인` 라벨과 합성 요약을 반환한다. 이를 실제 외부 AI 분석 결과로 표현하지 않는다.

현재 `DEFAULT_ANALYSIS_PROVIDER`는 `SyntheticAnalysisProvider`다. 따라서 프론트엔드 MVP에 표시되는 결과는 실제 AI 분석 결과가 아니라 합성 결과일 수 있다. 이번 결과 화면 구현은 API 응답 계약과 화면 흐름을 검증하기 위한 것이며 실제 AI 품질, 정확도 또는 법률적 적합성을 검증하는 단계가 아니다. 사용자 화면에 “실제 AI 분석 완료”, “AI가 법률 검토 완료”처럼 오해될 표현을 사용하지 않는다.

## 7. 화면 계약

### 7.1 업로드 화면

| 구분 | 계약 |
|---|---|
| 목적 | 사용자가 현재 지원 범위를 이해하고 분석할 TXT 파일 한 건을 선택·업로드하게 한다. |
| 표시 항목 | 서비스 보조 도구 안내, `.txt`·UTF-8·최대 1 MiB·한 파일 제한, PDF/OCR 미지원, 선택 파일명과 크기, 백엔드 연결 상태 |
| 사용자 동작 | 파일 선택, 선택 취소, 업로드, 연결 상태 다시 확인 |
| 연결 API | `GET /health`, `POST /documents/upload` |
| 표시 원칙 | 지원하지 않는 기능을 노출하지 않고, 클라이언트 사전 검증과 서버 검증을 구분하며, 실제 계약서·개인정보가 없는 합성 fixture 사용을 안내한다. |

사전 확인은 확장자와 파일 크기를 대상으로 한다. UTF-8 여부의 최종 판단은 서버 응답을 따른다. 파일 입력에는 한 파일만 허용하고 다중 업로드 UI를 제공하지 않는다.

### 7.2 업로드 결과 및 조항 확인 화면

| 구분 | 계약 |
|---|---|
| 목적 | 업로드 성공 여부와 서버가 분리한 조항을 사용자가 분석 전에 확인하게 한다. |
| 표시 항목 | 파일명, 크기, 문자 수, 문서 상태, 조항 수, 조항 순서·표식·제목·본문, 조항 경고, 미분류 구간 및 문서 경고 |
| 사용자 동작 | 조항 펼치기·접기, 다른 파일 선택, 분석 시작 |
| 연결 API | 업로드 응답, 필요 시 `GET /documents/{document_id}` |
| 표시 원칙 | 서버 응답 순서를 유지하고, 경고나 미분류 구간을 숨기지 않으며, 조항 분리가 정확한 법률 구조를 보장한다고 표현하지 않는다. |

`clause_count`가 0이거나 조항 목록이 비어 있으면 분석 시작을 정상 성공 흐름처럼 유도하지 않고 빈 상태와 재업로드 안내를 표시한다.

### 7.3 분석 요청 및 로딩·상태 화면

| 구분 | 계약 |
|---|---|
| 목적 | 분석 요청 접수와 현재 작업 상태를 명확하게 보여준다. |
| 표시 항목 | HTTP 요청 진행 중 로딩, `queued`, `processing`, `completed`, `failed` 상태 텍스트, HTTP 5xx 또는 요청 실패 안내, 재시도 안내 |
| 사용자 동작 | 분석 시작, 상태 다시 확인, 실패 후 파일 확인·재시도 |
| 연결 API | `POST /documents/{document_id}/analysis-jobs`, `GET /analysis-jobs/{job_id}` |
| 표시 원칙 | HTTP 요청 진행 여부와 서버 작업 상태를 구분하고 API 상태를 임의 예측하지 않는다. 현재 동기 처리 한계상 가짜 진행률이나 완료 예상 시간을 만들지 않는다. |

분석 요청 중에는 로딩 상태를 표시하고 중복 제출을 방지한다. 응답이 `completed`이면 결과 조회로 이동한다. `failed` 상태 응답을 실제로 확보한 경우에는 실패 상태와 재시도 안내를 표시한다. HTTP 5xx 또는 요청 실패이면서 `job_id`를 확보하지 못한 경우에는 상태를 추측하거나 상태 조회를 시도하지 않고 일반 실패 안내를 표시한다. 현재 취소 API가 없으므로 작동하는 취소 기능을 제공하지 않는다.

### 7.4 분석 결과 화면

| 구분 | 계약 |
|---|---|
| 목적 | 완료된 최신 분석 결과를 조항과 연결해 사용자가 확인할 항목을 정리하게 한다. |
| 표시 항목 | 조항 식별 정보, `display_label`, `summary`, `expert_review_recommended`, 서비스 한계 안내 |
| 사용자 동작 | 조항별 결과 확인, 원문 조항과 결과 대응 확인, 다른 합성 fixture로 다시 시작 |
| 연결 API | `GET /documents/{document_id}/analysis-results`, 필요 시 `GET /documents/{document_id}` |
| 표시 원칙 | 라벨 텍스트를 항상 표시하고, 결과를 법률 자문이나 최종 판단으로 표현하지 않으며, 전문가 검토 권고가 없더라도 안전을 보장하지 않는다고 안내한다. |

결과 API는 문서의 가장 최근 분석 작업을 반환한다. 사용자별 결과나 분석 이력 목록을 제공하는 API가 아니므로 이력 화면으로 확대 해석하지 않는다.

### 7.5 오류 및 빈 상태 화면

| 구분 | 계약 |
|---|---|
| 목적 | 연결, 업로드, 조회, 분석 실패 또는 데이터 없음 상태에서 사용자가 가능한 다음 행동을 알게 한다. |
| 표시 항목 | 실패한 단계, 안전한 일반 안내, 다시 연결·파일 재선택·재업로드·재시도 동작 |
| 사용자 동작 | 백엔드 연결 재확인, 입력 수정, 업로드 재시도, 분석 재시도, 처음으로 돌아가기 |
| 연결 API | 실패한 단계의 해당 API |
| 표시 원칙 | 서버가 제공하지 않은 실패 원인을 추측하지 않고, 내부 예외·원문·민감정보를 화면이나 console에 노출하지 않는다. |

빈 상태는 최소한 연결 불가, 파일 미선택, 조항 없음, 분석 결과 없음, 조회 대상 없음으로 구분한다. 준비 중 기능을 위한 빈 화면은 만들지 않는다.

## 8. 사용자 흐름

```text
서비스 진입
→ 백엔드 연결 상태 확인
→ TXT 파일 선택
→ 파일 형식과 크기 사전 확인
→ 문서 업로드
→ 업로드 결과와 분리된 조항 확인
→ 분석 시작
→ 분석 작업 상태 확인
→ 완료 시 결과 조회
→ 조항별 라벨, 요약, 전문가 검토 권고 표시
→ 실패 시 재시도 안내
```

단계별 식별자 전달 원칙:

1. 업로드 응답의 `document_id`를 문서 조회와 분석 생성에 사용한다.
2. 분석 생성 정상 응답의 `job_id`를 상태 조회에 사용한다.
3. 완료 상태가 확인되면 `document_id`로 최신 분석 결과를 조회한다.
4. 결과의 `reference_id`와 `clause_id`를 업로드 조항과 대응시킨다.
5. 브라우저 새로고침 이후 상태 복구 방식과 식별자 보관 기간은 이번 문서에서 확정하지 않는다.

## 9. 실패 및 오류 응답 계약

현재 API는 Provider 실패 사유 코드와 상세 메시지를 프론트엔드 응답 계약으로 노출하지 않는다. 실패 사유가 DB에 별도 저장되지도 않는다. 프론트엔드는 HTTP 상태나 내부 예외만으로 마스킹 실패, Provider 실패, schema 실패, 출력 안전성 실패 등을 추측해 사용자에게 단정하지 않는다.

분석 실패의 기본 안내는 다음 방향을 사용한다.

> 분석을 완료하지 못했습니다. 파일을 확인한 뒤 다시 시도해 주세요. 문제가 계속되면 잠시 후 다시 시도해 주세요.

처리 원칙:

- 서버가 명시적으로 제공한 안전한 입력 오류만 해당 범위에서 안내한다.
- 업로드의 확장자, 빈 파일, 1 MiB 초과, UTF-8 오류는 서버 `detail`과 현재 계약이 일치할 때 사용자 행동 중심으로 안내할 수 있다.
- 알 수 없는 오류는 내부 원인을 노출하지 않고 일반 안내와 재시도를 제공한다.
- `404`를 인증 또는 권한 오류로 해석하지 않는다. 현재 인증·인가가 없다.
- 네트워크 실패와 백엔드 오류를 구분할 수 있더라도 법률·분석 실패 원인을 추정하지 않는다.
- 실패 응답에 `job_id`가 없으면 존재하지 않는 작업 상태 조회를 시도하지 않는다.
- 재시도는 중복 작업이나 중복 요청 가능성을 고려해 사용자의 명시적 동작으로 수행한다.

### 현재 명시적 백엔드 오류

| 영역 | HTTP 상태 | 현재 `detail` | 사용자 안내 방향 |
|---|---:|---|---|
| 업로드 형식 | 400 | `Only .txt files are allowed.` | TXT 파일만 선택하도록 한국어로 안내 |
| 빈 파일 | 400 | `The uploaded file is empty.` | 내용이 있는 파일을 선택하도록 한국어로 안내 |
| 크기 초과 | 413 | `The uploaded file exceeds the 1 MB limit.` | 1 MiB 이하 파일을 선택하도록 한국어로 안내 |
| 문자 인코딩 | 400 | `The uploaded file must be UTF-8 encoded.` | UTF-8 TXT 파일을 사용하도록 한국어로 안내 |
| 문서 조회 | 404 | `Document not found.` | 문서를 찾을 수 없으므로 다시 업로드하거나 처음부터 진행하도록 안내 |
| 결과 조회 | 404 | `Analysis result not found.` | 아직 확인 가능한 분석 결과가 없다고 안내 |
| 작업 조회 | 404 | `Analysis job not found.` | 분석 작업을 찾을 수 없다고 안내 |

프론트엔드는 영문 `detail`을 그대로 최종 사용자에게 노출하지 않고, HTTP 상태와 허용된 `detail` 조합을 사용자 행동 중심의 한국어 메시지로 매핑하는 방향을 사용한다. 알 수 없는 `detail`, 내부 stack trace 또는 예상하지 못한 응답 본문은 그대로 노출하지 않는다.

## 11. 프론트엔드 폴더 구조 계약

`docs/05-folder-structure.md`의 기존 방향을 확장하여 API 타입 관리를 위한 `src/types/` 추가를 이번 계약에서 확정한다. `src/types/`는 기존 문서에 이미 확정되어 있던 폴더가 아니라 v0.4.0에서 새로 추가한 구조 계약이다.

```text
frontend/
├── package.json
├── vite.config.js
├── index.html
└── src/
    ├── api/
    ├── assets/
    ├── components/
    ├── pages/
    ├── routes/
    ├── hooks/
    ├── types/
    ├── utils/
    └── styles/
```

- `api/`: HTTP 호출 함수와 API 기본 설정
- `types/`: 문서, 조항, 분석 작업, 분석 결과 및 공통 오류 응답의 TypeScript 타입
- `components/`: 재사용 가능한 표시·입력 컴포넌트
- `pages/`: 화면 계약 단위의 페이지 구성
- `hooks/`: 화면에서 재사용할 요청·상태 로직이 실제로 필요할 때 사용

이 구조는 v0.4.1 scaffold 단계의 기준이며 이번 문서 작업에서 폴더나 코드를 생성하지 않는다. 실제 scaffold 반영 후 `docs/05-folder-structure.md`도 별도 문서 작업으로 갱신해야 한다.

## 12. CORS 선행 조건

현재 `backend/app/main.py`에는 `CORSMiddleware`, `allow_origins` 또는 그에 준하는 CORS 설정이 없다. Vite 개발 서버와 FastAPI 서버를 서로 다른 origin으로 실행하면 브라우저 API 연동 전에 백엔드 CORS 설정이 필요하다.

- 로컬 개발 프론트엔드 origin 후보: `http://localhost:5173`
- v0.4.1에서 실제 로컬 실행 주소와 포트를 확인한 뒤 명시적으로 허용한다.
- 필요한 HTTP 메서드와 헤더만 현재 API 사용 범위에 맞게 허용한다.
- 현재 인증은 없지만 추후 credentials를 사용할 경우 wildcard origin과 함께 사용할 수 없다는 경계를 고려한다.
- 운영 환경에서는 실제 배포된 프론트엔드 origin만 명시적으로 허용한다.
- 운영 환경의 무조건적인 `*` wildcard origin 허용을 금지한다.
- 개발 origin을 운영 설정에 관성적으로 포함하지 않는다.
- 배포 preview origin이나 추가 도메인은 필요성과 위험을 검토한 뒤 별도로 추가한다.

CORS는 브라우저 접근 제어 설정이며 인증·인가를 대체하지 않는다.

## 13. 장기 기획과 현재 범위의 구분

`docs/09-tech-stack.md`와 `docs/planning/user-scenarios.md`에는 텍스트 레이어 PDF를 사용하는 장기 서비스 기획과 후속 서비스 방향이 포함되어 있다. 이 PDF 기획은 폐기하지 않는다. 현재 기술 검증 MVP는 UTF-8 TXT 한 파일로 파이프라인과 화면 연동을 검증하며, TXT 지원을 최종 제품 방향으로 확정하지 않는다. 다음 항목은 현재 백엔드에 구현되지 않았으므로 v0.4의 1차 프론트엔드 MVP에서 제외하고 후속 단계로 둔다.

- 회원가입
- 로그인
- JWT
- 사용자별 접근 제어
- 근로계약서 및 주택 임대차계약서 선택
- PDF
- 분석 이력
- 사용자별 결과 조회

범위 원칙:

- 현재 구현되지 않은 기능을 작동하지 않는 “준비 중” 화면으로 먼저 만들지 않는다.
- 비활성 버튼, 가짜 메뉴, 동작하지 않는 라우트로 장기 기능을 노출하지 않는다.
- TXT 한 파일 업로드부터 최신 분석 결과 확인까지의 실제 연결 흐름에 집중한다.
- 장기 문서의 PDF 중심 시나리오보다 현재 `.txt` API 계약을 우선한다.
- 장기 문서의 인증 사용자 전제보다 현재 무인증 API 사실을 우선하되, 이를 안전한 운영 인증 구조로 오해하지 않는다.
- 합성 Provider 결과를 실제 AI 공급자 연결 완료로 표현하지 않는다.

## 16. 확인된 기존 문서·코드 불일치와 한계

1. `docs/09-tech-stack.md`는 JWT 인증, 사용자별 이력, MySQL, Alembic, PDF 처리 방향을 장기 MVP 기술 범위로 기술하지만 현재 v0.3.x 백엔드에는 구현되어 있지 않다.
2. `docs/planning/user-scenarios.md`는 인증 사용자, 계약서 유형 선택, PDF 업로드, 이력 재조회를 정상 흐름으로 두지만 현재 API는 무인증 TXT 한 파일 흐름만 제공한다.
3. `docs/planning/user-scenarios.md`에서 TXT는 기존 공식 업로드 형식이 아닌 것으로 기록되어 있으나 현재 백엔드의 유일한 허용 형식은 `.txt`다. 1차 프론트엔드는 현재 코드를 우선한다.
4. `docs/05-folder-structure.md`의 프론트엔드 기준 구조에는 API 타입 전용 `src/types/`가 없다. 이번 계약에서 추가를 확정했으며 실제 scaffold 반영 후 기존 폴더 구조 문서의 갱신이 필요하다.
5. `backend/app/main.py`에는 CORS 설정이 없어 Vite 개발 origin에서 브라우저 연동할 수 있는 선행 조건이 충족되지 않았다.
6. 분석 생성은 동기 실행이므로 장기 기획의 지속적인 대기·진행 상태 UX와 현재 실행 방식에 차이가 있다.
7. 분석 실패 상태는 DB에 저장되지만 파이프라인 예외가 다시 발생하므로 생성 API가 `failed` 상태 JSON을 반환한다는 계약은 현재 보장되지 않는다.
8. 실패 사유 코드와 상세 메시지가 프론트엔드 API 계약으로 노출되지 않고 실패 사유 DB 저장도 구현되지 않았다.
9. 현재 기본 Provider는 합성 Provider다. 실제 외부 AI 연결, 실제 네트워크 전송, 실제 공급자 오류 adapter는 구현되지 않았다.
10. FastAPI 앱 버전 문자열은 `0.3.2`로 남아 있다. 이번 문서 작업에서는 코드를 수정하지 않으며 FastAPI 앱 버전 문자열과 프로젝트 버전의 정합성 점검이 필요하다.

이 불일치는 이번 문서에서 기존 파일을 수정해 해소하지 않는다. 현재 구현 가능한 프론트엔드 범위를 좁히고 후속 단계의 선행 조건으로 기록한다.
