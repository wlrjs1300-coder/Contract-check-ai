## 분석 작업 생성 API

- 요청: `POST /documents/{document_id}/analysis-jobs`
- 요청 body: 없음
- 별도 `Content-Type`과 Authorization header: 없음
- `document_id`는 URL path segment로 인코딩한다.
- 응답 필드: `job_id`, `document_id`, `status`
- 응답 `document_id`가 요청한 값과 다르면 사용하지 않는다.

## 분석 상태 조회 API

- 요청: `GET /analysis-jobs/{job_id}`
- `job_id`는 URL path segment로 인코딩한다.
- 빈 `job_id` 또는 문서 ID이면 요청하지 않는다.
- 응답 `job_id`와 `document_id`가 현재 작업과 다르면 사용하지 않는다.
- 자동 폴링하지 않고 사용자가 누르는 “상태 다시 확인”으로 한 번씩 조회한다.

## 상태 모델

- `queued`: 분석 작업 대기
- `processing`: 분석 작업 처리 중
- `completed`: 분석 작업 완료
- `failed`: 분석 작업 실패

다른 문자열, 빈 식별자와 식별자 불일치는 예상하지 못한 응답으로 처리한다.

## 동기 실행 경계

현재 create API는 queued 작업을 저장한 뒤 요청 안에서 `run_analysis_pipeline`을 동기 호출한다. 정상 흐름에서는 queued와 processing이 응답으로 관찰되지 않고 completed가 바로 반환될 수 있다. 화면의 “분석 요청 중”은 HTTP 요청 진행 상태이며 DB processing 상태와 같은 의미로 단정하지 않는다.

현재 기본 Provider는 `SyntheticAnalysisProvider`다. 완료 표시는 API와 화면 흐름을 위한 합성 처리의 작업 완료이며 실제 외부 AI 분석 품질, 법적 검토 완료나 계약서 안전을 의미하지 않는다.

## 실패 응답 경계

파이프라인은 실패 시 rollback 후 AnalysisJob을 failed로 저장하고 예외를 다시 발생시킨다. 따라서 create API가 failed JSON과 `job_id`를 항상 반환한다고 가정하지 않는다.

- 정상 `status: failed`: 분석 작업 실패로 표시
- HTTP 또는 잘못된 응답: 분석 요청 실패로 표시
- 네트워크 실패: 서버 연결 실패로 표시
- `job_id` 미확보 실패: 상태 조회를 시도하지 않음
- 상태 조회 실패: 기존 분석 작업이 failed가 된 것으로 표현하지 않음

서버 detail, stack trace, Provider 내부 오류와 응답 전체를 사용자에게 노출하거나 console에 출력하지 않는다.

## 화면 상태와 사용자 흐름

1. TXT 문서 업로드와 조항 확인
2. 조항이 하나 이상이면 분석 시작
3. HTTP 요청 중 버튼과 문서 선택 비활성화
4. completed 또는 failed 표시
5. queued·processing이면 사용자가 단발성 상태 재조회
6. 조회 결과에 따라 현재 상태 갱신
7. 새 파일 선택 또는 새 업로드 시 이전 job·상태·오류 초기화

조항이 0개인 성공 문서는 분석 API를 호출하지 않고 문서 형식과 조항 표기를 다시 확인하도록 안내한다.

## 오류 메시지

- 정상 failed: `분석 작업을 완료하지 못했습니다. 다시 시도해 주세요.`
- 분석 HTTP·응답 오류: `분석 요청을 완료하지 못했습니다. 잠시 후 다시 시도해 주세요.`
- 분석 네트워크 오류: `서버에 연결하지 못했습니다. 잠시 후 다시 시도해 주세요.`
- 상태 조회 오류: `분석 상태를 확인하지 못했습니다. 잠시 후 다시 시도해 주세요.`
- 상태 조회 네트워크 오류: 서버 연결 실패와 상태 확인 실패를 함께 안내

## 알려진 한계

- 현재 백엔드는 동기 분석이므로 정상 create 응답은 대부분 completed일 수 있다.
- 실패 시 failed JSON과 `job_id` 확보가 보장되지 않는다.
- 작업 실패 상세 사유는 API에 구조화해 노출하지 않는다.
- 현재 결과는 합성 Provider 기반이며 상세 결과를 이번 화면에서 조회하지 않는다.
- 인증과 사용자별 접근 제어가 없어 다른 사용자와 문서·작업을 격리하는 운영 구조가 아니다.
- 비동기 작업 큐 도입 후 자동 폴링 정책과 중단·복구 전략을 별도로 정해야 한다.
