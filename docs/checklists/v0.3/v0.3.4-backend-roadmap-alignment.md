## 2. 작업 목적

기존 버전 계획에서는 v0.3을 화면·UX 설계, v0.4를 DB·API 설계, v0.5를 분석 파이프라인 설계로 정의했다.

그러나 실제 개발 과정에서는 백엔드 구현을 우선 진행했고, PR #35부터 PR #39까지의 작업이 v0.3.x 버전으로 GitHub에 병합됐다.

이번 작업에서는 기존 Git과 PR 기록을 변경하지 않고, 실제 진행 상태를 기준으로 로드맵과 체크리스트를 정렬한다.

## 7. 현재 구현 범위

### 구현 완료

- FastAPI 애플리케이션 기반
- 상태 확인 API
- UTF-8 텍스트 계약서 업로드
- 조항 분할
- 문서·조항 저장 및 조회
- 분석 작업 생성 및 상태 조회
- 분석 결과 저장 및 조회
- SQLAlchemy·SQLite 영속화
- 분석 파이프라인 서비스 분리
- 분석 결과 스키마 검증
- `reference_id` 검증
- 분석 실패 시 부분 결과 rollback
- 실패한 분석 작업 상태 보존
- pytest 및 Ruff 검증

### 아직 구현하지 않은 주요 범위

- 인증·인가
- 개인정보 마스킹의 백엔드 통합
- 실제 AI provider adapter
- 실제 분석 프롬프트와 응답 파싱
- 출력 안전성 필터의 백엔드 통합
- timeout·retry·backoff
- 분석 실패 사유 저장
- 프론트엔드와 결과 화면
- MySQL 전환과 마이그레이션
- 배포·CI 구성

## 9. 다음 단계 결정 원칙

로드맵 정리 완료 후 다음 백엔드 작업은 기존 문서의 버전 번호를 그대로 따르기보다 현재 구현과 의존성을 기준으로 결정한다.

우선순위 후보는 다음과 같다.

1. provider 인터페이스 설계
2. 외부 호출이 없는 합성 provider 구현
3. provider 응답 검증 및 저장 연결
4. 개인정보 마스킹과 출력 안전성 통합
5. 실제 외부 AI 연결 여부 별도 승인

실제 provider 연결부터 바로 시작하지 않는다.
