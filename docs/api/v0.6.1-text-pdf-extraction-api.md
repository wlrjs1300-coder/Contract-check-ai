# v0.6.1 텍스트 PDF 직접 추출 API

## 1. 범위

텍스트 레이어 PDF를 별도 extraction resource로 생성·조회하는 현재 구현 계약이다. 기존 `POST /documents/upload`의 UTF-8 TXT 동작은 변경하지 않는다.

스캔 PDF, 이미지 OCR, 여러 이미지, 사용자 수정·확인, 조항 분리와 분석 연결은 구현하지 않는다. PDF 추출 성공은 분석 완료가 아니며 결과는 항상 사용자 검토 필요 상태다.

## 2. Endpoint

### `POST /extractions`

multipart/form-data의 `file` 한 개를 받는다.

- 성공 상태: `201 Created`
- 성공 상태값: `review_required`
- 추출 방식: `direct`
- 사용자 검토 필요: `true`
- cleanup 성공이 검증된 뒤에만 DB 레코드와 성공 응답 생성

### `GET /extractions/{extraction_id}`

저장된 extraction과 페이지를 페이지 번호 순서로 반환한다.

- 성공 상태: `200 OK`
- 없는 식별자: `404 extraction_not_found`

사용자 확인 endpoint와 `processed` 전환 endpoint는 제공하지 않는다.

## 3. 성공 응답

```json
{
  "extraction_id": "opaque-extraction-id",
  "filename_display": "synthetic.pdf",
  "source_type": "pdf",
  "size_bytes": 1024,
  "page_count": 2,
  "extraction_status": "review_required",
  "extraction_method": "direct",
  "pages": [
    {
      "page_number": 1,
      "extraction_method": "direct",
      "text": "SYNTHETIC_TEXT_LAYER",
      "warnings": [],
      "requires_user_review": true
    }
  ],
  "warnings": [],
  "requires_user_review": true,
  "created_at": "2026-07-20T00:00:00"
}
```

예시는 구조 설명용 합성 값이다.

### Extraction

| 필드 | 형식 | 규칙 |
|---|---|---|
| `extraction_id` | string | UUID 기반 불투명 식별자 |
| `filename_display` | string | 경로·제어문자를 제거하고 255자로 제한한 표시명 |
| `source_type` | string | 현재 `pdf`만 가능 |
| `size_bytes` | integer | 서버가 streaming 저장 중 계산한 크기 |
| `page_count` | integer | 1~100, `pages` 길이와 일치 |
| `extraction_status` | string | 현재 성공 응답은 `review_required` |
| `extraction_method` | string | 현재 `direct` |
| `pages` | array | `page_number` 오름차순 |
| `warnings` | string[] | 문서 수준 고정 warning code |
| `requires_user_review` | boolean | 현재 항상 `true` |
| `created_at` | datetime | DB 생성 시각 |

### ExtractionPage

| 필드 | 형식 | 규칙 |
|---|---|---|
| `page_number` | integer | 1부터 시작하고 원본 PDF 순서 유지 |
| `extraction_method` | string | `direct` |
| `text` | string | PDF 텍스트 레이어에서 직접 추출한 텍스트 |
| `warnings` | string[] | 페이지 수준 고정 warning code |
| `requires_user_review` | boolean | 현재 항상 `true` |

bbox와 confidence는 제공하지 않는다. 지원하지 않는 값을 만들거나 추정하지 않는다.

## 4. 입력 검증

검증 순서:

1. 파일명 확장자가 대소문자 구분 없이 `.pdf`인지 확인
2. MIME이 `application/pdf`, `application/octet-stream` 또는 미제공인지 보조 확인
3. 최대 20 MiB까지 1 MiB chunk로 저장
4. 선두 5바이트가 `%PDF-`인지 확인
5. pypdf strict parser로 구조 열기
6. 암호화 여부 확인
7. 페이지가 1~100인지 확인
8. 페이지별 텍스트 직접 추출
9. 페이지당 200,000자, 문서 전체 2,000,000자 제한 확인
10. 빈 결과와 품질 warning 판정
11. 원본 temp cleanup과 삭제 검증
12. 성공한 경우에만 DB 저장

MIME은 보조 신호이며 파일명이나 MIME만으로 PDF를 신뢰하지 않는다. `%PDF-`와 parser 구조 검증도 polyglot 또는 모든 악성 PDF 탐지를 보장하지 않는다.

## 5. Warning

| code | 조건 | 처리 |
|---|---|---|
| `empty_page_text` | 일부 페이지에 추출 텍스트 없음 | 해당 페이지 사용자 검토 필요 |
| `text_layer_low_quality` | 비공백 문자 20자 미만 또는 의심 문자 비율 5% 초과 | OCR 필요 후보로 표시 |
| `ocr_required` | 빈 페이지 또는 저품질 텍스트 페이지 | 이번 API에서 OCR하지 않음 |
| `empty_pages_detected` | 문서에 빈 페이지가 하나 이상 있음 | 문서 수준 warning |
| `ocr_required_pages` | OCR 필요 후보 페이지가 하나 이상 있음 | 문서 수준 warning |

warning은 품질 신호이며 정확성 보장이나 OCR 판정의 최종 기준이 아니다. 일부 빈 페이지가 있어도 다른 페이지에 유효 텍스트가 있으면 `review_required`로 반환한다. 전체 페이지가 비어 있으면 성공하지 않는다.

## 6. 오류 응답

```json
{
  "detail": {
    "code": "file_type_mismatch",
    "message": "The uploaded file does not match the PDF format.",
    "retryable": false
  }
}
```

| HTTP | code | 조건 |
|---:|---|---|
| 400 | `unsupported_file_type` | `.pdf`가 아닌 확장자 |
| 400 | `file_type_mismatch` | MIME 충돌 또는 `%PDF-` 시그니처 불일치 |
| 400 | `empty_document` | 업로드 바이트가 없거나 페이지가 없음 |
| 400 | `encrypted_pdf` | 암호화 PDF |
| 400 | `corrupt_document` | strict parser가 손상·구조 오류로 거부 |
| 413 | `file_size_limit_exceeded` | 20 MiB 초과 |
| 422 | `page_limit_exceeded` | 100페이지 초과 |
| 422 | `text_limit_exceeded` | 페이지 또는 전체 추출 문자 제한 초과 |
| 422 | `extraction_empty` | 모든 페이지의 직접 추출 텍스트가 비어 있음 |
| 500 | `extraction_failed` | 페이지 텍스트 추출 또는 내부 처리 실패 |
| 500 | `temporary_storage_unavailable` | temp root·요청 디렉터리·서버 파일 생성 또는 쓰기 실패 |
| 500 | `original_cleanup_failed` | 원본 temp 또는 요청 디렉터리 삭제 검증 실패 |
| 404 | `extraction_not_found` | 조회할 extraction 없음 |

오류 응답에는 추출 텍스트, 원본 파일명, 임시 경로, parser 내부 예외와 stack trace를 포함하지 않는다.

`original_cleanup_failed`는 기존 오류보다 우선한다. cleanup 실패 시 extraction 레코드를 만들지 않고 `review_required` 성공 전환을 차단한다.

`temporary_storage_unavailable`은 `retryable: true`로 반환한다. 생성된 요청 디렉터리가 있으면 삭제를 시도하며, 그 삭제도 실패하면 `original_cleanup_failed`를 우선 반환한다. 두 오류 모두 절대 경로, 원본 파일명과 내부 예외를 노출하지 않는다.

## 7. 저장 모델

`extractions`:

- id
- filename_display
- source_type
- size_bytes
- page_count
- status
- method
- warnings
- requires_user_review
- created_at

`extraction_pages`:

- id
- extraction_id
- page_number
- method
- text
- warnings
- requires_user_review

원본 BLOB과 temp path는 DB에 저장하지 않는다. 사용자 확인 revision도 아직 없다. 추출 텍스트는 Phase 5 사용자 확인을 위해 저장되며 보존 기간과 삭제 전환은 아직 미결정이다.

## 8. 기존 TXT 호환성

- `/documents/upload` route·검증·응답 schema를 변경하지 않음
- 기존 `Document`, `Clause`, 분석 작업과 결과 route를 변경하지 않음
- frontend TXT API·type·validation을 변경하지 않음
- 전체 backend 회귀 테스트로 기존 TXT 동작 확인

## 9. 보안 및 운영 한계

- parser worker 격리와 강제 timeout은 아직 없음
- orphan sweeper는 아직 없음; 요청 내 cleanup service와 검증 경계만 구현
- cleanup은 서버가 생성 시 파일 시스템 identity를 기록한 `RequestDirectory`만 받는다. identity 없는 임의 경로는 삭제 대상으로 사용할 수 없다. 요청 디렉터리의 lexical 경로가 temp root의 직접 자식인지 확인하고 생성 시 identity가 유지되는지 다시 확인하며, root 자체, root 밖 경로, symlink와 Windows reparse point·junction은 재귀 삭제하지 않는다.
- 검증 직후 삭제 전에 경로가 교체될 수 있는 TOCTOU 위험은 표준 라이브러리 기반 현재 구현에 남아 있다. 다른 사용자·요청 경로를 삭제해도 안전하다고 가정하지 않으며, parser process 격리와 OS-level sandbox는 후속 범위다.
- 인증·소유권이 없어 실제 계약서 입력 금지
- 시각적 표, 복수 단, bbox와 페이지 rendering 검증 없음
- 이미지로만 구성된 스캔 PDF는 `extraction_empty`로 거부하거나 OCR 필요 warning 상태가 되며 OCR을 수행하지 않음
- pypdf 직접 추출은 PDF 텍스트 레이어의 정확성을 보장하지 않음
- 비동기 작업 queue와 처리 취소는 아직 없음
