# v0.6.3 이미지 OCR 기반 API

## 1. 구현 경계

`POST /extractions/images`는 JPG·JPEG·PNG 한 장 또는 여러 장을 검증·정규화하고 내부 OCR adapter 결과를 공통 extraction resource로 저장한다. 실제 OCR은 로컬 경로(`LocalKoreanOcrAdapter`)로 사용할 수 있으며, non-test 기본 동작은 fail-closed로 `ocr_unavailable` 반환이다.

기존 `POST /extractions` PDF와 `GET /extractions/{extraction_id}`, TXT·분석 API는 유지한다. 이미지 결과는 사용자 확인 전이므로 항상 `review_required=true`, `analysis_blocked=true`이며 기존 분석 job으로 전달하지 않는다.

## 2. Endpoint

### `POST /extractions/images`

multipart/form-data의 `files` 배열에 1~10개 이미지를 입력 순서대로 받는다. 파일명으로 정렬하거나 중복 이미지를 제거하지 않는다.

- 허용 확장자: `.jpg`, `.jpeg`, `.png`
- 허용 MIME: `image/jpeg`, `image/jpg`, `image/png`
- canonical page format: `jpeg` 또는 `png`
- 성공: `201 Created`
- extraction method: `ocr`
- production OCR 미설정: `503 ocr_unavailable`, `503 ocr_engine_unavailable`, `503 ocr_model_unavailable`

### `GET /extractions/{extraction_id}`

저장된 PDF 또는 이미지 extraction을 조회한다. 이미지 페이지의 순서·page identity·정규화 크기·block·경고·차단 상태를 동일하게 반환한다.

## 3. 초기 제한값

아래 값은 v0.6.3 기반 구현의 보수적인 초기값이며 Phase 4 부하·품질 검증 후 변경할 수 있다.

| 제한 | 값 |
|---|---:|
| 이미지 한 장 파일 크기 | 10 MiB |
| 요청 전체 파일 크기 | 30 MiB |
| 요청 이미지 수 | 10장 |
| 한 장 최대 가로·세로 | 각 10,000 px |
| 한 장 최대 픽셀 | 25,000,000 px |
| 요청 전체 최대 픽셀 | 100,000,000 px |
| 페이지 OCR 결과 | 200,000자 |
| 전체 OCR 결과 | 1,000,000자 |
| 페이지 OCR timeout | 10초 |
| 전처리 파생 PNG | 20 MiB |
| metadata | 256 KiB |

픽셀 제한은 압축 파일 크기가 아니라 decoder가 보고하고 완전히 읽은 실제 width×height로 검사한다.

## 4. 검증과 정규화

1. 확장자와 MIME 매핑 검사
2. JPEG·PNG signature와 확장자 일치 검사
3. trailing payload 없는 구조적 종료 검사
4. Pillow decoder format 일치와 전체 픽셀 load
5. 단일 frame 여부와 metadata·크기·dimension·pixel 제한 검사
6. EXIF orientation 적용 후 크기 재계산
7. metadata 제거, RGB 변환, PNG alpha를 흰 배경에 합성
8. 품질 warning 계산과 정규화 PNG 임시 생성
9. 내부 OCR adapter 실행과 timeout·결과 schema·문자 제한 검사
10. 원본·파생 파일 cleanup 검증
11. cleanup 성공 후에만 DB 저장과 성공 응답

기울기 보정은 구현하지 않았다. adapter 경계만 유지하고 `deskew_not_applied` warning으로 현재 한계를 표시한다. 과도한 자동 보정을 수행하지 않는다.

## 5. 이미지 성공 결과 추가 필드

문서 수준:

- `total_pages`, `completed_pages`, `failed_pages`
- `total_text_length`
- `review_required`, `analysis_blocked`
- `created_at`, `updated_at`

페이지 수준:

- `page_number`: 입력 순서에 따라 1부터 연속
- `page_id`: 요청 안에서 안정적인 불투명 식별자
- `source_format`: `jpeg` 또는 `png`
- `normalized_width`, `normalized_height`
- `text`, `text_length`
- `blocks`
- `warnings`
- `review_required`, `analysis_blocked`
- `failure`

block은 `block_index`, `text`, nullable `confidence`, nullable pixel `bbox`, `reading_order`를 가진다. adapter가 confidence나 bbox를 제공하지 않으면 null이며 가짜 값을 만들지 않는다. engine·vendor 이름은 응답과 DB에 넣지 않는다.

## 6. 상태와 경고

성공한 기반 결과도 사용자 확인 API가 없으므로 `extraction_status=review_required`, `analysis_blocked=true`다.

review warning 후보:

- `low_resolution`
- `possible_blur_or_blank`
- `possible_underexposure`, `possible_overexposure`
- `orientation_normalized`
- `metadata_removed`
- `alpha_background_applied`
- `deskew_not_applied`
- `ocr_confidence_unavailable`, `ocr_confidence_low`
- `ocr_block_confidence_low`
- `ocr_text_short`

warning은 품질 신호이지 정확성 보장이나 실패 확정이 아니다.

## 7. 오류 계약

| HTTP | code | 의미 |
|---:|---|---|
| 400 | `unsupported_file_type` | JPG·JPEG·PNG 외 확장자 |
| 400 | `extension_mime_mismatch` | 확장자와 MIME 불일치 |
| 400 | `invalid_image_signature` | signature·확장자·decoder format 불일치 또는 빈 파일 |
| 400 | `corrupted_image` | 손상·truncated·trailing payload 또는 완전 load 실패 |
| 400 | `unsupported_multiframe_image` | animated·multi-frame 이미지 |
| 413 | `image_too_large` | 이미지 한 장 파일 제한 초과 |
| 413 | `image_metadata_limit_exceeded` | metadata 제한 초과 |
| 413 | `image_dimensions_exceeded` | 가로·세로 제한 초과 |
| 413 | `image_pixel_limit_exceeded` | 한 장 픽셀 제한 초과 |
| 413 | `request_image_count_exceeded` | 이미지 장수 제한 초과 |
| 413 | `request_total_size_exceeded` | 요청 전체 바이트 제한 초과 |
| 413 | `request_total_pixel_limit_exceeded` | 요청 전체 픽셀 제한 초과 |
| 413 | `derived_image_limit_exceeded` | 정규화 파생 이미지 제한 초과 |
| 422 | `empty_ocr_result` | OCR 텍스트가 비어 있음 |
| 422 | `extraction_limit_exceeded` | 페이지·전체 추출 문자 제한 초과 |
| 500 | `ocr_failed` | adapter 예외, 일부 페이지 실패 또는 결과 무결성 실패 |
| 500 | `temporary_storage_unavailable` | 임시 파일 생성·쓰기 실패 |
| 500 | `original_cleanup_failed` | 원본·파생 파일 cleanup 검증 실패 |
| 503 | `ocr_unavailable` | non-test에서 OCR adapter 미설정 |
| 503 | `ocr_engine_unavailable` | OCR 실행 환경(예: Tesseract binary) 미설치 |
| 503 | `ocr_model_unavailable` | 언어팩(예: kor) 미설치 또는 초기화 실패 |
| 504 | `ocr_timeout` | 페이지 OCR timeout |

이미지 오류의 `detail.analysis_blocked`는 true다. 오류 응답에는 사용자 파일명, 절대 경로, EXIF, 이미지·OCR 원문, adapter 예외, engine 이름과 stack trace를 포함하지 않는다.

## 8. Cleanup

기존 identity 기반 `RequestDirectory`를 재사용한다. 원본과 정규화 PNG는 같은 요청 디렉터리 안에 무작위 서버 파일명으로 생성한다. validation, decoder, OCR, timeout과 성공 경로에서 디렉터리 전체 삭제를 검증한다. root·root 밖·다른 identity·symlink·junction·reparse point를 재귀 삭제하지 않는다. cleanup 실패는 기존 오류보다 우선하며 extraction DB 저장과 성공 응답을 차단한다.

## 9. 미구현

- 로컬 OCR 품질 기준 달성 평가(품질 지표 임계값 확정)
- 스캔 PDF·direct/OCR 혼합
- 실제 기울기 보정
- 사용자 텍스트 확인·수정 API/UI
- 분석 pipeline 연결
- process·OS sandbox와 강제 worker 종료
- background queue와 orphan sweeper
- 외부 OCR Provider

Thread 기반 timeout은 응답 대기를 중단하지만 실행 중 native 작업을 강제 종료하는 sandbox가 아니다. 실제 엔진 도입 전 process 격리와 자원 회수를 검증해야 한다.
