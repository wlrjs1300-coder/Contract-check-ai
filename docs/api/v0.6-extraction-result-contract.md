# v0.6 추출 결과 API 계약 초안

## 1. 목적과 상태

직접 추출 PDF, OCR 추출 PDF·이미지와 TXT를 페이지 단위 공통 모델로 표현하기 위한 문서 스키마 초안이다. 구현된 API나 데이터베이스 schema가 아니며 필드명, endpoint와 상태 코드는 Phase 2 API 설계에서 변경될 수 있다.

v0.6.1에서는 텍스트 PDF의 `direct` 추출을 `POST /extractions`로 구현했다. v0.6.3은 `POST /extractions/images`에 다중 JPG·JPEG·PNG 검증·정규화와 OCR adapter 기반 공통 page·block 결과를 추가했다. v0.6.6에서는 `OCR_ADAPTER=local_korean`(또는 `local`, `tesseract`)일 때 로컬 실제 OCR 경로를 허용하며, non-test 기본은 `ocr_unavailable` fail-closed다. 이미지 결과는 `review_required=true`, `analysis_blocked=true`다. 사용자 확인 revision, `processed` 전환과 분석 연결은 아직 구현하지 않았다. 세부 계약은 v0.6.1 PDF와 v0.6.3/0.6.6 이미지 API 문서를 따른다.

핵심 원칙은 원본 추출본과 사용자 확인 텍스트를 분리하고, 페이지 순서와 근거 위치를 보존하며, 사용자가 확인한 텍스트만 조항 분리·분석에 사용하는 것이다.

## 2. 용어

- 원본 파일: 사용자가 업로드한 PDF·이미지·TXT 바이트
- 추출 텍스트: 직접 추출 또는 OCR 추출이 최초 생성한 변경 불가 텍스트
- 사용자 확인 텍스트: 사용자가 추출 텍스트를 확인하거나 수정해 확정한 텍스트
- 문서 페이지: 단일 PDF 페이지, 여러 이미지 중 한 장, 또는 TXT를 정규화한 한 단위
- OCR 경고: 누락·회전·낮은 품질 등 사용자가 확인해야 할 비확정 신호
- 분석 가능: 모든 페이지가 유효하고 사용자 확인이 끝나 조항 분리로 전달할 수 있는 상태
- 분석 차단: 입력·추출·확인 조건이 충족되지 않아 분석을 시작할 수 없는 상태

## 3. 문서 응답 후보

```json
{
  "document_id": "opaque-document-id",
  "source_type": "pdf",
  "filename_display": "sample.pdf",
  "page_count": 2,
  "extraction_status": "review_required",
  "extraction_method": "mixed",
  "pages": [],
  "warnings": ["page_requires_review"],
  "requires_user_review": true,
  "user_confirmed": false,
  "confirmed_at": null,
  "created_at": "2026-07-19T00:00:00Z"
}
```

예시는 형식 설명용 가상 값이며 실제 문서 내용을 포함하지 않는다.

### 3.1 문서 필드

| 필드 | 형식 | 필수 | 의미·검증 |
|---|---|---:|---|
| `document_id` | string | 예 | 외부에 노출 가능한 불투명 식별자 |
| `source_type` | `pdf` / `jpg` / `jpeg` / `png` / `txt` / `image_set` 후보 | 예 | 서버가 시그니처로 판별한 원본 유형 |
| `filename_display` | string | 예 | 경로·제어문자를 제거하고 길이를 제한한 표시명 |
| `page_count` | integer | 예 | 1 이상, `pages.length`와 일치 |
| `extraction_status` | enum | 예 | 수신부터 사용자 확인까지의 처리 상태 |
| `extraction_method` | `direct` / `ocr` / `mixed` | 예 | 문서 전체에 사용된 방법 요약 |
| `pages` | array | 예 | `page_number` 오름차순의 문서 페이지 |
| `warnings` | string[] | 예 | 문서 수준의 고정 경고 코드, 원문 미포함 |
| `requires_user_review` | boolean | 예 | 경고·OCR·혼합 처리 등 검토 필요 여부 |
| `user_confirmed` | boolean | 예 | 현재 revision 전체를 사용자가 확정했는지 여부 |
| `confirmed_at` | RFC 3339 timestamp 또는 null | 예 | 확정 전에는 null |
| `created_at` | RFC 3339 timestamp | 예 | resource 생성 시각, UTC 권장 |

`filename_display`는 서버 경로에 사용하지 않는다. 원본 파일명을 장기 저장할지는 별도 보존 정책으로 정한다.

## 4. 페이지·block 모델

```json
{
  "page_number": 1,
  "extraction_method": "ocr",
  "text": "SYNTHETIC_PAGE_TEXT",
  "blocks": [
    {
      "block_id": "page-1-block-1",
      "text": "SYNTHETIC_BLOCK_TEXT",
      "bbox": {"x": 0.1, "y": 0.2, "width": 0.5, "height": 0.1},
      "confidence": null,
      "warnings": ["confidence_unavailable"]
    }
  ],
  "confidence": null,
  "warnings": ["user_review_required"],
  "requires_user_review": true
}
```

### 4.1 페이지 필드

| 필드 | 형식 | 필수 | 의미·검증 |
|---|---|---:|---|
| `page_number` | integer | 예 | 1부터 시작하고 중복·누락 없이 연속 |
| `extraction_method` | `direct` / `ocr` | 예 | 이 페이지의 처리 방식 |
| `text` | string | 예 | 변경하지 않는 원본 추출 텍스트 |
| `blocks` | array | 예 | 순서가 유지된 block; block 미지원 엔진은 빈 배열 가능 |
| `confidence` | number 또는 null | 예 | provider 원시값을 명시된 범위로 정규화하거나 null |
| `warnings` | string[] | 예 | 페이지 수준 고정 경고 코드 |
| `requires_user_review` | boolean | 예 | 페이지 확인 필요 여부 |

### 4.2 block 필드

| 필드 | 형식 | 필수 | 의미·검증 |
|---|---|---:|---|
| `block_id` | string | 예 | 문서 안에서 유일한 불투명 식별자 |
| `text` | string | 예 | block의 원본 추출 텍스트 |
| `bbox` | object 또는 null | 아니요 | 페이지 좌표에 정규화된 선택 필드 |
| `confidence` | number 또는 null | 예 | 엔진이 제공하지 않으면 null |
| `warnings` | string[] | 예 | block 수준 고정 경고 코드 |

`bbox` 후보는 `x`, `y`, `width`, `height`를 0 이상 1 이하의 정규화 좌표로 표현한다. 회전 정규화 후 사용자에게 보이는 페이지를 좌표 기준으로 삼는다. 엔진이 안정적인 위치를 제공하지 않으면 bbox를 만들지 않는다.

## 5. confidence 원칙

- confidence는 OCR 엔진의 내부 추정 또는 정규화된 품질 신호이며 정확성 보장 수치가 아니다.
- 서로 다른 엔진의 confidence를 직접 같은 의미로 비교하지 않는다.
- 직접 추출이나 일부 OCR 엔진은 confidence를 제공하지 않을 수 있으므로 null을 정상 값으로 허용한다.
- confidence가 높아도 글자·숫자·조항이 틀릴 수 있고, 낮아도 텍스트가 맞을 수 있다.
- confidence만으로 분석 가능 상태를 만들거나 사용자 검토를 생략하지 않는다.
- 정규화 방법, 원시값 보존 여부와 경고 임계값은 엔진 스파이크 후 결정한다.

## 6. 원본 추출본과 사용자 수정본

원본 추출 `pages[].text`와 `blocks[].text`를 수정 요청으로 덮어쓰지 않는다. 사용자 확인은 별도 revision resource 후보로 표현한다.

```json
{
  "document_id": "opaque-document-id",
  "revision_id": "opaque-revision-id",
  "based_on_extraction_revision": 1,
  "pages": [
    {
      "page_number": 1,
      "confirmed_text": "SYNTHETIC_CONFIRMED_TEXT"
    }
  ],
  "user_confirmed": true,
  "confirmed_at": "2026-07-19T00:10:00Z"
}
```

필수 규칙:

- 수정본은 어떤 extraction revision과 페이지를 기준으로 했는지 기록한다.
- 빈 페이지, page 번호 누락·중복과 허용 문자 수 초과는 거부한다.
- 원본 추출본과 수정본의 diff를 원문 로그로 남기지 않는다.
- 재추출이 발생하면 이전 확인을 자동 승계하지 않고 재확인 필요 상태로 둔다.
- 조항 분리와 분석에는 최신의 명시적으로 확정된 `confirmed_text`만 사용한다.
- 사용자 확인 주체를 기록하려면 인증·인가가 먼저 필요하다.

## 7. extraction 상태 후보

| 상태 | 의미 | 분석 가능 여부 |
|---|---|---|
| `received` | 업로드 resource 생성 | 차단 |
| `validating` | 시그니처·구조·자원 검사 | 차단 |
| `extracting` | 직접 추출 또는 OCR 진행 | 차단 |
| `review_required` | 추출 완료, 사용자 검토 필요 | 차단 |
| `confirmed` | 사용자 확인 텍스트 확정 | 후속 조항 분리 가능 |
| `processed` | 조항 분리와 원본·파생 파일 삭제 검증 완료 | 분석 가능 |
| `rejected` | 입력 계약 위반 | 차단 |
| `extraction_failed` | 추출 실패 또는 빈 결과 | 차단 |
| `failed` | 추출 이후 내부 처리 실패 | 차단 |

문서 위험 라벨은 extraction 상태가 아니다. `위험 없음`에 해당하는 분석 결과가 있더라도 추출 성공·근거 검증과 별개이며 안전을 보장하지 않는다. `분석 실패`는 결과를 생성하지 못한 상태로서 위험이 없다는 뜻이 아니다.

## 9. API 오류 코드 후보

일관된 envelope 후보:

```json
{
  "error": {
    "code": "file_type_mismatch",
    "message": "업로드한 파일 형식을 확인해 주세요.",
    "retryable": false,
    "correlation_id": "opaque-correlation-id"
  }
}
```

| HTTP 후보 | code | 상황 | 재시도 방향 |
|---:|---|---|---|
| 400 | `unsupported_file_type` | 지원하지 않는 형식 | 지원 형식 재선택 |
| 400 | `file_type_mismatch` | 확장자·MIME·시그니처 불일치 | 정상 파일 재선택 |
| 400 | `encrypted_pdf` | 암호화 PDF | 암호 해제 후 재업로드 |
| 400 | `corrupt_document` | parser·decoder가 손상 판정 | 정상 원본 재업로드 |
| 400 | `empty_document` | 페이지·픽셀·텍스트 없음 | 내용 있는 파일 선택 |
| 413 | `file_size_limit_exceeded` | 바이트 제한 초과 | 파일 축소·분할 |
| 422 | `page_limit_exceeded` | 페이지·장수 제한 초과 | 묶음 분할 |
| 422 | `image_resolution_too_low` | 판독 불가능한 저해상도 | 재촬영 |
| 422 | `page_order_invalid` | 순서 누락·중복 | 순서 재확인 |
| 409 | `confirmation_required` | 사용자 확인 전 분석 요청 | 확인 완료 후 재요청 |
| 409 | `stale_extraction_revision` | 재추출 후 이전 수정본 제출 | 최신 결과 재확인 |
| 422 | `extraction_empty` | 유효 추출 텍스트 없음 | 재촬영·다른 파일 |
| 500 | `extraction_failed` | 내부 추출 실패 | 안전한 재시도 안내 |
| 503 | `ocr_temporarily_unavailable` | OCR worker 일시 장애 | 제한된 재시도 |
| 500 | `original_cleanup_failed` | 원본 또는 처리용 파생 파일 삭제 검증 실패 | 격리·운영 경보·재삭제, `processed` 전환과 분석 진행 차단 |

`original_cleanup_failed`는 일반 분석 성공이나 `위험 없음`으로 대체하지 않는다. 삭제 성공이 검증되기 전에는 `processed` 또는 분석 가능 상태로 전환하지 않는다. 이는 v0.6의 개인정보 보호 우선 기본값이며 별도 보안 승인 없이는 완화하지 않는다.

HTTP 상태와 코드의 최종 매핑은 기존 FastAPI 오류 계약과 비동기 작업 설계를 검토한 뒤 확정한다. 프론트는 미확인 내부 사유를 추측하지 않는다.

## 10. 기존 문서 응답 호환 전략

현재 성공 응답은 `document_id`, `filename`, `content_type`, `size_bytes`, `character_count`, `status`, `clause_count`, `clauses`, `unclassified_sections`, `document_warnings`를 제공한다. v0.6은 이 응답을 곧바로 위 extraction 응답으로 교체하지 않는다.

후보 전략:

1. 별도 `/extractions` resource를 만들고, 후속 사용자 확인 endpoint가 구현된 뒤 `processed` 상태에서 기존 문서 resource를 생성한다. v0.6.1은 extraction 생성·조회까지만 구현했다.
2. 기존 `/documents/upload`의 TXT 동작은 유지하고 새 형식만 버전이 있는 endpoint로 도입한다.
3. 기존 필드를 유지하면서 extraction 세부를 선택 필드 또는 link로 추가한다.

v0.6.1은 1안의 별도 extraction resource를 선택했다. 기존 문서 resource 생성과 사용자 확인 API는 Phase 5까지 확정하지 않는다. 어떤 전략이든 다음을 보장한다.

- 기존 TXT 업로드·조회·분석 테스트가 변경 없이 통과
- 새로운 필드를 기존 클라이언트의 필수 응답으로 갑자기 만들지 않음
- `document_id`와 조항·분석 결과 연결을 유지
- `source_hash` 등 내부 검증값을 사용자 표시 필수값으로 만들지 않음
- API 버전 변경이 필요하면 migration 기간과 deprecation을 문서화

| endpoint·작업 모델 | 동기 처리 시간, 재시작 복구 | 텍스트 PDF 처리 시간·API 흐름 시험 | API/Backend | Phase 2 설계 중 |

## 12. 계약 테스트 후보

- 페이지 번호가 1부터 연속이고 `page_count`와 일치함
- 직접 추출·OCR·혼합 문서가 같은 schema를 만족함
- bbox·confidence 미제공 결과가 null 또는 생략 규칙을 지킴
- 원본 추출 텍스트가 사용자 수정으로 변경되지 않음
- 최신 extraction revision에 대한 확인만 수락함
- 확인 전 조항 분리·분석 요청을 차단함
- 빈 텍스트·페이지 순서 오류·OCR 실패를 `위험 없음`으로 반환하지 않음
- 오류 envelope에 원문, 파일명, 로컬 경로와 내부 예외가 없음
- 기존 TXT 문서 응답과 frontend contract test가 유지됨
