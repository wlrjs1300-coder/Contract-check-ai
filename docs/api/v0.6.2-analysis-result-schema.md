# v0.6.2 고객 가치 분석 결과 목표 스키마

## 1. 문서 성격

이 문서는 후속 구현을 위한 목표 API·도메인 계약이다. 현재 API는 조항별 `clause_id`, `reference_id`, `display_label`, `summary`, `expert_review_recommended`만 반환하며, 현재 DB와 frontend를 이 문서에 맞게 변경하지 않는다. 기본 `SyntheticAnalysisProvider` 결과는 실제 분석 품질을 증명하지 않는다.

## 2. 상위 응답 후보

```json
{
  "document_id": "opaque-document-id",
  "job_id": "opaque-job-id",
  "status": "completed",
  "result_state": "review_required",
  "items": [],
  "missing_terms": [],
  "dates": [],
  "money": [],
  "obligations": [],
  "warnings": [],
  "schema_version": "v0.6.2-draft"
}
```

`result_state` 후보:

| 값 | 의미 |
|---|---|
| `review_required` | 결과가 있으나 사용자 확인 필요 |
| `additional_confirmation` | 근거·문맥·추출 품질이 충분하지 않음 |
| `no_findings` | 정해진 검사 범위에서 표시할 항목을 찾지 못함; 안전 보장 아님 |
| `analysis_failed` | 분석 작업 실패 |
| `extraction_failed` | 원문 추출 실패로 분석 불가 |

작업 상태 `queued`, `processing`, `completed`, `failed`와 결과 의미 상태를 혼합하지 않는다.

## 3. AnalysisResultItem 목표 필드

| 필드 | 형식 | 필수 | 계약 |
|---|---|---:|---|
| `result_id` | string | 예 | 불투명 결과 식별자 |
| `clause_id` | string/null | 조건부 | 조항 결과일 때 현재 문서 조항과 일치 |
| `reference_id` | string | 예 | 검증된 문서·조항 reference |
| `category` | enum | 예 | 결과 분류 코드 |
| `title` | string | 예 | 짧고 비단정적인 제목 |
| `severity` | enum | 예 | 표시 우선순위, 법적 위험 확정값 아님 |
| `display_label` | enum | 예 | 사용자용 텍스트 라벨 |
| `reason` | string | 예 | 왜 확인해야 하는지 근거 중심 설명 |
| `user_impact` | string | 예 | 발생 가능한 실무 영향; 확정 표현 금지 |
| `evidence_type` | enum | 예 | 결과의 근거 유형 |
| `evidence_references` | string[] | 조건부 | 원문·비교·계산 근거 reference; 근거 유형별 최소 개수 적용 |
| `source_quote` | string/null | 조건부 | `evidence_type=source_quote`일 때 필수인 최소 원문 일부; 전체 원문 금지 |
| `searched_scope` | string[] | 조건부 | `document_scope_absence`에서 실제 검색한 문서·페이지·텍스트 범위 |
| `expected_term` | string/null | 조건부 | `document_scope_absence`에서 찾으려 한 중요 항목 |
| `calculation_basis` | string/null | 조건부 | 파생 일정·금액의 산식과 기준 값 |
| `assumptions` | string[] | 조건부 | 파생 결과의 명시적 가정; 가정이 없으면 빈 배열 |
| `page_number` | integer/null | 조건부 | 확인된 페이지가 있을 때 1부터 시작 |
| `clause_marker` | string/null | 아니요 | 제1조 등 표시용 표지 |
| `source_start` | integer/null | 조건부 | 확인 텍스트 기준 시작 offset |
| `source_end` | integer/null | 조건부 | exclusive 종료 offset |
| `checkpoints` | string[] | 예 | 사용자가 확인할 구체 항목 |
| `questions_to_ask` | object[] | 예 | 대상·목적·질문을 가진 실행 항목 |
| `missing_information` | string[] | 예 | 근거 부족·별지 미입력 등 미확인 정보 |
| `expert_review_recommended` | boolean | 예 | 전문가 검토 권고 여부 |
| `expert_review_reason` | string/null | 조건부 | 권고 시 구체 이유 |
| `review_required` | boolean | 예 | 사용자 확인 필요 여부; 현재 기본 true 후보 |
| `confidence` | number/null | 아니요 | 내부 품질 신호; 정확성 보장 수치 아님 |
| `model_version` | string | 예 | 합성 또는 승인된 모델 버전 식별자 |
| `prompt_version` | string | 예 | 프롬프트 계약 버전; 합성 규칙도 명시 |
| `rule_version` | string | 예 | 후처리·분류·안전 규칙 버전 |
| `created_at` | datetime | 예 | 결과 생성 시각 |

`summary`는 현재 하위 호환 필드로 유지할 수 있으나 목표 구조에서는 `reason`과 `user_impact`를 합쳐 축약한 표시용 파생 값으로 정의한다. migration·호환 기간은 구현 단계에서 결정한다.

필수 여부는 `category`와 `evidence_type`의 조합으로 검증한다. 모든 결과에는 `evidence_type`이 필요하지만 원문 인용이 모든 결과에 필요한 것은 아니다.

## 4. Enum 후보

### category

- `attention_clause`
- `ambiguous_term`
- `automatic_renewal`
- `termination`
- `penalty`
- `liability`
- `confidentiality`
- `unilateral_change`
- `inspection_payment`
- `restoration`
- `cost_transfer`
- `missing_term`
- `date_schedule`
- `money_payment`
- `obligation`
- `responsibility_imbalance`
- `additional_confirmation`

### severity

| 값 | 표시 목적 |
|---|---|
| `critical_attention` | 놓치면 영향이 클 수 있어 가장 먼저 확인 |
| `attention` | 협상·확인이 필요한 후보 |
| `informational` | 일정·금액·의무 등 참고 구조화 |
| `undetermined` | 근거·문맥 부족으로 우선순위 확정 불가 |

### display_label

현재 라벨 `안전`, `주의`, `위험`, `추가 확인`을 호환 후보로 유지한다. `안전`은 문서 전체 안전을 의미하지 않으며 결과 단위에서도 정해진 검사에서 주의 신호를 찾지 못했다는 제한된 의미만 허용한다. 장기적으로 `확인`, `우선 확인`, `정보`, `추가 확인` 같은 비법률적 라벨로 변경할지는 사용자 연구 후 결정한다.

### evidence_type

| 값 | 사용 조건 | 조건부 필수 필드 |
|---|---|---|
| `source_quote` | 특정 원문 구절이 결과를 직접 뒷받침 | `source_quote`, 1개 이상의 `evidence_references` |
| `document_scope_absence` | 확인된 문서 범위에서 기대 항목을 찾지 못함 | `searched_scope`, `expected_term`, 별지·첨부 미입력 가능성을 포함한 `missing_information` |
| `cross_clause_comparison` | 둘 이상의 조항·문장을 비교 | 2개 이상의 `evidence_references` |
| `derived_schedule` | 원문 날짜·기간에서 일정 산출 | 1개 이상의 `evidence_references`, `calculation_basis`, `assumptions` |
| `derived_amount` | 원문 금액·비율에서 금액 산출 | 1개 이상의 `evidence_references`, `calculation_basis`, `assumptions` |

`source_quote`가 없는 `document_scope_absence` 결과에 가짜 인용을 생성하지 않는다. 모든 근거 유형은 검증 가능해야 하며 reference가 필요한 유형은 현재 문서 범위의 reference만 허용한다.

## 5. 질문 구조

```json
{
  "audience": "counterparty",
  "purpose": "검수 기한 확인",
  "question": "검수 의견은 결과물 제출 후 며칠 이내에 전달되나요?"
}
```

`audience` 후보는 `counterparty`, `negotiation`, `expert`다. 질문은 원문에서 확인되지 않은 사실을 사실처럼 전제하지 않는다.

## 6. 누락 가능성 구조

```json
{
  "result_id": "opaque-result-id",
  "category": "missing_term",
  "evidence_type": "document_scope_absence",
  "evidence_references": [],
  "contract_type": "service_contract",
  "expected_term": "inspection_criteria",
  "status": "potentially_missing",
  "searched_scope": ["confirmed_text", "document_body"],
  "related_reference_ids": [],
  "missing_information": ["별지 입력 여부를 확인할 수 없음"],
  "review_required": true
}
```

계약 유형 코드는 구현 전 allowlist로 확정해야 하며 위 예시의 값은 구조 설명용이다. 상태는 `potentially_missing`, `found`, `not_evaluated`를 구분한다. 사용자 화면에는 `누락 가능성`으로만 표시한다. 누락 가능성은 존재하지 않는 문장을 인용하지 않고 검토 범위, 기대 항목, 검색 범위와 별지·첨부 미입력 가능성을 근거로 제시한다.

## 7. 날짜·금액·의무 후보 구조

### StructuredDate

- `date_type`
- `raw_value`
- `normalized_value` nullable
- `evidence_type`: 직접 날짜는 `source_quote`, 계산 일정은 `derived_schedule`
- `evidence_references`, `calculation_basis`, `assumptions`
- `timezone` nullable
- `reference_id`, `source_quote`, `page_number`
- `review_required`, `warnings`

### StructuredMoney

- `money_type`
- `raw_value`
- `amount` nullable
- `currency` nullable
- `tax_included` nullable
- `evidence_type`: 직접 금액은 `source_quote`, 계산 금액은 `derived_amount`
- `evidence_references`, `calculation_basis`, `assumptions`
- `payment_condition` nullable
- 근거 필드와 경고

파싱 실패 값을 `0`으로 저장하지 않는다.

### StructuredObligation

- `actor`
- `action`
- `deadline` nullable
- `conditions`
- `consequence` nullable
- 근거 필드와 확인 필요 상태

## 8. 근거 무결성 규칙

1. `reference_id`와 `evidence_references`는 필요한 근거 유형에서 요청 문서에 존재해야 한다.
2. `clause_id`가 있으면 같은 문서의 reference와 일치해야 한다.
3. `source_start < source_end`이며 확인 텍스트 경계 안이어야 한다.
4. `evidence_type=source_quote`이면 `source_quote`가 해당 범위의 정규화된 최소 인용과 일치해야 한다.
5. `document_scope_absence`는 검토 범위·기대 항목·검색 범위·별지나 첨부 미입력 가능성을 가져야 한다.
6. `cross_clause_comparison`은 서로 비교한 reference를 두 개 이상 가져야 한다.
7. `derived_schedule`과 `derived_amount`는 원문 reference, 계산 기준과 가정을 가져야 한다.
8. 페이지 기반 입력이면 `page_number`가 reference mapping과 일치해야 한다.
9. 근거가 없거나 다른 문서·조항을 가리키면 결과를 폐기한다.
10. 과도한 원문 복제, 개인정보 재생성, 문서 지시문에 영향받은 출력은 안전 검증에서 차단한다.

offset은 마스킹 전 추출 텍스트, 사용자 확인 텍스트 또는 마스킹 입력 중 어느 기준인지 반드시 버전과 함께 정의해야 한다. 현재는 미결정이며 Phase 5~6에서 확정한다.

## 9. 검증 규칙

- 필수 문자열 공백 금지, 배열 개수·문자 길이 제한
- enum allowlist와 schema version 검증
- `expert_review_recommended=true`이면 reason 필수
- `severity=undetermined` 또는 근거 부족이면 `review_required=true`
- 날짜·금액 정규화 실패는 nullable과 warning으로 보존
- `model_version`, `prompt_version`, `rule_version` 누락 결과 저장 금지
- 실패·빈 결과·추가 확인을 정상적인 안전 결과로 변환 금지
- confidence를 백분율 정확도로 사용자에게 표현 금지

## 10. 개인정보·보존

- 인용은 최소화하고 전체 원문을 결과 item마다 중복 저장하지 않는다.
- 분석 결과 보존과 원본 파일 보존을 분리한다.
- 결과·근거 접근에는 향후 문서 소유권과 workspace 권한이 필요하다.
- 로그, analytics, 결제 Provider에 source quote나 계약 내용을 보내지 않는다.
- 모델·프롬프트·규칙 버전은 비밀값이나 원문을 포함하지 않는 식별자다.

## 11. 구현 전 결정 사항

- 하위 호환 API 경로와 schema versioning
- DB 정규화와 JSON 필드 경계
- source offset 기준 revision
- 최대 인용 길이와 배열 제한
- 계약 유형 taxonomy
- confidence 계산·노출 여부
- 실제 Provider와 전문가 평가 기준

이 결정과 migration 없이 현재 API를 이 목표 구조로 변경하지 않는다.
