# v0.6.9 Customer Value Analysis Results API

## 목적
- 1차 고객 가치 분석 결과 API에서 증거 연결 및 결과 요약 계약을 v0.6.9 기준으로 고정한다.
- frontend 미구현 전제 하에서 backend 응답의 구조적 계약을 수립한다.
- 가짜 법률 자문, 확정형 판단, 최종 판정 표현을 피한다.

## questions / suggestions / facts evidence 필수 정책
- `questions_to_ask` 항목의 `related_evidence_ids`는 생성 시 명시되어야 한다.
- 항목이 존재하지만 `related_evidence_ids`가 빈 배열이면 실패한다.
- `related_evidence_ids`는 해당 finding의 evidence 식별자로만 허용한다.
- `suggestion_evidence_missing` 오류 코드를 통해 누락을 반환한다.
- `extracted_facts[].evidence`는 필수이며, 증거 id 누락/유효성 오류 시 `fact_evidence_missing`을 반환한다.
- `fact.evidence`에 허용되지 않는 evidence id가 있으면 실패한다.
- 임의 자동 폴백으로 첫 evidence를 연결하지 않는다.

## severity
- 허용: `critical`, `high`, `medium`, `low`, `info`
- `critical_count`, `high_count`, `medium_count`, `low_count`, `info_count`를 반환하고 `total_findings`와 합이 일치해야 한다.

## action priority
- 허용: `before_signing`, `negotiate`, `clarify`, `monitor`, `expert_review`, `informational`
- `analysis_summary.top_priorities[].action_priority`에 포함된다.

## expert review reason code 허용 목록
- `critical_severity`
- `unlimited_liability_suspected`
- `high_penalty`
- `rights_waiver_suspected`
- `intellectual_property_transfer`
- `privacy_responsibility`
- `non_compete_restriction`
- `complex_governing_law`
- `low_ocr_confidence_on_key_clause`
- `multiple_high_findings`
- `key_fact_conflict`

- `expert_review_recommended=True` 이고 `expert_review_reason_codes`가 비어 있으면 실패.
- `expert_review_reason_codes`가 존재하면 `expert_review_summary`는 필수.

## analysis_summary 계약
- 필수 필드: `overall_risk_level`, `overall_display_label`, `total_findings`, `critical_count`, `high_count`, `medium_count`, `low_count`, `info_count`, `top_priorities`, `key_dates`, `key_amounts`, `key_obligations`, `missing_terms_count`, `ambiguity_count`, `expert_review_recommended`, `snapshot_version`, `snapshot_stale`
- `top_priorities`는 최대 3개 객체 배열이며 각 항목은 다음 형식:
  - `finding_id`
  - `severity`
  - `title`
  - `action_priority`
- `total_findings`는 severity count 합과 일치해야 한다.

## 날짜·금액·의무 정규화
- 날짜(`date_value`)와 금액(`amount_value`) 및 의무(`obligation_party`, `obligation`)는 fact 레벨에서 정규화되어 전달한다.
- 계약일로부터 N일과 같은 상대 날짜는 절대 날짜로 임의 변환하지 않는다.
- 금액은 숫자형 값 또는 금액 문자열로 저장 가능, `currency`를 함께 유지한다.

## 누락·모호 표현 정책
- 누락 확정/위법/무효 같은 단정 표현 금지.
- evidence와 맥락이 있어야 하며, 신뢰도는 낮을 수 있음을 명시한다.
- 누락 가능성은 `missing_terms_count` 및 `ambiguity_count`로 집계한다.

## plan-independent 동등성
- plan(plan code/등급) 변경은 노출되는 결과 형식과 필수 검증 규칙을 바꾸지 않는다.

## 보안/법률 경계
- 법률 자문/최종 판단 표현을 포함하지 않는다.
- LLM 최종 판단에 의한 판정값을 반환하지 않는다.
- 실제 API 키/토큰/secret, 계약 원문, 개인정보를 결과 본문에 직접 노출하지 않는다.
- 실제 실서비스 연결은 프론트엔드 미구현 상태에서 backend 단독 계약으로 보존한다.
