# v0.7.4 Alembic baseline 및 ownership migration

## 목적

v0.7.3 스키마를 Alembic baseline으로 고정하고, v0.7.4 인증·소유권 변경을 별도 revision으로 적용한다.

## revision 체인

1. `0001_current_schema_baseline`
2. `0002_auth_ownership`

`0001_current_schema_baseline`은 v0.7.3 ORM schema와 parity를 이룬다. `0002_auth_ownership` 적용 후 Alembic head는 현재 ORM metadata와 parity를 이룬다.

## baseline parity

- Python-side default를 임의의 `server_default`로 만들지 않는다.
- `Clause.reference_id`는 unique index를 가진다.
- `ExtractionPage.page_number` 단독 index는 만들지 않는다.
- baseline에서 v0.7.3 모델의 테이블, 컬럼, 제약, index를 재현한다.
- `downgrade base`로 revision 전체를 되돌리는 경로를 검증한다.

## ownership migration

`0002_auth_ownership`은 다음 변경을 적용한다.

- `users` 테이블 생성
- `users.email` unique index 생성
- `auth_version >= 1` `CheckConstraint` 생성
- `Document.owner_id`에 `NOT NULL`, 사용자 FK `RESTRICT`, index 적용
- `Extraction.owner_id`에 `NOT NULL`, 사용자 FK `RESTRICT`, index 적용

`owner_id`는 `AnalysisJob`, `Clause`, `AnalysisResultItem`, `ExtractionPage`에 추가하지 않는다. 해당 리소스는 부모 `Document`, `AnalysisJob`, `Extraction` 경계를 통해 제한한다.

## legacy data fail-closed

`0002_auth_ownership`은 빈 개발 DB에서만 진행할 수 있다.

- migration 시작 전에 `documents` 또는 `extractions` row 존재 여부를 검사한다.
- 하나라도 존재하면 `users` 테이블이나 `owner_id` 컬럼을 생성하기 전에 즉시 중단한다.
- legacy row를 임의 사용자에게 자동 배정하지 않는다.
- 실패 후 revision은 `0001_current_schema_baseline`을 유지한다.
- 기존 legacy row를 변경하거나 삭제하지 않는다.
- 부분 DDL을 남기지 않는다.

## schema 변경 기준

- production application startup에서는 `Base.metadata.create_all()`을 사용하지 않는다.
- production schema 변경은 Alembic migration을 기준으로 한다.
- 테스트용 격리 SQLite DB 초기화에서는 `Base.metadata.create_all()`을 사용한다.

## 검증 범위

- `0001_current_schema_baseline`과 v0.7.3 ORM schema parity
- `0002_auth_ownership` 적용 후 head와 현재 ORM metadata parity
- legacy row 존재 시 migration 중단 및 데이터 보존
- 중단 시 부분 DDL 미잔존
- SQLite 외래 키 활성화
- upgrade head 및 downgrade base
