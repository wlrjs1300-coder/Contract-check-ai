# v0.7.5 저장 암호화 migration 설계

## 기본 정보

- 버전: v0.7.5
- 브랜치: feat/v0.7.5-encryption-key-management
- 현재 단계: 설계 초안 검토 중

## 목적과 범위

현재 문서는 v0.7.5 구현 전에 migration 기준을 확정하기 위한 문서이다. v0.7.5 전체에는 migration 구현과 테스트가 포함된다. 정확한 컬럼 구조와 revision은 아직 확정하지 않는다.

기존 revision은 다음과 같다.

- `0001_current_schema_baseline`
- `0002_auth_ownership`

다음 revision 후보는 `0003_data_at_rest_encryption`이며, 이름과 구조는 구현 검토 후 확정한다.

## 기본 전제

- 현재 개발 DB에는 실제 개인정보가 없다는 전제를 검증한다.
- 민감 row가 확인되면 migration을 fail-closed로 중단한다.
- production의 평문 dual-read fallback은 기본적으로 금지한다.
- 복호화 실패 시 평문으로 자동 fallback하지 않는다.
- migration과 테스트는 합성 데이터만 사용한다.

## 전략 비교

### A. 개발 DB 초기화 후 encrypted schema 전환

장점:

- 구현이 가장 단순하다.
- 평문 fallback이 필요하지 않다.
- 실제 개인정보가 없는 현재 단계에 적합하다.

단점:

- 기존 합성 개발 데이터가 삭제된다.

### B. 합성 데이터 one-time re-encryption

장점:

- migration 경로를 실제로 검증할 수 있다.
- 합성 fixture 성격의 개발 데이터를 보존할 수 있다.

단점:

- script의 idempotency, 부분 실패, 재실행 안전성을 검증해야 한다.

### C. 실데이터 자동 backfill

현재 범위 밖이다. 실제 데이터와 운영 승인이 없으며 부분 실패, rollback, 감사, backup 통제까지 별도 설계가 필요하다.

## 권장 전략

- v0.7.5에서는 A를 기본 전략으로 채택한다.
- 필요하면 B를 합성 데이터 전용으로 제한한다.
- C는 운영 단계 전에 별도 보안 승인을 받아야 한다.
- 보관형 실운영은 v0.7.5 migration 완료만으로 승인되지 않는다.

## 전환 불변 조건

nullable encrypted column을 검토할 경우 다음 조건을 모두 적용한다.

- plaintext와 encrypted 값의 동시 존재를 금지한다.
- exactly-one application validation 또는 DB constraint를 검토한다.
- encrypted 값이 없고 plaintext가 있는 production row는 fail-closed 처리한다.
- migration 완료 뒤 plaintext column 제거 계획을 둔다.
- 장기 dual-write를 금지한다.
- plaintext fallback에는 별도 승인, 종료 기한, telemetry, 삭제 조건이 모두 필요하다.
- 복호화 실패 시 내부 컬럼, ciphertext, nonce, key 정보, 평문을 노출하지 않는다.

## 대상과 순서

1. A/B/C/D 분류와 schema별 allowlist 확정
2. crypto core와 keyring 검증 완료
3. encrypted backing column 구조 확정
4. 개발 DB의 실제 개인정보 부재 확인
5. A 전략 적용 또는 합성 데이터 전용 B 전략 승인
6. upgrade·downgrade·재실행·부분 실패 검증
7. raw DB 평문 부재와 API 회귀 검증
8. plaintext column 제거 및 잔여 데이터 확인

## 실패 처리

- keyring, envelope, owner context가 유효하지 않으면 migration을 중단한다.
- 부분 암호화 상태를 정상 완료로 표시하지 않는다.
- 실패 시 raw SQL, 원문, 파일명, 경로, key, 내부 예외를 로그나 응답에 남기지 않는다.
- rollback 뒤에도 평문·암호문 동시 저장이나 장기 dual-write 상태가 남지 않아야 한다.
- 재실행은 합성 데이터에서 idempotency를 검증한 뒤 허용한다.

## 구현 전 미확정 사항

- 정확한 encrypted 컬럼 형식
- JSON envelope의 DB 타입
- 조건부 B등급 필드의 적용 범위
- filename 호환 방식
- revision 이름과 upgrade/downgrade 구현
- 제한된 재암호화 helper의 batch 크기와 transaction 경계

## 검증 기준

- 실제 개인정보가 있는 DB에서 fail-closed
- 합성 개발 DB 초기화 경로 통과
- 선택 시 합성 one-time re-encryption의 idempotency
- production 평문 fallback 거부
- 평문·암호문 동시 저장 거부
- migration 전후 API 계약 유지
- raw DB에서 A등급 평문 검색 실패
- downgrade와 실패 응답의 민감정보 비노출
