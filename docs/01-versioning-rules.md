# 버전 관리 규칙

## 1. 목적

ContractCheck AI 프로젝트는 작업을 버전 단위로 나누어 관리한다.

각 버전은 하나의 명확한 목적, 작업 범위, 검증 기록, 릴리스 기록을 가진다.

## 2. 버전 형식

```text
v<major>.<minor>.<patch>
```

예시:

```text
v0.0.1
v0.1.3
v0.2.5
v1.0.4
```

## 3. 버전 의미

| 구분 | 의미 |
|---|---|
| major | 큰 릴리스 단위. MVP 구현 완료 시 v1.0 사용 |
| minor | 큰 작업 영역 단위 |
| patch | 세부 작업 단위 |

## 4. 큰 흐름

```text
v0.0 프로젝트 운영 준비
v0.1 서비스 기획
v0.2 개인정보/보안 설계
v0.3 화면/UX 설계
v0.4 DB/API 설계
v0.5 분석 파이프라인 설계
v0.6 개발 환경 세팅
v1.0 MVP 구현
```

## 6. 버전 계획 운영 원칙

버전 계획은 고정된 계약이 아니라 실제 진행 상태를 반영하는 기준표로 관리한다.

- 버전 번호와 작업명은 실제 진행에 따라 추가, 통합, 이동, 축소할 수 있다.
- 세부 버전은 필요 시 통합, 이동, 추가, 축소할 수 있다.
- 모든 계획 항목에 반드시 개별 체크리스트를 만드는 것은 아니다.
- 실제 독립 산출물이나 검토 단위가 있을 때만 별도 체크리스트와 브랜치를 만든다.
- 서로 밀접한 항목은 한 버전 또는 한 PR로 통합할 수 있다.
- 계획이 변경되면 변경 전 계획, 변경 후 계획, 변경 사유, 영향 범위, 다음 작업을 문서에 남긴다.
- 과거 단계에서 발견된 미완료 항목이나 보완 이슈는 반드시 별도 버전으로 분리하지 않아도 된다.
- 보완 이슈는 관련성이 높은 다음 작업에 통합할 수 있다.
- `main` 병합과 태그 생성 여부는 실제 GitHub 상태를 기준으로 기록한다.

|---|---|

v0.5.1은 파이프라인 전체 구조와 텍스트 추출 방식을 다룬다.

1. AI 응답 생성
2. JSON 스키마 검증
3. reference_id 검증
4. 금지 표현 및 법률 확정·보장 의미 검사
5. 검증 실패 시 차단, 재작성, 재시도 또는 실패 처리
6. 안전한 결과만 표시 및 저장

본 항목의 상세 설계는 v0.2.1 핵심 기술 스파이크 결과에 의존한다.

변경 사유:

- 문서 단계가 지나치게 세분화되어 실제 코드 검증 전에 문서만 늘어나는 위험이 확인되었다.
- 한국어 개인정보 탐지, 조항 분할 정확도, AI 공급자 데이터 정책처럼 실험으로 먼저 확인해야 정확해지는 항목이 있음에도 이를 검증할 공식 단계가 로드맵에 없었다.
- 서로 밀접하게 연관된 세부 항목을 하나의 산출물로 묶는 것이 검토와 유지보수에 더 효율적이다.

영향 범위:

- v0.2~v0.5의 세부 버전 번호와 작업명이 위 표와 같이 변경된다.
- 각 통합 문서 내부에는 기존 세부 항목이 체크리스트 단위로 그대로 유지되며, 삭제되지 않는다.
- v0.6, v1.0의 큰 목적은 변경하지 않는다.

유지되는 세부 항목:

- 위 표의 "변경 전 계획" 25개 항목은 모두 해당 통합 문서의 내부 체크리스트 범위로 이어진다.

## 15. 변경 가능성

본 버전 계획은 기준표이며, 개발 과정에서 변경될 수 있다.

변경 시 다음 내용을 체크리스트에 기록한다.

- 변경 전 계획
- 변경 후 계획
- 변경 사유
- 영향 범위
- 다음 작업

## Checklist-first workflow

새 작업을 시작하기 전에는 반드시 현재 버전의 체크리스트를 먼저 확인한다.

확인해야 할 내용은 다음과 같다.

- 현재 진행 중인 버전
- 이전에 완료된 작업
- 아직 완료되지 않은 작업
- 이번 작업에서 진행할 범위
- 다음 단계로 이동 가능한지 여부

각 작업이 끝나면 해당 체크리스트를 최신화하고, 필요한 경우 커밋 메시지를 함께 정리한다.

각 버전의 마지막 번호는 final-review 단계로 사용한다.  
final-review 단계에서는 해당 버전에서 작성한 문서, 구현한 기능, GitHub 반영 상태, 다음 버전 이동 가능 여부를 최종 점검한다.
