# ADR-011: Contract Data Lifecycle and Processing Modes

## 상태
제안(Approved)

## 배경
현재 v0.7.x 단계에서 계약서 데이터의 수집, 추출, 분석, 보관, 삭제 경계가 모드별로 일관되게 설명되어 있지 않았다. 특히 원본/중간 데이터 보존 범위와 외부 Provider 전달 범위를 분리하지 않으면 개인정보 노출과 삭제 실패 대응이 불명확해진다. 본 결정은 일회성 분석을 기본으로 두고, 보관형은 강한 전제 조건에서만 허용하도록 하는 것을 목표로 한다.

## 결정
- 기본 모드는 `일회성 안전 분석`으로 운영한다.
- 업로드 원본과 OCR/추출 과정 중간 데이터는 기본적으로 영구 저장하지 않는다.
- 분석 결과 저장은 raw 원본 저장과 분리된 경로/구조로 운영한다.
- 보관형 모드는 인증, 소유권/접근 통제, 암호화가 완료된 조건에서만 허용한다.
- 외부 Provider에는 마스킹된 최소 데이터만 전달한다.
- 잔존 PII 또는 고위험 조건에서 fail-closed를 적용한다.
- 실제 개인정보 실사용은 보안 게이트 통과 전에는 허용하지 않는다.
- 외부 Provider와 로컬 Provider를 교체 가능한 경계로 유지한다.

## 처리 모드
- 일회성 안전 분석
  - 기본값: `single-pass` 안전 모드.
  - 목적: 분석 결과 산출 후 최소 잔존.
  - 삭제 우선: 분석 완료 후 원본 및 임시 산출물의 정리 절차 실행.
- 보관형 계약 관리
  - 조건: 인증, 소유권 확인, 암호화, 접근권한 정책 모두 충족 시 활성화.
  - 목적: 반복 조회·비교·재분석이 필요한 장기 보관.
  - 보존 대상: 정책에서 허용된 데이터만 제한 기간 보관.

## 외부 Provider 경계
- Provider 입력은 `masked_` 계열의 최소 필드로 제한한다.
- 내부 원본/추출 텍스트를 전달하지 않는다.
- 민감 정보 탐지 정책에 걸린 항목은 Provider 호출을 차단할 수 있다.
- Provider 출력은 내부 정책 스키마로 재검증 후 보존한다.
- Provider 교체 시 동일 보안 경계(마스킹, 재검증, 실패 처리)를 반드시 준수한다.

## 대안 비교
1. 기본 영구 보존
  - 장점: 사후 분석/분쟁 대응 용이
  - 단점: 삭제 및 접근 통제 부담이 커지고 개인정보 유출 노출면적이 커짐
2. 완전 비보관(권장)
  - 장점: 최소 노출, 규제 대응성 우수
  - 단점: 추후 재분석과 감사 대응 비용 증가
3. 게이트 기반 선택 모드
  - 장점: 기본은 안전하고 필요 시 보관형을 제한 허용
  - 단점: 운영 승인 과정이 추가되나 통제 가능성 확보

## 선택 이유
- 제품 목적이 “안전 분석” 중심으로 빠른 가치 전달이므로 기본 비보관이 적합하다.
- 보안 요구를 충족하면서도 추후 계약 관리 확장을 위해 모드 전환 점을 미리 열어둔다.
- Provider 교체 유연성을 유지해 운영 리스크를 관리한다.

## 결과
- 일회성 안전 분석을 기본 모드로 결정한다.
- 원본 및 중간 데이터의 영구 보존을 금지하고 보관형 조건을 분리한다.
- 분석 결과 저장은 원본 저장과 논리적으로 분리한다.
- 외부 Provider는 마스킹된 데이터만 전달하고 실패·개인정보 경보에서 차단을 우선한다.

## 후속 작업
- 일회성 모드 정합성 문서(v0.7.2-data-lifecycle-security)를 기준으로 삭제 확인 지표를 정비한다.
- 보관형 모드 승인 체크리스트를 운영 정책과 연동한다.
- 고위험 실패 상태에서의 운영 알림 템플릿을 최소 권한으로 표준화한다.
