# ADR-006: OCR 처리 전략

- Status: Proposed
- Date: 2026-07-19
- Scope: PDF·JPG·JPEG·PNG의 텍스트 추출 경계

## Context

현재 서비스는 UTF-8 TXT만 받아 조항 분리와 합성 분석 흐름을 검증한다. 실제 사용자가 보유한 계약 문서는 텍스트 레이어 PDF, 스캔 PDF, 휴대전화로 촬영한 이미지가 섞일 수 있다. v0.6은 원본 파일을 안전하게 처리하면서 페이지 순서와 원문 위치를 보존하고, 사용자가 추출 텍스트를 확인·수정할 수 있는 기반이 필요하다.

OCR은 개인정보가 포함된 원본 또는 파생 이미지에 접근하고 CPU·메모리·비용을 크게 사용할 수 있다. 따라서 엔진 이름을 먼저 고정하지 않고 처리 위치와 교체 경계를 먼저 결정한다.

## Decision drivers

- 한국어 계약 문장의 인식 품질
- 표, 조항 번호, 들여쓰기와 복수 단 보존 가능성
- Windows 개발 환경에서의 설치·재현성
- Linux 배포 환경에서의 패키징·관측 가능성
- 원본 계약서와 개인정보의 외부 전송 최소화
- CPU, 메모리, 임시 디스크와 동시 처리 자원
- native binary, 언어팩과 모델 설치 난이도
- 요청당 비용과 운영 예산의 예측 가능성
- 다중 페이지와 여러 이미지의 순서·부분 실패 처리
- EXIF 회전, 기울기, 저해상도와 촬영 노이즈 처리
- confidence와 bbox 등 공통 결과 계약 지원
- 장애 시 대체 구현으로 전환할 수 있는 adapter 경계

## Options considered

### Option A: 서버 내부 OCR

서비스가 관리하는 실행 환경에서 OCR 엔진과 전처리를 수행한다.

장점:

- 원본과 파생 이미지가 외부 서비스 경계를 넘지 않는다.
- 요청 단위 외부 전송 동의와 공급자 보존 정책 의존을 줄인다.
- 합성 fixture 기반의 결정론적 회귀 테스트와 버전 고정이 상대적으로 쉽다.
- 네트워크 장애와 외부 rate limit에서 독립적이다.
- 페이지 단위 전처리와 중간 산출물 삭제를 직접 통제할 수 있다.

단점:

- 한국어 언어팩, native binary, 모델과 OS별 설치 관리가 필요하다.
- CPU·메모리 사용량과 동시성 제한을 직접 운영해야 한다.
- 표·레이아웃·저품질 사진 품질이 요구 수준에 못 미칠 수 있다.
- 모델 취약점과 공급망 업데이트를 서비스 운영자가 관리해야 한다.
- serverless 또는 작은 인스턴스에서는 실행 시간·디스크 제한과 맞지 않을 수 있다.

### Option B: 외부 OCR API

암호화된 네트워크를 통해 원본 또는 파생 페이지를 외부 OCR 서비스에 보내 결과를 받는다.

장점:

- 설치와 모델 운영 부담이 작고 확장된 문서·레이아웃 기능을 빠르게 시험할 수 있다.
- 다중 페이지, 표, 방향 인식과 한국어 품질이 더 나을 가능성이 있다.
- 서비스가 제공하는 확장성과 관측 기능을 활용할 수 있다.

단점:

- 계약 원문과 개인정보가 제3자 처리 경계로 이동한다.
- 보존, 학습 사용, 처리 지역, 하위 처리자와 삭제 정책을 계약으로 확인해야 한다.
- 사용자 고지와 명시적 동의 없이 기본 경로로 사용할 수 없다.
- 호출 비용, rate limit, timeout, 공급자 장애와 정책 변경에 종속된다.
- provider별 bbox·confidence·페이지 모델 차이를 정규화해야 한다.
- 민감 문서의 재시도와 지원 로그가 추가 사본을 만들 수 있다.

### Option C: 혼합 방식

직접 추출과 서버 내부 OCR을 기본으로 하고, 품질 부족 또는 사용자가 선택한 경우에만 외부 OCR을 별도 경로로 사용한다.

장점:

- 기본 개인정보 노출을 최소화하면서 어려운 문서의 대체 경로를 둘 수 있다.
- 비용을 제한하고 여러 엔진의 품질을 비교할 수 있다.
- adapter를 통해 내부·외부 구현을 교체하기 쉽다.

단점:

- 품질 판정, 사용자 동의, 경로 선택과 재시도 상태가 복잡해진다.
- 같은 문서에서 엔진별 결과 차이와 페이지 순서를 관리해야 한다.
- 외부 전송 경계가 UI·API·감사 기록에 명확히 드러나야 한다.
- 두 운영 방식을 동시에 유지하면 테스트와 장애 대응 비용이 커진다.

## Proposed direction

1차 권장 방향은 **텍스트 PDF 직접 추출 + 서버 내부 OCR 우선 스파이크 + 교체 가능한 OCR adapter**이다. 이 방향은 최종 OCR 엔진 채택이 아니다. 한국어, 표와 촬영 이미지 품질이 최소 기준을 충족하는지 먼저 검증한다.

내부 OCR이 품질 또는 운영 자원 기준을 충족하지 못하면 혼합 방식을 다음 후보로 검토한다. 외부 OCR은 자동 fallback이 아니라 사용자에게 전송 대상, 목적, 처리자, 보존·삭제와 대안을 알리고 별도 동의를 받은 요청에만 허용한다.

### v0.6.6 구현 상태

Phase 3 기반에서 정리된 `OcrAdapter` 구조를 유지하면서 v0.6.6에서 `LocalKoreanOcrAdapter`를 추가해 로컬 OCR 실행 경로를 연동했다.

`OCR_ADAPTER` 설정이 `local`, `local_korean`, `tesseract`일 때만 로컬 OCR 경로가 활성화되며, 기본 동작은 non-test 환경에서 `ocr_unavailable`로 실패한다. 테스트 환경(`APP_ENV=test`)에서는 이전 정책대로 기본 동작은 synthetic 경로다.

실행 가능한 기반은 Tesseract 의존성 기반이며, 품질 스파이크는 별도 평가셋에서 계속 수행한다. Engine/모델 누락 등은 명시 오류 코드로 구분된다. Thread timeout은 응답 대기 제어에 한정되므로 process 격리와 동시성 제한은 후속 보안/운영 과제다.

## External transfer conditions

외부 OCR 호출을 검토하려면 최소한 다음 조건이 충족되어야 한다.

- 전송 전 명확한 고지와 요청 단위의 명시적 동의
- 동의 거부 시 내부 처리 또는 중단 경로 제공
- 전송 데이터 최소화와 불필요한 메타데이터 제거
- 전송 중 암호화와 인증 정보의 안전한 관리
- 입력의 학습 사용 여부와 opt-out 가능성 확인
- 기본 저장 여부, 보존 기간, 즉시 삭제 기능과 삭제 증빙 확인
- 처리 지역, 하위 처리자, 접근 통제와 사고 통지 조건 확인
- 계약상 데이터 처리 조건과 책임 주체 확인
- timeout, 재시도, 중복 전송, 장애 시 폐기 정책
- 원문·응답·지원 로그에 민감정보를 남기지 않는 관측 설계

이 조건이 확인되지 않으면 외부 전송을 구현하지 않는다.

## Consequences

- OCR 호출부는 provider별 타입을 도메인에 노출하지 않는 adapter가 필요하다.
- 직접 추출과 OCR 추출을 같은 페이지·블록 모델로 정규화해야 한다.
- 문서 페이지별 처리 방식, 경고와 실패를 보존해야 한다.
- OCR 경고가 있는 페이지는 사용자 검토 필요 상태로 두고, 확인 전에는 분석 차단을 유지해야 한다.
- 내부 OCR 배포 이미지 또는 호스트 의존성이 늘 수 있으며 자원 격리와 timeout이 필수다.
- 외부 OCR을 나중에 추가해도 기본 경로로 자동 활성화하지 않는다.
- 사용자가 확인한 텍스트만 조항 분리와 분석 입력이 된다.
- confidence는 품질 신호로만 쓰며 정확성이나 안전을 보장하지 않는다.
- 원본 파일과 처리용 파생 파일의 삭제 성공이 검증되기 전에는 분석 가능 상태로 전환하지 않는다.

## Security implications

- parser와 decoder는 신뢰할 수 없는 입력을 처리하므로 격리, 자원 제한, 보안 업데이트와 실패 폐기가 필요하다.
- 원본, 렌더링 이미지와 마스킹 전 추출 텍스트는 민감정보로 동일하게 취급한다.
- 삭제 검증 실패 항목은 격리하고 운영 경보와 재삭제 대상으로 두며, 별도 보안 승인 없이는 이 차단 정책을 완화하지 않는다.
- 문서 내용은 OCR 또는 분석 시스템을 제어하는 지시가 아니라 데이터로만 처리한다.
- 로그에는 원문, 원본 파일명, bbox 내부 텍스트와 외부 응답 원문을 남기지 않는다.
- 외부 전송에는 별도 동의와 감사 가능한 메타데이터가 필요하다.

## Rollback / replacement strategy

- OCR adapter의 입력은 정규화된 문서 페이지, 출력은 공통 extraction model로 제한한다.
- 엔진별 confidence와 bbox는 optional metadata로 격리하고 핵심 분석 조건으로 직접 사용하지 않는다.
- 엔진 변경 시 동일 합성 fixture와 고정 평가 rubric을 다시 실행한다.
- 신규 엔진 품질·삭제·자원 검증이 실패하면 기능 flag 또는 설정으로 해당 경로를 비활성화한다.
- 외부 OCR 장애나 정책 변경 시 자동으로 다른 외부 서비스에 원본을 보내지 않는다. 내부 경로 또는 사용자 재시도 안내로 되돌린다.
- 저장된 사용자 확인 텍스트가 있다면 원본 재전송 없이 후속 조항 분리를 재개할 수 있는지 보존 정책과 함께 검토한다.
