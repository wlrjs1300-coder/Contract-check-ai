# ContractCheck AI 프로젝트 개요

## 프로젝트 개요

ContractCheck AI는 계약 문서를 조항 단위로 분리하고 개인정보 보호와 출력 검증 경계를 거쳐 검토 신호를 보여 주는 계약서 리스크 관리 기술 검증 MVP다.

- 형태: 개인 프로젝트
- 현재 상태: 개발 중 · 기술 검증 MVP
- 현재 입력: UTF-8 TXT 한 파일, 최대 1 MiB
- 현재 분석: `SyntheticAnalysisProvider`의 합성 결과
- 현재 검증 범위: 로컬 Frontend·Backend·SQLite 통합 흐름

실제 외부 분석 서비스, 법률 서비스, 인증과 운영 배포는 현재 범위에 포함되지 않는다.

## 개발 배경

계약 문서는 길고 구조가 다양해 일반 사용자가 검토 지점을 빠르게 찾기 어렵다. 문서 전체를 불투명한 단일 입력과 출력으로 처리하면 결과의 근거 조항을 확인하기 어렵고, 개인정보가 외부 분석 경계로 전달되거나 결과에 다시 나타날 위험도 있다.

이 프로젝트는 분석 모델의 성능보다 먼저 문서 입력, 조항 식별, 개인정보 마스킹, Provider 격리, 결과 검증과 사용자 화면 연결을 하나의 검증 가능한 흐름으로 만드는 데 집중했다.

## 해결하려는 문제

1. 문서를 조항 단위로 나누어 결과 근거를 확인할 수 있게 한다.
2. `clause_id`와 `reference_id`로 원문 조항과 결과의 잘못된 연결을 차단한다.
3. Provider 입력 전에 개인정보를 탐지·마스킹하고 잔여 개인정보를 다시 검사한다.
4. Provider 출력도 신뢰하지 않고 schema, 개인정보 재생성과 법률 확정 표현을 검사한다.
5. 실패 시 검증되지 않은 부분 결과가 남지 않도록 transaction을 rollback한다.
6. 동기 Backend의 실제 응답 경계에 맞춰 Frontend 상태와 재시도 흐름을 제공한다.

## 담당 범위

개인 프로젝트로 다음 범위를 설계·구현·검증했다.

- 서비스 및 MVP 범위 기획
- FastAPI API와 SQLAlchemy 데이터 모델
- TXT 검증과 조항 분할
- 분석 작업 상태와 결과 저장 흐름
- Provider 인터페이스와 합성 Provider
- 개인정보 탐지·마스킹과 출력 안전성
- React Frontend와 API 연동
- Frontend·Backend 자동 테스트
- Git 브랜치·PR 기반 버전 관리 문서화
- 실제 브라우저 통합 검증

## 핵심 구현

### 문서 처리

- `.txt` 확장자, 빈 파일, 최대 1 MiB와 UTF-8 검증
- 문단 marker 기반 조항 분리
- 조항별 순서, 유형, 제목, 본문, 경고와 식별자 생성
- 미분류 구간과 문서 경고를 별도 보존

### 분석 파이프라인

- 분석 작업의 `queued` → `processing` → `completed` 또는 `failed` 전이
- 조항 단위 Provider 최소 입력 생성
- Provider 실행 오류의 재시도 가능 여부 분류
- 결과 schema와 허용 라벨 검증
- 조항과 결과의 `reference_id` 일치 검증
- 실패 시 부분 결과 rollback과 `failed` 상태 보존

### 개인정보와 출력 안전성

- 규칙 기반 개인정보 탐지와 토큰 마스킹
- 기존 마스킹 토큰 충돌 회피
- Provider 전달 직전 잔여 개인정보 검사
- Provider 요약의 개인정보 재생성 검사
- 법률 확정·보장 의미와 직접 행동 지시 검사
- 모든 검증을 통과한 결과만 저장

### Frontend

- 파일 사전 검증과 안전한 오류 매핑
- 문서 메타데이터, 조항, 경고와 미분류 영역 표시
- 분석 요청과 네 가지 상태 표시
- 자동 폴링 대신 사용자가 요청하는 단발 상태 조회
- 완료 결과와 업로드 조항의 일대일 연결 검증
- 새 문서 선택 시 이전 상태 초기화
- 모바일 레이아웃과 접근성 상태 제공

## 기술적 의사결정

### TXT로 MVP 입력을 제한한 이유

PDF 파서와 OCR의 정확도 문제를 분석 파이프라인 검증과 분리하기 위해 UTF-8 TXT 한 파일로 범위를 제한했다. TXT는 최종 제품 방향이 아니라 현재 API와 안전성 경계를 검증하기 위한 입력이다.

### 조항 단위 분석을 선택한 이유

전체 문서 단위 결과보다 어느 조항에서 결과가 나왔는지 추적하기 쉽다. 조항별 실패와 결과 매핑을 테스트할 수 있고, Provider에 전달하는 텍스트 범위도 줄일 수 있다.

### `reference_id`를 검증한 이유

결과 형식이 올바르더라도 다른 조항의 결과가 연결되면 사용자에게 잘못된 근거를 보여 줄 수 있다. 문서 ID와 조항 순서를 기반으로 한 식별자를 파이프라인과 Frontend 양쪽에서 검증한다.

### Provider 입력 전에 마스킹한 이유

외부 경계로 전달될 수 있는 정보의 양을 줄이기 위해 조항 본문을 그대로 넘기지 않는다. 탐지·마스킹 후 잔여 개인정보를 재검사하며 탐지가 완전하다고 가정하지 않는다.

### Provider 출력도 검증한 이유

Provider는 schema에 맞지 않는 값, 다른 조항 식별자, 개인정보 또는 법률 확정 표현을 반환할 수 있다. 따라서 출력은 저장 전에 schema, 참조, 개인정보와 표현 안전성을 모두 통과해야 한다.

### 합성 Provider로 먼저 검증한 이유

실제 외부 전송, 비용, 공급자 정책과 모델 품질을 도입하기 전에 API, DB transaction, 오류 처리와 화면 흐름을 결정론적으로 검증할 수 있다. 현재 결과는 실제 분석 품질 평가 자료가 아니다.

### 자동 폴링을 도입하지 않은 이유

현재 분석 작업 생성 API는 같은 HTTP 요청 안에서 파이프라인을 동기 실행한다. 정상 흐름에서 `completed`가 바로 반환될 수 있으므로 장시간 자동 폴링을 기본 동작으로 만들지 않고, 중간 상태가 관찰될 때만 수동 재조회를 제공한다.

### 부분 결과를 rollback하는 이유

여러 조항 중 일부만 검증된 상태로 노출되면 완전한 분석으로 오해될 수 있다. 하나의 조항이라도 실패하면 transaction을 rollback하고 작업 실패 상태만 보존한다.

## 어려웠던 점과 해결

### Provider 출력의 개인정보 재생성

입력을 마스킹하더라도 출력이 개인정보 형태를 다시 만들 가능성을 별도 위험으로 다뤘다. 결과 요약에 탐지 로직을 다시 적용하고 검출 시 전체 결과 저장을 중단했다.

### 조항과 결과의 잘못된 연결

Backend에서는 `reference_id`를 검증하고 Frontend에서는 `clause_id`와 `reference_id`, 결과 개수와 중복을 함께 검사하도록 구성했다.

### 분석 실패 시 부분 저장

파이프라인 전체를 하나의 transaction 경계로 처리했다. 예외 시 rollback 후 기존 작업을 다시 조회해 `failed`로 저장한다.

### 동기 작업과 화면 상태의 차이

HTTP 요청 로딩 상태와 서버의 `processing` 상태를 분리해 표현했다. 실패 JSON과 작업 ID가 항상 제공되지 않는 현재 API 한계도 Frontend 오류 분기에 반영했다.

### 320px viewport 오버플로

실제 Chrome 통합 검증에서 `body` 최소 너비와 세로 스크롤바의 조합을 원인으로 확인했다. 고정 최소 너비를 제거하고 320px 회귀 테스트를 추가했다.

## 현재 상태

**개발 중 · 기술 검증 MVP**

Frontend, Backend와 SQLite의 핵심 흐름은 합성 데이터로 통합 검증했다. 공개 저장소에서 구조와 설계 의도를 설명할 수 있는 단계이지만 실제 계약서를 처리하는 운영 서비스는 아니다.

## 현재 한계

- UTF-8 TXT 한 파일과 최대 1 MiB 제한
- PDF·OCR 미지원
- 실제 외부 Provider와 실제 분석 품질 검증 없음
- 인증·인가와 사용자별 데이터 격리 없음
- 분리된 조항 본문이 SQLite에 저장됨
- 동기 분석과 자동 폴링·작업 복구 없음
- 실패 사유 DB 저장 없음
- Reviewer workflow 없음
- 운영 배포와 보관·삭제 정책 미완료
- 실제 screen reader와 전문 색상 대비 검증 미완료

### 300자 안팎 상세 소개

ContractCheck AI는 계약 문서 분석에서 입력과 출력 모두를 신뢰하지 않는 구조를 검증한 개인 프로젝트입니다. UTF-8 TXT를 조항 단위로 분리하고, Provider 전달 전에 개인정보를 마스킹하며, 반환 결과의 schema·조항 식별자·개인정보 재생성·법률 확정 표현을 검사합니다.
