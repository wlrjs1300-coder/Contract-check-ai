# 검토 워크플로우 규칙

## 1. 목적

본 문서는 ContractCheck AI 프로젝트에서 작업 단계별 검토 기준을 정의한다.

매 작업마다 과도한 수동 검토를 반복하지 않도록, 작업 유형에 따라 확인해야 할 범위와 다음 단계 이동 기준을 명확히 한다.

## 2. 기본 원칙

- 모든 작업은 현재 버전 체크리스트를 먼저 확인한 뒤 진행한다.
- 작업 범위는 현재 버전의 목적과 체크리스트에 맞춘다.
- 작업이 끝나면 체크리스트를 최신화하고 변경 내용을 검토한다.
- 일반 작업은 검토 후 커밋, 작업 브랜치 push, `develop` 대상 PR 순서로 진행한다.
- `main` 병합과 태그 생성은 큰 단계 또는 정식 릴리스에서만 진행한다.
- 공개 저장소에는 최종 코드, 설계 문서, 검증 기록만 남긴다.
- 작업 중간 산출물, 임시 메모, 검토 전 초안은 공개 저장소에 포함하지 않는다.

## 3. 검토 수준 구분

### 문서/기획 작업

- [ ] 현재 버전 체크리스트를 확인했는가?
- [ ] 문서의 목적과 대상 독자가 명확한가?
- [ ] 프로젝트 방향, 서비스 포지션, 기존 규칙 문서와 충돌하지 않는가?
- [ ] 법률 자문 또는 실사용 법률 검토 대체로 오해될 표현이 없는가?
- [ ] 공개 저장소에 남겨도 되는 정제된 문장으로 작성했는가?
- [ ] 작업 중간 산출물이나 검토 전 초안이 포함되지 않았는가?
- [ ] 체크리스트의 완료 항목과 남은 항목을 최신화했는가?

### 설계 작업

- [ ] 현재 버전 체크리스트와 설계 범위가 일치하는가?
- [ ] 설계 결정의 목적과 이유가 문서에 남아 있는가?
- [ ] 개인정보 보호, 원본 파일 처리, 로그 보안 기준을 고려했는가?
- [ ] 구현 단계에서 참고할 수 있을 만큼 입력, 출력, 책임 범위가 명확한가?
- [ ] 아직 확정하지 않은 내용은 남은 이슈로 분리했는가?
- [ ] 기존 문서나 후속 단계에 영향을 주는 변경 사항을 기록했는가?

### 구현 작업

- [ ] 현재 버전 체크리스트와 구현 범위가 일치하는가?
- [ ] 변경 파일이 작업 목적에 필요한 범위로 제한되어 있는가?
- [ ] 기존 폴더 구조, 네이밍, 코드 스타일을 따랐는가?
- [ ] 원본 계약서, 개인정보, API Key, Secret, Token이 코드나 로그에 포함되지 않는가?
- [ ] 주요 성공 흐름과 예외 흐름을 확인했는가?
- [ ] 필요한 테스트 또는 수동 확인을 수행했는가?
- [ ] 구현 결과와 검증 결과를 체크리스트에 반영했는가?

### 릴리스 작업

#### 일반 작업 검토

- [ ] 작업 범위가 현재 체크리스트와 일치하는가?
- [ ] 문서 또는 코드 품질을 확인했는가?
- [ ] 보안 및 공개 기록 기준을 확인했는가?
- [ ] Git diff로 변경 범위를 확인했는가?
- [ ] `develop` 대상 PR을 준비했는가?

#### 정식 릴리스 검토

- [ ] 현재 버전 체크리스트가 작성 완료 상태인가?
- [ ] `develop` 기준 통합 검증을 완료했는가?
- [ ] `develop`에서 `main`으로 PR을 생성했는가?
- [ ] `main` 병합을 완료했는가?
- [ ] `main` 병합 후 안정 상태를 검증했는가?
- [ ] `main` 최신 커밋 기준으로 태그를 생성하고 push했는가?
- [ ] 커밋 메시지가 작업 목적을 설명하는가?
- [ ] PR 제목과 본문이 변경 내용, 검증 결과, 남은 이슈를 포함하는가?

Git 명령의 세부 확인 목록은 `docs/06-checklist-template.md`의 Git 검증 섹션을 기준으로 한다.

## 4. 사용자 직접 확인 기준

사용자가 매번 모든 문장을 직접 검토하지 않아도 되도록, 직접 확인 범위는 다음 기준을 중심으로 한다.

- 작업 의도: 이번 작업이 현재 버전 목적과 맞는가?
- 변경 범위: 변경 파일이 요청한 범위를 벗어나지 않았는가?
- 공개 가능 여부: 공개 GitHub 저장소에 남겨도 되는 내용인가?
- PR 변경 파일 수: PR에 포함된 파일 수와 경로가 예상 범위 안인가?

세부 문장 검토, 누락 항목 확인, 커밋/PR 문구 정리는 보조 검토를 통해 진행한다.

## 5. PR 전 필수 확인

PR을 만들기 전에는 다음 항목을 확인한다.

- [ ] 변경 파일 수를 확인했다.
- [ ] 변경 파일 경로를 확인했다.
- [ ] 민감정보 포함 여부를 확인했다.
- [ ] 작업 중간 산출물 포함 여부를 확인했다.
- [ ] 자동 삽입된 공동 작성자 문구 포함 여부를 확인했다.
- [ ] 체크리스트 최신화 여부를 확인했다.

## 6. 다음 단계 이동 기준

다음 버전 또는 다음 작업 단계로 이동하기 전에는 다음 조건을 만족해야 한다.

- [ ] 현재 버전 체크리스트 작성이 완료되었다.
- [ ] 변경 파일 범위가 적절하다.
- [ ] 공개 저장소에 남겨도 되는 내용만 포함되어 있다.
- [ ] Git 상태가 clean이다.
- [ ] 필요한 일반 작업 PR 또는 정식 릴리스 절차가 완료되었다.

## 7. 운영 기준

- 작은 문서 수정은 문서/기획 작업 기준으로 검토한다.
- 아키텍처, API, DB, 보안 정책 변경은 설계 작업 기준으로 검토한다.
- 실제 코드 변경은 구현 작업 기준으로 검토한다.
- 커밋, PR, 병합 작업은 일반 작업 검토 또는 정식 릴리스 검토 기준으로 구분해 확인한다.
- 태그 작업은 정식 릴리스 검토 기준으로만 확인한다.
- 여러 유형이 섞인 작업은 가장 높은 검토 수준을 적용한다.
