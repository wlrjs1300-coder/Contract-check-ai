# v0.6.2 결제·구독 도메인 초안

## 1. 범위

향후 결제·구독·권한을 구현하기 위한 도메인 경계를 정의한다. 실제 Provider, DB migration, API, 회원가입, 인증·인가, webhook endpoint와 환불 처리는 구현하지 않는다. Provider는 선정하지 않았고 가격도 확정하지 않는다.

## 2. 핵심 원칙

- 결제 상태, 상품 권한, 문서 접근 권한을 분리한다.
- `is_paid` boolean 하나로 권한을 표현하지 않는다.
- 결제 성공은 권한 부여 요청의 근거이지 authorization 성공 자체가 아니다.
- webhook은 Provider 상태 동기화 수단이며 서명 검증과 idempotency 없이는 반영하지 않는다.
- 계약 원문·추출 텍스트·분석 결과를 결제 Provider에 전송하지 않는다.
- 결제 이력의 법정 보존과 계약 문서의 개인정보 보존을 분리한다.

## 3. 도메인 후보

| 엔터티 | 책임 | 주요 관계·제약 |
|---|---|---|
| `User` | 계정 주체 후보 | 현재 미구현; purchase·subscription·ownership과 연결 |
| `Plan` | Free·One·Plus·Business 정책 버전 | 가격과 분리, 유효 기간과 상태 필요 |
| `Product` | 판매 가능한 상품 정의 | One purchase 또는 subscription 상품 유형 |
| `Price` | 통화·금액·청구 주기 가설 | Provider price id와 내부 버전 분리, 동의 이력 필요 |
| `Entitlement` | 기능 권한 코드 정의 | 코드 유일, 설명·전제 조건·상태 |
| `PlanEntitlement` | plan별 권한·한도 매핑 | 유효 기간, unit, limit nullable |
| `Subscription` | 반복 계약 상태와 기간 | user/workspace, plan, period, cancel mode |
| `Purchase` | One 등 1회 구매 | 대상 문서 binding은 별도 권한 grant 후보 |
| `PaymentEvent` | Provider 이벤트 수신 원장 | Provider event id 유일, 원문 최소화·암호화 검토 |
| `UsageLedger` | 권한 사용 예약·확정·환원 | idempotency, source, billing period |
| `DocumentOwnership` | 문서 소유자·workspace 범위 | 결제와 별도 authorization 근거 |
| `RetentionPolicy` | 데이터 종류별 보존·삭제 규칙 | plan 기본값보다 보안·사용자 선택 우선 가능 |
| `RefundRecord` | 환불·취소·chargeback 기록 | purchase/subscription/payment event와 연결 |

추가 후보로 내부 `EntitlementGrant`와 `EntitlementProjection`을 검토한다. 구매·구독 이벤트를 매 요청마다 직접 해석하지 않고 유효 권한 projection을 만들되, 원장과 재구축 가능해야 한다.

## 4. 상태 모델

Subscription 상태 후보:

- `trialing`: 검증 전 체험 기간
- `active`: 현재 기간 권한 유효
- `past_due`: 결제 실패 후 정책 결정 대기
- `canceled`: 해지 확정; 종료 시점 별도 기록
- `expired`: 유효 기간 종료
- `suspended`: 정책·보안·분쟁으로 임시 정지

`canceled`만으로 즉시 종료인지 기간 종료 해지인지 알 수 없으므로 `cancel_effective_at`, `cancel_mode`(`immediate`, `period_end`)를 별도로 둔다. `past_due` 유예 기간은 미결정이다.

## 5. 결제와 권한 흐름 후보

```text
가격·약관 표시
→ 사용자 동의 이력 생성
→ Provider 결제 요청 후보
→ 결제 결과 수신
→ PaymentEvent 서명·중복·순서 검증
→ Purchase/Subscription 상태 전이
→ Entitlement projection 갱신
→ 문서 소유권과 사용량 확인
→ 기능 authorization
```

결제 성공과 권한 부여는 동일 transaction으로 가정하지 않는다. projection 실패 시 결제를 성공으로 기록하더라도 기능 접근은 안전하게 보류하고 운영 복구 대상으로 둔다.

## 6. Webhook 설계 원칙

- Provider별 endpoint와 secret을 격리한다.
- raw body 기준 서명을 검증하고 timestamp 허용 범위를 둔다.
- Provider event id를 유일하게 저장해 중복 이벤트를 멱등 처리한다.
- 이벤트 생성 시각, 수신 시각, Provider object version을 기록한다.
- 순서가 뒤바뀐 이벤트는 현재 aggregate version보다 오래된지 확인하고 무조건 덮어쓰지 않는다.
- 처리 상태 `received`, `verified`, `applied`, `ignored`, `failed` 후보를 분리한다.
- 실패 재처리는 같은 이벤트 id에서 멱등해야 한다.
- webhook payload에 계약 내용이나 문서 식별자를 넣지 않는다. 내부 billing subject id만 사용한다.
- secret, 카드 정보와 전체 payload를 application log에 남기지 않는다.

실제 서명 방식, endpoint, 재시도와 보존 기간은 Provider 선정 후 확정한다.

## 7. 상태 전이 규칙 후보

- 결제 성공: Purchase 또는 Subscription 활성화 후보 → entitlement projection 별도 갱신
- 결제 실패: 새 권한 부여 금지; 기존 권한 유예 여부는 미결정
- 기간 종료 해지: `cancel_effective_at`까지 권한 유지 후보
- 즉시 해지: 보안·환불 정책과 동의 조건에 따라 즉시 회수 후보
- 환불: RefundRecord 생성, 원 payment와 entitlement grant에 reversal 연결
- chargeback: 즉시 정지 후보지만 문서 삭제와 결제 분쟁 보존은 별도 판단
- plan 변경: 적용 시점과 proration은 Provider·정책 결정 전 확정 금지

## 8. Idempotency와 동시성

- 구매 생성, 분석 차감, 환불과 webhook 적용은 각각 idempotency key가 필요하다.
- 외부 요청 idempotency와 내부 ledger idempotency를 구분한다.
- 동일 결제 이벤트와 동일 사용량 source는 한 번만 확정한다.
- optimistic version 또는 DB 잠금 후보로 out-of-order 상태 덮어쓰기를 막는다.
- 권한 예약과 usage commit 사이 실패 시 release 또는 재처리 가능한 상태를 유지한다.

## 9. 가격·약관 동의 이력

최소 후보:

- 내부 `price_id`, product·plan version
- 표시 통화·금액·세금 정보
- 청구 주기
- 약관·개인정보 고지 version
- 동의 시각과 동의 주체
- 적용 시작·종료 시각

카드 번호, CVC 등 결제 수단 원문은 저장하지 않는다. Provider token도 최소화하고 secret으로 취급한다.

## 10. 보존·소유권 경계

- `DocumentOwnership`은 user/workspace와 document를 연결하며 plan과 분리한다.
- Business는 역할 기반 `viewer`, `reviewer`, `manager`, `owner` 후보와 문서 scope가 필요하다.
- 구독 해지로 문서 소유권 자체가 사라지지 않는다. 접근·내보내기 유예와 삭제 일정을 별도 정책으로 처리한다.
- 사용자 삭제 요청은 계약 데이터, 분석 결과, 계정, 결제 법정 보존 데이터를 데이터 종류별로 처리한다.
- 결제 이벤트와 세금·환불 기록에는 계약 원문을 포함하지 않는다.
- analytics에는 파일명, source quote, 조항, 금액·당사자 등 계약 내용을 보내지 않는다.

## 11. RetentionPolicy 후보

필드 후보:

- `data_category`: original, extracted_text, confirmed_text, analysis_result, report, audit, billing
- `scope_type`, `scope_id`
- `retention_duration`
- `legal_basis_or_purpose`
- `delete_on_subscription_end`
- `grace_period`
- `approved_at`, `approved_by`
- `policy_version`

원본 파일은 처리 후 삭제 기본값을 유지한다. 상품 기능 때문에 보존하려면 목적, 기간, 접근 통제, 암호화, 사용자 고지·동의와 삭제 검증을 갖춘 별도 보안 승인이 필요하다.

## 12. 장애와 복구

- Provider 성공·내부 실패: 결제 원장을 보존하고 권한 projection 재처리
- 내부 성공·Provider 불확실: 사용자에게 중복 결제를 유도하지 않고 상태 조회
- webhook 장기 실패: 운영 경보와 reconciliation 후보
- 중복 결제: 자동 단정 환불 대신 식별·지원 절차
- entitlement service 장애: 권한 확대보다 안전한 제한을 기본으로 하되 이미 구매한 사용자 복구 절차 필요

## 14. 제외 범위

실제 DB schema, migration, Provider SDK, webhook endpoint, 결제 UI, 인증, authorization, 카드 처리, 환불 실행, 세금 계산과 가격 확정은 포함하지 않는다.
