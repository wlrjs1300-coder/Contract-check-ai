# v0.6 문서 입력 계약

## 1. 목적과 적용 범위

이 문서는 PDF와 이미지 입력을 조항 분리 파이프라인에 안전하게 연결하기 위한 v0.6 설계 계약이다. 현재 구현된 UTF-8 TXT 업로드 동작을 기준선으로 삼되, 이 문서 자체는 API·저장소·데이터베이스 구현 완료를 의미하지 않는다.

우선순위는 다음과 같다.

- P0: PDF, JPG, JPEG, PNG 입력과 필요한 OCR 추출
- 유지: UTF-8 TXT 입력은 테스트와 보조 입력으로 계속 지원
- 텍스트 PDF: 텍스트 레이어에서 직접 추출
- 스캔 PDF 및 이미지: OCR 추출
- 다중 페이지 PDF와 여러 이미지의 순서 보존
- 추출 텍스트를 사용자가 확인·수정한 뒤에만 분석으로 전달
- 원본 파일 보호와 확정적인 삭제

실제 분석 품질, 외부 OCR 서비스, 실제 분석 Provider, 법률적 판단은 이 계약의 완료 범위가 아니다.

## 2. 현재 구현과 목표 계약의 경계

현재 `POST /documents/upload`는 multipart의 `file` 한 개를 받아 파일명 확장자가 `.txt`인지 확인하고, 전체 파일을 메모리에서 읽어 1 MiB 제한과 UTF-8 디코딩을 검사한다. 성공하면 즉시 조항을 분리하고 `Document`와 `Clause` 레코드를 저장하며 `processed` 응답을 반환한다. 파일 시그니처, PDF·이미지 decoder, 임시 원본 저장과 OCR 단계는 없다.

v0.6 목표 흐름은 업로드 검증, 텍스트 추출, 사용자 확인, 조항 분리를 서로 다른 상태로 분리한다. 아래 상태·수치·응답은 설계 후보이며 Phase 2 이후 구현과 테스트 전에는 현재 API의 동작으로 간주하지 않는다.

## 3. 지원 형식과 식별 규칙

### 3.1 형식 매핑

| 원본 유형 | 허용 확장자 | 허용 MIME 후보 | 필수 파일 시그니처 또는 내용 검사 | 처리 분기 |
|---|---|---|---|---|
| PDF | `.pdf` | `application/pdf` | 시작 바이트 `%PDF-`, parser 구조 검사 | 페이지별 직접 추출 또는 OCR 추출 |
| JPEG | `.jpg`, `.jpeg` | `image/jpeg` | JPEG SOI `FF D8 FF`, decoder 검사 | EXIF 회전 정규화 후 OCR 추출 |
| PNG | `.png` | `image/png` | PNG 서명 `89 50 4E 47 0D 0A 1A 0A`, decoder 검사 | 방향·색상 정규화 후 OCR 추출 |
| TXT | `.txt` | `text/plain` 후보 | 바이너리 형식 부재, UTF-8 또는 UTF-8 BOM 엄격 디코딩 | 직접 추출 |

확장자, 클라이언트 제공 MIME, 원본 파일명은 모두 신뢰하지 않는다. 서버가 제한된 선두 바이트와 parser·decoder 결과로 형식을 판별하고, 확장자·MIME·시그니처가 충돌하면 `file_type_mismatch`로 거부한다. MIME은 보조 신호이며 단독 허용 근거가 아니다.

시그니처가 맞더라도 전체 구조가 유효하다는 뜻은 아니다. 실제 parser 또는 decoder가 제한된 자원 안에서 구조를 끝까지 검증해야 한다. polyglot 의심, 뒤에 붙은 실행 가능 payload, 비정상적으로 중첩된 객체가 감지되면 분석을 진행하지 않는다.

### 3.2 허용하지 않는 파일

- GIF, WebP, TIFF와 HEIC 등 계약에 없는 이미지 형식
- DOC, DOCX, HWP 등 편집 문서 형식
- ZIP, RAR 등 압축 파일과 내부 파일 묶음
- 실행 파일, 스크립트, HTML, SVG
- 암호화 또는 암호가 필요한 PDF
- parser가 손상으로 판정한 PDF, decoder가 열지 못하는 이미지
- 페이지가 없거나 렌더링 가능한 내용이 없는 PDF
- 알파 채널만 있거나 유효 픽셀이 없는 빈 이미지
- 제한을 넘는 파일, 페이지, 픽셀 또는 이미지 묶음

## 4. 처리 분기

### 4.1 TXT

UTF-8과 UTF-8 BOM을 허용하고 엄격하게 디코딩한다. NUL 바이트 등 바이너리 신호, 빈 텍스트, 공백만 있는 텍스트는 거부한다. TXT는 OCR을 거치지 않는 보조 입력이며 최종 제품의 주 입력 방향을 대체하지 않는다.

### 4.2 PDF

PDF는 전체 문서에 하나의 방식만 강제하지 않고 문서 페이지별로 판별한다.

1. 제한된 parser로 암호화, 손상, 페이지 수와 구조 예산을 검사한다.
2. 각 문서 페이지의 텍스트 레이어를 직접 추출한다.
3. 추출된 글자 수, 인쇄 가능한 문자 비율, 공백·깨짐 비율, 페이지 면적 대비 텍스트 블록을 평가한다.
4. 충분하고 읽을 수 있는 텍스트는 `direct`로 유지한다.
5. 텍스트가 없거나 품질 기준을 충족하지 못하는 페이지는 안전하게 렌더링한 뒤 `ocr`로 보낸다.
6. 직접 추출과 OCR 추출이 섞인 PDF는 `mixed`로 표시한다.

텍스트 PDF와 스캔 PDF의 최종 판별 임계값은 확정하지 않는다. Phase 2 스파이크에서 합성 fixture로 한글, 표, 조항 번호, 깨진 글꼴과 혼합 PDF를 평가한 뒤 정한다. 추출 결과가 비어 있거나 비정상 문자로만 구성되면 성공으로 처리하지 않는다.

### 4.3 이미지

- JPEG의 EXIF orientation을 읽어 픽셀 방향을 정규화하되 EXIF 메타데이터 자체는 신뢰하거나 영구 보존하지 않는다.
- PNG와 JPEG는 decoder가 보고한 실제 폭·높이·총 픽셀 수를 기준으로 제한한다.
- 회전·기울기 보정은 원본을 덮어쓰지 않고 처리용 파생 이미지에만 적용한다.
- 지나치게 낮은 해상도, 심한 흐림, 과다 노출 등 추출 신뢰를 떨어뜨리는 조건은 OCR 경고와 사용자 검토 필요 상태를 만든다.
- 유효 텍스트가 추출되지 않으면 분석 차단 상태로 둔다.

### 4.4 여러 이미지의 순서

여러 이미지 입력은 사용자가 확인할 수 있는 명시적 순서를 가진다. 브라우저 선택 순서 또는 별도 `ordinal`을 최초 순서로 사용하되 파일명 정렬을 암묵적으로 사용하지 않는다. 서버는 각 항목에 1부터 시작하는 문서 페이지 번호와 원래 항목 식별자를 부여한다. 사용자는 OCR 시작 전 순서를 확인·변경할 수 있어야 하며, 확정된 순서를 추출·조항 분리·결과 근거까지 유지한다.

중복 페이지 의심은 경고하되 자동 삭제하지 않는다. 항목 누락, 중복 번호, 순서 충돌이 있으면 분석을 차단한다.

## 5. 자원 제한 초안

아래 값은 보안 예산과 UX 검토를 시작하기 위한 **초기 제안값**이다. 최종 지원 한도가 아니며 parser·OCR 스파이크, 메모리·CPU 부하 테스트와 배포 자원 측정 후 확정한다.

| 제한 | 초기 제안값 | 최종 결정 조건 |
|---|---:|---|
| TXT 단일 파일 | 현재와 동일한 1 MiB | 기존 호환 테스트 유지 |
| PDF 단일 파일 | 20 MiB | 압축 해제·렌더링 최대 메모리와 처리 시간 측정 |
| PDF 페이지 수 | 100페이지 | 한글 PDF의 p95 처리 시간과 동시 요청 부하 측정 |
| 이미지 한 장 | 10 MiB | decoder 메모리와 실제 픽셀 예산 측정 |
| 이미지 묶음 | 최대 20장, 합계 50 MiB | 업로드·OCR 큐와 UX 테스트 후 확정 |
| 이미지 한 장 총 픽셀 | 40 megapixels | decoder/OCR 메모리 상한 검증 후 확정 |
| 사용자 확인 텍스트 | 페이지당 200,000자 | 직접 추출 폭주와 DB/응답 크기 테스트 후 확정 |

압축된 파일 크기만 검사하지 않는다. PDF 객체 수, 페이지 렌더링 해상도, 이미지 실제 픽셀, 추출 문자 수, 처리 시간과 프로세스 메모리에도 독립 예산을 둔다. 제한값은 서버와 프론트에서 같은 계약으로 표시하되 서버 검증을 최종 기준으로 한다.

낮은 해상도의 초기 경고 후보는 짧은 변 기준 800px 미만 또는 유효 DPI 150 미만이다. 이 값은 자동 거부 기준이 아니라 사용자 검토 필요를 유도할 품질 신호이며, 스파이크 후 조정한다.

## 6. 상태 전이 계약 초안

```text
received
→ validating
→ extracting
→ review_required
→ confirmed
→ splitting
→ cleanup_verified
→ processed
```

분기 상태는 다음과 같다.

- `rejected`: 형식, 크기, 암호화, 손상 또는 자원 제한 위반으로 시작 전에 거부
- `extraction_failed`: 직접 추출과 OCR 추출을 완료하지 못함
- `review_required`: 추출은 끝났지만 사용자의 확인 또는 수정이 필요함
- `confirmed`: 사용자 확인 텍스트와 페이지 순서가 명시적으로 확정됨
- `cleanup_verified`: 원본 파일과 처리용 파생 파일의 삭제 성공이 검증됨
- `processed`: 확인된 텍스트로 조항 분리가 끝나고 삭제 검증까지 통과하여 분석 가능
- `failed`: 확인 후 조항 분리 등 내부 처리 실패
- `deleted`: 원본 파일 삭제가 검증됨. 문서 처리 상태와 별도 수명주기 상태로 관리하는 방안을 우선 검토

상태 이름과 HTTP resource 분리는 API 설계 때 변경될 수 있다. 어떤 이름을 선택하더라도 `위험 없음`, `분석 실패`, `OCR 실패`는 서로 다른 의미로 유지한다.

## 7. 사용자 검토와 분석 시작 조건

### 7.1 사용자 검토 필요 조건

다음 중 하나라도 해당하면 `requires_user_review=true`로 둔다.

- OCR을 사용한 페이지가 하나 이상 있음
- 직접 추출 품질 경고 또는 OCR 경고가 있음
- 추출 텍스트가 비거나 문자가 깨진 페이지가 있음
- 낮은 해상도, 회전·기울기 보정, 표·복수 단 의심이 있음
- confidence가 제공되지 않거나 초기 경고 임계값 아래인 블록이 있음
- 직접 추출과 OCR 추출이 혼합됨
- 여러 이미지의 순서를 확인해야 함
- 사용자가 추출 텍스트를 수정함

confidence는 엔진별 내부 신호일 뿐 정확성 보장이 아니다. 임계값만으로 검토를 생략하지 않는다.

### 7.2 분석 가능 조건

아래 조건을 모두 충족해야 조항 분리와 분석을 시작할 수 있다.

- 모든 문서 페이지의 처리 상태가 성공이며 페이지 순서가 연속적임
- 추출 텍스트가 비어 있지 않고 자원 제한 이내임
- 원본 추출본과 사용자 수정본의 연결이 유지됨
- `user_confirmed=true`이고 확인 시각이 기록됨
- 차단 수준 OCR 경고가 해소되거나 페이지를 다시 업로드함
- 원본 파일과 처리용 파생 파일의 삭제 성공이 검증됨
- 조항 분리 입력으로 사용자 확인 텍스트만 선택됨

OCR 실패 상태에서 분석을 강행하거나 빈 텍스트를 `위험 없음`으로 해석하는 동작을 금지한다.
삭제 검증이 실패하면 해당 항목을 격리하고 운영 경보와 재삭제 대상으로 등록한다. 삭제 성공이 검증되기 전에는 `processed` 또는 분석 가능 상태로 전환하지 않으며, 삭제 실패를 `위험 없음`이나 분석 성공으로 숨기지 않는다. 이는 v0.6의 개인정보 보호 우선 기본값이며 별도 보안 승인 없이는 완화하지 않는다.

## 8. 실패와 재시도

| 실패 | 사용자 처리 | 서버 처리 |
|---|---|---|
| 형식·시그니처 불일치 | 올바른 형식으로 다시 선택 | 원본 즉시 삭제, 세부 parser 정보 비노출 |
| 암호화·손상·빈 PDF | 암호 해제된 정상 파일 요청 | 추출 시작 금지, 원본 삭제 |
| 자원 제한 초과 | 페이지 분할 또는 작은 파일 요청 | 요청 중단, 큐·메모리 예산 회수 |
| 직접 추출 품질 부족 | OCR 전환 안내 | 페이지 단위 OCR 후보로 전환 |
| OCR 실패·빈 결과 | 재촬영·재업로드 안내 | 분석 차단, 원본 삭제 확인 |
| 일시적 내부 오류 | 동일 입력을 사용자가 명시적으로 재시도 | idempotency와 중복 문서 정책 검토 |
| 삭제 실패 | 분석이 차단되었음을 안내 | 민감 경로 비노출 경고, 격리·운영 경보·재삭제; 삭제 검증 전 `processed` 전환 금지 |

재시도는 실패한 페이지 단위가 바람직하지만 구현 복잡성과 원본 삭제 정책을 스파이크에서 검토한다. 원본이 이미 삭제된 뒤에는 서버가 임의 재사용하지 않고 사용자에게 재업로드를 요청한다.

## 9. 기존 TXT API 호환 전략

- 현재 `POST /documents/upload`의 TXT 요청과 성공 응답은 회귀 테스트로 보호한다.
- v0.6 비동기 또는 다단계 추출 계약을 기존 endpoint에 바로 섞지 않는다. 별도 extraction resource 또는 버전이 있는 endpoint를 우선 검토한다.
- 기존 `UploadedDocument` 응답을 갑자기 필수 필드가 늘어난 형태로 변경하지 않는다. 새 추출 필드는 별도 응답 또는 선택 필드로 도입한다.
- 기존 TXT는 내부적으로 `source_type=txt`, `extraction_method=direct`, 단일 문서 페이지로 정규화할 수 있으나 외부 응답 변경 전 호환 테스트가 필요하다.
- 조항 분리 이후의 기존 `document_id`, `clauses`, 분석 작업과 결과 조회 연결은 유지하는 것을 목표로 한다.
- API 경로, 상태 코드, 동기·비동기 경계는 Phase 2 API 설계에서 확정한다.

## 10. 미확정 사항과 결정 계획

## 11. 완료 경계

이 계약의 승인은 구현 착수 조건일 뿐 PDF·이미지 지원 완료가 아니다. Phase 2 이후 parser·decoder·OCR·삭제·사용자 확인 테스트가 통과하기 전까지 실제 파일 입력을 지원한다고 표현하지 않는다.
