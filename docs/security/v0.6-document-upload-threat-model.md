# v0.6 문서 업로드·OCR 위협 모델

## 1. 범위와 가정

이 문서는 신뢰할 수 없는 PDF·JPG·JPEG·PNG·TXT가 브라우저에서 업로드되어 검증, 직접 추출 또는 OCR 추출, 사용자 확인, 조항 분리로 이동하는 경계를 다룬다. 현재 인증 없는 기술 검증 MVP는 공개 업로드 운영에 충분하지 않다.

보호할 주요 자산은 원본 파일, 마스킹 전 추출 텍스트, 사용자 확인 텍스트, 페이지 순서와 위치, 서비스 가용성, 임시 저장소, credential, 분석 결과의 근거 연결과 사용자 신뢰다.

기본 신뢰 경계:

```text
사용자 장치
→ 업로드 API
→ 형식 검증 격리 영역
→ PDF parser / image decoder
→ 직접 추출 또는 OCR worker
→ 사용자 확인
→ 마스킹·조항 분리
→ 분석 Provider 경계
```

## 3. 공통 보안 통제

### 3.1 입력 처리

- 검증은 가능한 한 streaming으로 수행하고 전체 파일을 무제한 메모리에 적재하지 않는다.
- 형식별 parser·decoder를 allowlist로 제한하고 자동 형식 탐지를 실행 경로 선택의 유일한 근거로 사용하지 않는다.
- parser·OCR worker에는 원본 처리에 필요하지 않은 network, 저장소와 credential 접근 권한을 주지 않는다.
- 파일, 페이지, 객체, 픽셀, 문자, 시간, 메모리와 동시성에 독립된 예산을 둔다.
- 원본과 파생 파일은 전용 임시 root에 두고 성공·실패 모두 삭제한다.
- temp root·요청 디렉터리·서버 파일 생성이나 쓰기가 실패하면 `temporary_storage_unavailable`로 처리하고 DB 레코드를 만들지 않는다. 일부 요청 디렉터리 cleanup도 실패하면 `original_cleanup_failed`를 우선한다.
- 원본 파일과 처리용 파생 파일의 삭제 성공이 검증되기 전에는 `processed` 또는 분석 가능 상태로 전환하지 않는다. 실패 항목은 격리하고 운영 경보와 재삭제 대상으로 두며, 이를 `위험 없음`이나 분석 성공으로 숨기지 않는다. 별도 보안 승인 없이는 이 기본값을 완화하지 않는다.

### 3.2 추출·분석 무결성

- 각 페이지의 불투명 id, `page_number`, extraction method와 경고를 결과에 결합한다.
- 원본 추출본을 변경하지 않고 사용자 수정본을 revision으로 분리한다.
- 사용자 확인 전 조항 분리·분석을 차단한다.
- 조항과 분석 결과는 기존 `clause_id`와 `reference_id` 검증을 유지하고 페이지 근거를 추가한다.
- OCR 누락과 분석 실패를 `위험 없음`으로 표시하지 않는다.
- 실제 분석 결과는 원문 근거, 판단 이유, 사용자가 확인할 사항과 권장 질문을 구조화해야 하며 최종 판단이나 보장을 제공하지 않는다.

### 3.3 관측과 사고 대응

- 원문 없는 correlation id, 처리 단계, 오류 분류와 자원 수치만 기록한다.
- cleanup 실패, worker crash, 비정상 egress, 제한 초과 급증과 orphan 최고 age에 경보를 둔다.
- 취약 parser가 확인되면 해당 형식을 feature-level로 비활성화할 수 있어야 한다.
- 외부 OCR 장애 시 사용자 동의 없이 다른 공급자나 지역으로 자동 전송하지 않는다.

## 4. 보안 테스트 게이트

Phase별 최소 게이트:

- Phase 2: PDF 시그니처·암호·손상·페이지·객체·timeout 및 cleanup
- Phase 3: JPEG/PNG signature·EXIF·pixel bomb·decoder 오류 및 cleanup
- Phase 4: 혼합 PDF, 비동기 순서, 부분 실패, 동시성·자원 제한
- Phase 5: 원본/수정본 분리, stale revision, 확인 전 분석 차단
- Phase 6: 페이지→조항→결과 reference 연결과 instruction-injection 방어
- Phase 7: 외부 전송 동의·egress·보존 조건과 근거 기반 출력 검증
- Phase 8: fuzz·부하·강제 종료·orphan·backup·로그·삭제 복구 검증

운영 입력을 열기 전에 high-impact 위협의 예방·탐지·실패 테스트가 모두 통과해야 한다.

## 5. 잔여 위험과 미결정 사항

완전한 OCR, 개인정보 탐지 또는 법률적 적합성을 보장할 수 없다는 잔여 위험을 사용자에게 명확히 알린다.

현재 경로 방어는 Python 표준 라이브러리 수준이며 완전한 sandbox를 보장하지 않는다. parser process 격리, OS-level sandbox와 비정상 종료 파일을 회수할 orphan sweeper는 후속 범위다.
