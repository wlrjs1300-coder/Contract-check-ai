# v0.7.4 인증·소유권·사용자 격리 보안 설계

## 목적

JWT 인증과 부모 리소스 기반 소유권 검증으로 사용자 간 데이터 접근을 격리하고, 인증·권한·내부 오류의 응답 경계를 고정한다.

## 현재 구현 상태

- `/auth/register`, `/auth/login`, `/auth/me` 구현
- `Document`와 `Extraction`에 `owner_id` 저장
- `AnalysisJob`은 `Document` join으로 소유권 판정
- `Clause`, `AnalysisResultItem`, `ExtractionPage`는 부모 리소스 경계로 제한
- cross-user `Document`, `Extraction`, `AnalysisJob` 접근에 고정 `404` 적용
- SQLite 연결에서 외래 키 제약 활성화
- Alembic baseline 및 ownership revision parity 검증
- 동일 ID의 타 사용자 `Document` 충돌 방어

## 소유권 경계

### 직접 소유권 저장

- `Document`
- `Extraction`

### 부모 경계를 통한 소유권 판정

- `Document.owner_id`를 통한 `AnalysisJob` 판정
- `Clause` → `Document.owner_id`
- `Document.owner_id`를 기준으로 `AnalysisJob`을 거쳐 `AnalysisResultItem` 판정
- `ExtractionPage` → `Extraction.owner_id`

하위 테이블에 `owner_id`를 중복 저장하지 않는다. 조회·생성·분석 요청은 현재 인증 사용자와 부모 리소스 소유자가 일치할 때만 진행한다.

## cross-user 비노출 정책

- 타 사용자 `Document`: `404`
- 타 사용자 `Extraction`: `404`
- 타 사용자 `AnalysisJob`: `404`
- missing resource와 동일한 일반 오류 본문 사용
- `owner_id`, email, filename, 조항 본문, SQL, DB 오류 비노출
- 소유권 충돌 시 타 사용자 리소스 재사용·변경 및 `AnalysisJob` 생성 금지

extraction과 동일한 ID의 타 사용자 `Document`가 있으면 해당 문서를 읽거나 새 문서를 같은 PK로 생성하지 않는다. 조회 후 생성 사이의 `IntegrityError`도 rollback하고 일반 오류로 변환한다.

## JWT strict claim 검증

- PyJWT HS256 고정
- `JWT_SECRET` UTF-8 기준 최소 32바이트
- TTL 5~60분, 기본 15분
- 필수 claim: `sub`, `iat`, `exp`, `jti`, `iss`, `aud`, `ver`
- issuer, audience, signature 검증
- `sub`, `jti`: canonical lowercase UUID 문자열
- `iat`, `exp`, `ver`: 정확한 `int` 타입
- `bool`, `float`, 숫자 문자열 coercion 금지
- `exp > iat` 및 미래 `iat` 제한
- `auth_version` 불일치 및 inactive user 거부
- payload에 email, `owner_id`, resource ID 미포함

## 비밀번호 보안

- pwdlib Argon2 사용
- 길이 12~128자
- null byte 거부
- 앞뒤 공백 임의 strip 금지
- 평문 저장 금지
- 없는 email도 dummy Argon2 verify 경로 실행
- 존재하지 않는 계정, 잘못된 비밀번호, inactive user의 로그인 실패 응답 통일

## migration 보안

revision 체인은 `0001_current_schema_baseline` → `0002_auth_ownership`이다.

- `0001_current_schema_baseline`은 v0.7.3 ORM schema와 parity를 이룬다.
- `0002_auth_ownership` 적용 후 head는 현재 ORM metadata와 parity를 이룬다.
- `documents` 또는 `extractions`에 legacy row가 있으면 `0002_auth_ownership`을 즉시 중단한다.
- legacy row를 임의 사용자에게 자동 배정하지 않는다.
- 중단은 `users` 테이블과 `owner_id` 컬럼 생성 전에 수행한다.
- 중단 후 기존 revision과 row를 유지하고 부분 DDL을 남기지 않는다.
- `Document.owner_id`와 `Extraction.owner_id`는 `NOT NULL`, 사용자 FK `RESTRICT`, index를 가진다.
- `users.email`은 unique index를 가지며 `auth_version >= 1`을 제약으로 검증한다.

## schema 초기화 경계

- production application startup에서는 `Base.metadata.create_all()`을 사용하지 않는다.
- Alembic migration을 production schema 변경 기준으로 사용한다.
- 테스트용 격리 SQLite DB 초기화에서는 `Base.metadata.create_all()`을 사용한다.

## 오류 계약

- 인증 실패: `401`과 고정 인증 오류
- authenticated cross-user resource: `404`와 missing resource와 동일한 오류
- 내부 실패: 세부 정보를 숨긴 일반 `500` 계약
- DB `IntegrityError`: rollback 후 raw SQL, DB 오류, 식별자, 소유자 정보 비노출

## 미구현 또는 범위 외

- refresh token
- token blacklist 및 revocation
- rate limiting 및 account lockout
- 운영 비밀번호 정책
- 관리자 및 RBAC
- 실제 개인정보 pilot 승인

실제 개인정보를 포함한 계약서는 별도 보안 승인 전 개발·검증에 사용하지 않는다.
