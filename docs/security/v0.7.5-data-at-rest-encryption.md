# v0.7.5 저장 암호화·키 관리 보안 설계

## 기본 정보

- 버전: v0.7.5
- 브랜치: feat/v0.7.5-encryption-key-management
- 현재 단계: 설계 초안 검토 중

## 목적

현재 문서는 v0.7.5 구현을 위한 저장 암호화와 키 관리 기준을 확정한다. v0.7.5 전체에는 구현, migration, rotation 기반, fail-closed 오류 처리와 테스트가 포함된다. 완료 후에도 보관형 실운영과 실제 개인정보 사용은 별도 보안 게이트와 승인을 거쳐야 한다.

## 인증 주체와 소유권 경계

- 인증 주체는 `User`이다.
- `Document`, `Extraction`은 `owner_id`를 직접 저장한다.
- `AnalysisJob`과 `Clause`는 `Document.owner_id`를 상속한다.
- `AnalysisResultItem`은 `AnalysisJob`과 `Document` 관계를 통해 소유자를 검증한다.
- `ExtractionPage`는 `Extraction.owner_id`를 상속한다.
- 하위 리소스 암복호화에는 부모 관계에서 검증한 소유자 값을 사용한다.

## `extra_data` 3분류

### 평문 운영 메타데이터

상태 전이, 동시성, checksum 검증, 운영 관찰성, 생명주기 관리에 필요한 다음 값은 원문을 포함하지 않는다는 schema 검증 아래 평문으로 둘 수 있다.

- `review_status`, `review_version`, `snapshot_version`
- `analysis_blocked`, `can_confirm`, `blocking_reasons`
- `confirmation_checksum`, `evidence_snapshot_hash`
- 길이·페이지 count와 처리 방식 count
- `source_format`, 정규화 크기, `text_changed`
- `reviewed_at`, `confirmed_at`
- `classification`, `failure`, `validation_status`
- `confidence_score`, `is_stale`, `text_source`, `method`
- warning/error code

사용자 원문을 포함할 수 있는 자유 형식 문자열은 이 그룹에 포함하지 않는다.

### 암호화 대상 민감 하위 필드

- `confirmation_snapshot`, `reviewed_text`, `final_text`
- `blocks[*].text`
- `analysis_value`의 자연어 설명
- evidence의 excerpt 또는 text
- extracted facts의 raw/source text
- `expert_review_summary`, `recommendation`, `risk_reason`, `practical_impact`
- `questions_to_ask`, `negotiation_suggestions`
- 계약 당사자, 주소, 금액, 날짜 등 원문에서 직접 파생된 식별 가능 값

### 저장 금지

- raw Provider request/response
- redaction token mapping 원문
- 마스킹 전 전체 문서 복사본과 raw PII value
- Secret, access token, Authorization header
- 내부 파일 경로와 임시 OCR 이미지
- 중복 원문 snapshot과 디버깅용 전체 payload

v0.7.5 구현에서 모델별·schema별 allowlist와 validator를 도입한다. 임의 key 금지는 아직 구현 완료 상태가 아니며 전체 `extra_data`를 투명하게 한 번에 암호화하지 않는다.

## Envelope

```json
{
  "enc_version": 1,
  "key_id": "synthetic-key-v1",
  "algorithm": "AES-256-GCM",
  "nonce": "...",
  "ciphertext": "..."
}
```

- `nonce`는 Base64로 인코딩한다.
- `ciphertext`는 AESGCM 반환값 전체를 Base64로 인코딩하며 authentication tag가 결합 payload에 포함된다.
- plaintext AAD는 envelope에 저장하지 않는다.
- `algorithm`, `enc_version`, `key_id`는 엄격한 allowlist와 원래 타입을 검증한다.
- 누락 필드와 추가 필드는 기본적으로 fail-closed 처리한다.
- malformed Base64, unknown key, unsupported version·algorithm, tamper는 모두 안전한 오류로 차단한다.

## AAD 서버 재구성

복호화 시 서버는 신뢰 가능한 DB 관계와 호출 context로 다음 값을 재구성한다.

- `resource_type`
- `record_id`
- `field_name`
- `owner_id`
- `schema_version`

권장 canonical JSON 예시는 다음과 같다.

```json
{
  "domain": "contract-check",
  "enc_version": 1,
  "field_name": "body",
  "owner_id": "synthetic-owner-id",
  "record_id": "synthetic-record-id",
  "resource_type": "clause",
  "schema_version": 1
}
```

UTF-8, key 정렬, 불필요한 공백 제거, 숫자·문자열 타입 변환 금지, 누락 값 금지를 적용한다. envelope나 외부 입력에 포함된 AAD를 복호화 입력으로 사용하지 않는다.

## owner binding과 소유권 이전

현재 서비스는 소유권 이전을 지원하지 않으므로 v0.7.5에서 `owner_id`를 AAD에 포함하는 정책을 권장한다. 이는 다른 소유자 context의 복호화와 다른 사용자 row로의 ciphertext 복사를 차단한다.

이 정책은 소유권 이전 시 재암호화가 필요하다는 비용이 있다. 소유자 값만 바꾸는 작업을 금지하며, 향후 이전 기능을 도입하기 전에 소유자 변경과 재암호화를 하나의 transaction으로 처리하고 rollback과 감사 절차를 설계해야 한다.

## Filename 정책

- 내부 저장 식별자와 사용자 표시용 파일명의 분리를 검토한다.
- 현재 API의 `filename`, `filename_display` 계약을 즉시 제거하지 않는다.
- 표시명이 필요하면 권한 확인 뒤 암호화 값을 복호화하거나, 안전한 표시명으로 정규화하거나, 사용자가 설정한 별도 제목을 저장하는 방식을 비교한다.
- 실제 basename의 평문 DB 저장 여부는 B등급 조건부 결정이다.
- migration과 API 호환성을 확인하기 전 무작위 치환을 확정하지 않는다.
- 경로 정보는 제거하고 basename만 다루는 현재 안전화 원칙을 유지한다.
- 복호화 실패 시 내부 경로, ciphertext, nonce를 응답하지 않는다.

## Crypto core·Service·Model 역할

### Crypto core

- keyring parsing과 envelope schema validation
- AES-256-GCM encrypt/decrypt
- Base64 strict 처리와 canonical AAD 생성
- 오류 코드 정규화와 key status 검증

### Service layer

- resource context와 검증된 소유자 수집
- record, field, schema version 확정
- 민감 JSON 하위 필드 명시적 암복호화
- allowlist 검증과 double encryption 방지
- 권한 확인 뒤 API 직렬화 직전에 명시 복호화

### Model layer

- encrypted backing column과 envelope 저장
- 평문 property 최소화
- 암복호화가 자동으로 숨어 실행되지 않도록 명시적 경계 유지

`TypeDecorator`는 context가 단순한 scalar에만 제한적으로 검토한다. nested JSON 또는 소유자·record·field context가 필요한 필드에는 기본적으로 사용하지 않는다. scalar도 encrypted backing column과 명시 helper 방식을 우선 검토한다. 복호화 평문은 ORM 객체에 장시간 보관하지 않는다.

## Key status와 rotation

- `active`: encrypt/decrypt 가능
- `decrypt_only`: decrypt만 가능하며 신규 encrypt 금지
- `retired`: runtime keyring에서 제거되며 사용 ciphertext가 없어야 함

v0.7.5에는 active key 신규 쓰기, 기존 `key_id` 복호화, key usage inspection, rotation-ready architecture, 제한된 재암호화 helper와 idempotency 테스트를 포함한다.

retired 처리 전 사용 row 0건, backup·snapshot 영향, 재암호화 완료, 감사 기록, rollback 가능성을 확인한다. background bulk worker, KMS 자동 rotation, multi-region rotation, backup 전체 재암호화와 대시보드 orchestration은 후속 범위이다.

## 오류 계약

- startup에서 keyring 누락·손상, active key 불일치, key 길이 오류를 fail-fast 처리한다.
- runtime에서 unknown key, envelope tamper, AAD mismatch, unsupported version·algorithm을 fail-closed 처리한다.
- 복호화 실패 시 평문 fallback하지 않는다.
- 오류 응답과 로그에는 raw key, 평문, ciphertext, nonce, 환경변수 원문, 내부 경로, stack trace를 포함하지 않는다.
- 소유권 경계 실패는 기존 일반적인 `404` 계약을 유지한다.

## Migration과 평문 fallback

- 실제 개인정보가 없는 개발 DB 초기화를 기본 전략으로 한다.
- 필요하면 합성 데이터 전용 one-time re-encryption을 별도로 승인한다.
- 실데이터 자동 backfill은 현재 범위 밖이다.
- plaintext와 encrypted 값 동시 존재, 장기 dual-write, production dual-read를 금지한다.
- encrypted 값이 없고 plaintext만 있는 production row는 fail-closed 처리한다.
- 복호화 실패 시 자동 평문 fallback하지 않는다.
- encrypted 컬럼을 추가한다면 plaintext column 제거 계획과 exactly-one 검증이 필요하다.

## 테스트 기준

### Keyring

- keyring·active key 누락, malformed JSON, 중복 ID
- malformed Base64, 31/33-byte key, unsupported status
- decrypt-only 신규 encrypt 거부, retired key 실패
- raw key app state 미저장

### Envelope·AAD

- round trip, nonce별 ciphertext 차이, 한글·빈 문자열·대용량 처리
- ciphertext/authentication tag·nonce tamper
- unknown key, version·algorithm mismatch, 누락·추가 필드
- 잘못된 resource, record, field, owner, schema version
- 외부 AAD 무시와 다른 row ciphertext 복사 실패
- null byte 정책과 double encryption 방지

### 저장 경계·rotation·비노출

- A등급 및 민감 JSON 필드의 raw DB 평문 검색 실패
- ORM/API 정상 복호화와 production fallback 거부
- plaintext·encrypted 동시 저장 거부
- active·decrypt_only·retired 상태와 점진 재암호화 idempotency
- 예외·응답·로그의 key·평문·ciphertext·nonce 비노출
- cross-user `404` 유지

## 범위 밖

- 실제 개인정보 파일럿 승인
- 운영 KMS 또는 vault 확정
- background bulk re-encryption worker
- backup 전체 재암호화
- multi-region rotation
- 소유권 이전 기능
