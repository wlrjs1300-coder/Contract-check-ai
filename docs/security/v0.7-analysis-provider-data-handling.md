# v0.7.0 Analysis Provider Data Handling

## 기본 원칙

- 문서 본문 전체, 조항 원문, source_text는 분석 provider 요청 본문/로그/DB에 직접 저장하지 않는다.
- provider raw request/response, 내부 오류 메시지, stack trace를 분석 결과나 API 응답으로 노출하지 않는다.
- 실패 시에도 result item을 저장하지 않으며 job은 `failed`로 종료한다.

## 민감 정보 노출 금지

- secret-like 문자열, 경로, 원문 clause 텍스트, 사용자 식별자, 파일명은 로그에서 마스킹 또는 미기록
- provider가 반환한 원시 텍스트는 재사용/재전파 전에 정책 검증을 거침
- 실제 계약서와 실제 개인정보는 테스트/fixture에서 사용하지 않음

## 실패/예외 처리 경계

- expected failure는 계약 코드로 변환
- unexpected `ValueError/RuntimeError`만 예외로 올리되 API message에 내부 문맥을 누출하지 않음
- `original_cleanup_failed`, `temporary_storage_unavailable`, provider failure 코드는 사용하지 않는 경로에만 보존
- contract_check.db 변경 없이 테스트는 임시/fixture DB로 수행
- v0.7.1 확장:
  - 마스킹 단계에서 잔여 PII가 보이면 `provider_data_minimization_failed`로 차단.
  - 마스킹 토큰 충돌이 탐지되면 `provider_redaction_token_collision`로 차단.
  - Provider 출력에서 PII 의심 문자열이 재생성되면 `provider_output_pii_detected`로 차단.

## 사용자 입력 분리

- provider 요청에는 사용자 ID, 원본 파일명, 원본 bytes를 포함하지 않음
- 외부 URL 호출, shell/tool execution, 외부 search tool 사용은 하지 않음
- prompt injection 유사 입력은 clause data로만 전달하며 instruction로 바인딩하지 않음

## 운영/로깅

- request_id, attempt_number, reason_code, retryable 정도만 진단용으로 보존
- 결과 summary는 최소화하고 사용자 화면에서는 최종 상태만 노출
- OpenAPI를 통해 raw provider 스키마를 과도하게 노출하지 않음

## 미완성 항목

- 진정한 token-level 마스킹은 별도 범위
- real provider credential 회전과 감사 추적은 다음 단계
- provider response sandbox, timeout hardening, 장기 감사는 추후 단계
