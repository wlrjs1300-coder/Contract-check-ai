# v0.7.2 계약 데이터 생명주기

## 목적

계약 분석 요청의 생명주기를 통합해 원본, 중간 산출물, 분석 결과, 로그의 생성·보관·삭제 정책을 일회성 안전 분석 기준으로 정렬한다.

보관형 계약 관리 모드는 목표 설계이며 현재 구현 상태가 아니다.

## 현재 구현 상태

- 원본 업로드 파일은 임시 처리 후 cleanup 경로가 존재한다.
- 조항 원문 일부는 SQLite에 저장된다.
- 인증과 사용자별 ownership은 구현되어 있지 않다.
- 사용자 삭제, 회원 탈퇴, 보관기간 만료에 따른 자동 파기는 구현되어 있지 않다.
- orphan cleanup startup sweep는 v0.7.3에서 구현되었다.
- redaction token mapping 전용 저장소와 TTL은 구현되어 있지 않다.
- 실제 외부 Provider 호출은 구현되어 있지 않다.
- 현재 개발·검증 단계에서는 실제 개인정보 계약서를 사용하지 않는다.
- PII 게이트와 분석 실패 rollback은 구현되어 있다.

## 목표 운영 상태

- 일회성 안전 분석을 기본 모드로 운영한다.
- 원본과 중간 데이터가 생성된 작업은 cleanup 검증이 끝나기 전까지 완료로 확정하지 않는다.
- 보관형 계약 관리 모드는 인증·소유권·암호화·삭제 정책을 모두 충족한 뒤에만 활성화한다.
- 분석 처리 상태, cleanup 상태, deletion 상태를 분리한다.
- 삭제 확인 기록에는 원문, 실제 파일명, 실제 경로, 개인정보를 포함하지 않는다.

### 상태 규칙

- 원본 또는 중간 데이터가 생성된 작업은 cleanup 상태가 `completed`가 되기 전까지 처리 상태를 `completed`로 전이하지 않는다.
- cleanup 실패 시 처리 상태는 `failed`, 안전한 오류 코드는 `original_cleanup_failed`로 기록한다.
- 분석 결과가 생성됐더라도 cleanup 실패 시 사용자에게 성공 결과를 확정 제공하지 않는다.
- 실패·취소·timeout 경로에서도 cleanup을 수행한다.
- cleanup 재시도 횟수와 간격은 후속 구현에서 정하되 무한 재시도는 금지한다.
- cleanup 실패의 실제 원문, 파일명, 경로, 내부 예외는 API 응답과 로그에 노출하지 않는다.
- 사용자 삭제, 회원 탈퇴, 보관기간 만료, 관리자 비상 삭제는 분석 실패가 아니며 deletion 상태로 추적한다.
- deletion 실패는 기존 분석 처리 상태를 소급해 `failed`로 바꾸지 않고 `deletion_failed`와 안전한 삭제 오류 코드로 기록한다.
- `analysis stalled` 같은 새 상태는 추가하지 않는다.

## 일회성 안전 분석 흐름

```text
업로드
→ 파일 검증
→ 격리 임시 저장
→ 텍스트·OCR 추출
→ 조항 분리
→ PII 탐지·마스킹
→ 잔여 PII 검사
→ 최소 데이터 분석
→ Provider 출력 검증
→ 결과 임시 확정
→ 원본·중간 데이터 cleanup
→ cleanup 검증
→ 최종 결과 확정
→ completed
```

## 보관형 계약 관리 모드

보관형 계약 관리 모드는 현재 미구현 목표 상태다.

활성화 전 필수 조건:

- 인증
- 사용자별 ownership
- IDOR 차단
- 저장 암호화
- 키 관리와 교체
- 보관기간 정책
- 사용자 삭제
- 탈퇴·만료 파기
- 접근 감사
- 관리자 최소 권한
- 백업 삭제 정책
- 실제 데이터 처리 승인

원본 보관은 기본값이 아니다. 사용자의 명시적 선택과 별도 고지·동의 또는 적절한 법적 근거가 있는 경우에만 허용한다.

## 재시작과 orphan cleanup

### 현재 상태

- 자동 orphan cleanup는 startup sweep로 보강되었다.
- 서버 재시작 후 잔여 임시 파일과 중간 데이터를 스캔 대상으로 식별·삭제한다.
- durable cleanup queue와 장기 운영 알림은 향후 단계에서 보완한다.

### 목표 상태

- startup sweeper 또는 durable cleanup queue를 구현한다.
- 불투명 작업 ID를 사용해 잔여물을 탐지한다.
- 정해진 횟수만 cleanup을 재시도한다.
- 반복 실패 시 보안 경보와 `original_cleanup_failed`를 기록한다.
- 원문, 실제 파일명, 실제 경로는 로그에 남기지 않는다.

## 삭제 의미와 성공 판정

### 삭제 의미

- 논리 삭제: 일반 조회 경로에서 데이터를 숨기는 상태
- 접근 차단: API와 운영 콘솔에서 조회할 수 없게 하는 상태
- 물리 삭제: 실제 파일·레코드·cache key를 제거하는 상태
- 임시 파일 cleanup: 작업 중 생성된 파일과 디렉터리를 제거하는 처리
- backup 만료 파기: 보관기간이 지난 백업을 제거하는 처리
- cryptographic erasure 후보: 복호화 키를 안전하게 폐기해 데이터 복구 가능성을 제거하는 방식
- 삭제 tombstone: 복원이나 재동기화 시 삭제 대상이 다시 나타나지 않도록 남기는 비민감 삭제 표식
- 삭제 확인 기록: 삭제 요청·완료·실패 상태를 기록하는 비민감 감사 이벤트

### 삭제 성공 판정

- 활성 API에서 조회할 수 없음
- DB 활성 레코드가 제거됐거나 비가역적으로 접근 차단됨
- 대상 파일이 존재하지 않음
- filesystem·application cache key가 없음
- redaction token mapping이 없음
- raw Provider payload가 없음
- 삭제 tombstone 또는 동등한 통제가 존재함
- backup 복원 시 삭제 대상이 재등장하지 않음
- 개인정보 없는 삭제 감사 이벤트가 존재함

삭제 확인 기록에는 원문, 실제 파일명, 실제 경로, 개인정보를 포함하지 않는다.

## 실제 개인정보 파일럿 허용 게이트

현재 개발·검증 단계에서는 실제 개인정보 계약서 사용을 금지한다.

이 제한은 최종 제품의 영구 제한이 아니다. 다음 조건을 모두 충족하고 별도 승인 절차를 통과한 뒤에만 제한적 파일럿 허용을 검토한다.

- 인증
- 사용자별 ownership
- IDOR 차단
- TLS
- 임시 파일 파기
- orphan cleanup
- 저장 암호화
- 키 관리·교체
- 사용자 삭제
- 탈퇴·만료 파기
- PII 독립 평가
- 외부 전송 fail-closed
- 로그·APM 본문 차단
- Secret 관리
- 접근·삭제 감사 기록
- Provider 데이터 처리 정책 검토
- backup 삭제 정책
- 운영 보안 점검
- 제한적 파일럿 승인

게이트 일부만 충족한 상태에서는 실제 개인정보 계약서 사용 금지 상태를 유지한다.
