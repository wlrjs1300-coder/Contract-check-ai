# v0.6 원본 문서 생명주기 정책

## 1. 목적과 기본 원칙

원본 계약서는 식별정보, 연락처, 주소, 서명과 계약 조건을 포함할 수 있는 민감정보로 취급한다. 파일 형식이나 사용자가 실제 개인정보가 없다고 표시했는지와 관계없이 동일한 보호 수준을 적용한다. 직접 추출과 OCR 추출로 만든 추출 텍스트에도 같은 원칙을 적용한다.

현재 MVP 기본 정책은 **처리에 필요한 가장 짧은 시간 동안 격리된 임시 저장소에 두고, 성공·실패와 무관하게 원본과 처리용 파생 파일을 삭제하는 것**이다. 원본 파일과 원문 전체 텍스트를 데이터베이스, backup 또는 일반 로그에 저장하지 않는다.

이 문서는 목표 정책이다. 현재 TXT 구현은 파일을 메모리에서 읽고 분리된 조항 본문을 SQLite에 저장하며, 별도 임시 원본 파일은 만들지 않는다. v0.6 구현은 이 정책에 맞는 수명주기 상태와 삭제 검증을 추가해야 한다.

## 2. 데이터 분류

| 데이터 | 분류 | 기본 보존 |
|---|---|---|
| 원본 파일 | 최고 민감도 | 처리 직후 삭제 |
| PDF 렌더링 페이지·보정 이미지 | 원본과 동일 | 해당 페이지 추출 직후 삭제 |
| 마스킹 전 추출 텍스트 | 원본과 동일 | 사용자 확인 텍스트 또는 후속 안전 입력 생성 직후 삭제 |
| 사용자 확인 텍스트 | 민감 | 분석에 필요한 최소 범위와 기간만 저장 |
| 마스킹된 조항 | 민감도 감소, 여전히 보호 대상 | 분석·결과 연결에 필요한 기간 |
| 원본 파일명 | 개인정보 가능 메타데이터 | 표시용 최소값만, 보존 필요성 별도 검토 |
| 크기·페이지 수·처리 상태 | 운영 메타데이터 | 정의된 운영 기간 |
| hash | 재식별·연결 위험이 있는 메타데이터 | 목적과 기간을 명시한 경우만 |

마스킹은 익명화를 보장하지 않는다. 사용자 확인 텍스트와 마스킹된 조항에도 접근 통제와 삭제 정책을 적용한다.

## 3. 수명주기

### 3.1 수신과 임시 저장

1. 서버는 전용 임시 root 아래 요청별 무작위 디렉터리를 만든다.
2. 경로는 서버가 생성한 불투명 식별자만 사용하고 클라이언트 파일명을 경로에 결합하지 않는다.
3. 디렉터리와 파일 권한은 서비스 계정의 최소 권한으로 제한한다.
4. 저장과 동시에 바이트 수 제한을 적용하며 제한을 넘으면 더 읽지 않고 폐기한다.
5. 형식 검증 전 파일을 실행하거나 일반 문서 viewer로 열지 않는다.
6. 원본과 파생 파일은 저장소 작업 디렉터리, 정적 파일 경로와 backup 대상 밖에 둔다.

임시 root의 구체 경로는 OS별 안전한 임시 디렉터리와 배포 volume을 비교한 뒤 설정으로 확정한다. 기본값이 저장소 내부나 사용자의 홈 폴더가 되지 않도록 검증한다.

### 3.2 파일명 처리

- 디스크 파일명은 UUID 등 암호학적으로 충분한 무작위 값과 서버가 판별한 안전한 확장자를 사용한다.
- 원본 파일명은 경로, shell, parser 옵션이나 로그에 사용하지 않는다.
- 화면 표시가 필요하면 경로 구분자와 제어문자를 제거하고 길이를 제한한 `filename_display`만 별도 메타데이터로 다룬다.
- 원본 파일명에는 이름·주소·계약 상대가 포함될 수 있으므로 기본 로그와 감사 이벤트에서 제외한다.
- 표시용 파일명 보존 기간과 사용자 삭제 방법은 인증·이력 정책과 함께 확정한다.

### 3.3 검증·추출·확인

- parser·decoder·OCR worker만 해당 요청 디렉터리에 제한적으로 접근한다.
- PDF 페이지 렌더링과 이미지 보정은 원본을 덮어쓰지 않고 별도 무작위 파생 파일로 만든다.
- 페이지 처리가 끝날 때마다 더 이상 필요 없는 파생 파일을 즉시 삭제한다.
- 추출 텍스트는 사용자 확인을 위해 필요한 경로로만 전달하며 원문 전체를 console, 예외, tracing, analytics에 기록하지 않는다.
- 사용자가 수정한 텍스트는 원본 추출본을 덮어쓰지 않는다. 수정본과 페이지·block 연결, 확인 시각을 별도로 유지한다.
- 사용자 확인 후 분석에는 사용자 확인 텍스트만 사용한다.
- OCR 경고가 남은 사용자 검토 필요 상태에서는 분석을 시작하지 않는다.
- 원본 추출 텍스트는 사용자 확인 텍스트 또는 후속 안전 입력을 생성한 직후 삭제한다. 목적, 기간, 접근 통제, 저장 시 암호화, 사용자 고지·동의와 삭제 검증을 갖춘 별도 승인이 있을 때만 예외적으로 보존한다.
- 사용자 확인 텍스트의 보존 기간은 별도 미결정 사항으로 유지하며 원본 추출 텍스트의 예외 보존과 동일하게 취급하지 않는다.

### 3.4 정상 종료와 실패 삭제

삭제는 성공 경로의 마지막 줄에만 의존하지 않는다. 요청 또는 worker의 `finally` 성격 cleanup에서 다음을 수행한다.

1. 열려 있는 file handle과 parser resource를 닫는다.
2. 파생 이미지, 임시 텍스트와 원본 파일을 삭제한다.
3. 요청별 임시 디렉터리가 비었는지 확인하고 제거한다.
4. 삭제 성공·실패를 원문 없는 메타데이터로 기록한다.
5. 삭제 실패 시 경로 전체나 파일명을 로그에 남기지 않고 내부 opaque id와 오류 분류만 기록한다.
6. 삭제 실패 항목은 재사용하지 못하도록 격리하고 orphan 정리 대상으로 표시한다.
7. 원본 파일과 처리용 파생 파일의 삭제 성공이 검증된 뒤에만 `processed` 또는 분석 가능 상태로 전환한다.

업로드 거부, parser 오류, OCR timeout, 사용자의 취소, 분석 차단과 서버 5xx 모두 같은 cleanup 의무를 가진다. 삭제 검증이 실패하면 해당 항목을 격리하고 운영 경보와 재삭제 대상으로 등록하며 분석 진행을 차단한다. 삭제 실패를 `위험 없음`이나 분석 성공으로 숨기지 않는다. 이는 v0.6의 개인정보 보호 우선 기본값이며 별도 보안 승인 없이는 완화하지 않는다.

## 4. 비정상 종료와 orphan 정리

프로세스 강제 종료에서는 in-process cleanup이 실행되지 않을 수 있다. 다음 방어를 함께 둔다.

- 요청 디렉터리에 생성 시각과 opaque owner id만 담은 민감하지 않은 marker 사용 검토
- 애플리케이션 시작 시와 주기적 작업에서 TTL을 넘은 전용 임시 디렉터리 탐색
- symlink·junction을 따라가지 않고 전용 root의 실제 경로 안인지 재검증한 뒤 삭제
- 처리 중인 lease와 orphan을 구분해 활성 작업을 삭제하지 않음
- 삭제 실패 횟수와 최고 잔존 시간을 원문 없는 metric으로 경보
- worker 종료, OS 재시작과 disk-full 상황을 포함한 복구 테스트

orphan TTL은 운영 보존 기간과 다르며 **초기 제안값 1시간**으로 스파이크한다. 최종값은 최대 정상 처리 시간과 재시작 복구 방식을 측정한 뒤 확정한다.

## 5. 보존 예외

원본 파일, 처리용 파생 파일 또는 마스킹 전 추출 텍스트의 보존은 기본 동작이 아니다. 장애 조사, 사용자가 요청한 이력 또는 별도 제품 기능을 이유로 보존하려면 다음이 선행되어야 한다.

- 목적, 데이터 항목, 보존 기간과 삭제 시점을 문서화한 별도 승인
- 사용자의 명확한 고지와 명시적 동의
- 인증·소유권·접근 통제와 사용자 삭제 기능
- 저장 시 암호화, key 관리와 접근 감사
- backup과 복제본까지 포함한 만료·삭제 절차
- 운영자 접근 승인과 정기 권한 검토
- 정책 종료 후 복원본에서 재등장하지 않는 삭제 검증

승인이 없으면 원본 파일, 처리용 파생 파일과 마스킹 전 추출 텍스트를 보존하지 않는다. 사용자 확인 텍스트의 보존 기간은 Phase 5 전까지 미결정 사항으로 유지한다.

## 6. 외부 Provider 전송 조건

외부 OCR 또는 분석 Provider에 원본, 페이지 이미지나 마스킹 전 텍스트를 전송하는 것은 별도 처리 목적이다. 기본 정책에서 금지하며 다음 조건을 모두 검토·승인한 경우에만 요청 단위로 허용한다.

- 전송 대상과 목적, 처리자, 처리 지역, 보존과 삭제에 대한 사전 고지
- 사용자의 명시적 동의 및 동의하지 않는 대안
- 필요한 페이지만 보내는 데이터 최소화와 EXIF 등 불필요한 메타데이터 제거
- 전송 중 암호화와 최소 권한 credential
- 학습 사용 여부, 보존 기간, 하위 처리자와 지원 로그 정책 확인
- timeout·재시도 시 중복 전송 제한
- 공급자 삭제 요청과 증빙 가능성
- 외부 응답 원문을 일반 로그에 남기지 않는 처리

마스킹 전 데이터 전송이 반드시 필요한지 먼저 검토하며, 가능한 경우 로컬 추출·마스킹 후 최소화된 입력만 분석 Provider에 전달한다.

## 7. 접근 권한과 감사 기록

- 서비스 계정, OCR worker와 cleanup worker를 역할별 최소 권한으로 분리한다.
- 일반 개발자·운영자가 원본 디렉터리를 탐색하는 경로를 제공하지 않는다.
- 디버그 endpoint, 정적 파일 serving과 관리자 다운로드를 만들지 않는다.
- 감사 기록 후보는 opaque document id, actor/service role, event type, 형식, 크기 구간, 페이지 수, 처리·삭제 결과, 시각과 오류 분류다.
- 파일명, 원문, 추출 텍스트, 페이지 이미지, 개인정보와 local path는 감사 기록에서 제외한다.
- 감사 기록의 접근·보존 기간도 별도 정책으로 제한한다.

인증이 없는 현재 MVP를 공개 사용자 업로드 환경으로 취급하지 않는다. 사용자별 소유권과 조회 격리가 필요한 공개 운영 전에는 인증·인가 결정을 완료해야 한다.

## 8. 로그와 관측 금지선

- 원문 전체 또는 일부의 console 출력 금지
- 마스킹 전 텍스트의 application·access·trace 로그 금지
- multipart body, OCR 요청·응답 body와 예외의 raw dump 금지
- 원본 파일명, 로컬 절대 경로와 query string 노출 금지
- 지원 도구, error tracking, analytics로 원본 전송 금지
- metric label에 document id나 사용자 입력을 사용하지 않음

오류는 `invalid_signature`, `parser_timeout`, `ocr_empty_result`, `cleanup_failed` 같은 고정 분류와 opaque correlation id로 관측한다.

## 9. 백업과 저장 계층

- 원본·파생 파일용 임시 root는 filesystem snapshot, volume backup과 동기화 도구에서 제외한다.
- container image와 build context에 임시 데이터가 들어가지 않게 한다.
- crash dump, swap, core dump와 APM capture가 민감 버퍼를 포함할 가능성을 운영 설계에서 검토한다.
- 데이터베이스에는 원본 파일 BLOB과 원문 전체를 저장하지 않는다.
- 사용자 확인 텍스트 또는 조항 저장이 필요하면 목적·암호화·소유권·보존·삭제를 Phase 5 전에 확정한다.
- backup에 포함되는 데이터는 만료 후 backup에서도 복원되지 않도록 삭제·복원 정책을 시험한다.

## 10. 개발·테스트 환경

- 실제 계약서, 고객 데이터와 실제 개인정보를 로컬·CI·저장소에 사용하지 않는다.
- 합성 fixture만 사용하며 이름·주소·번호도 명백한 가상 값으로 만든다.
- 저장소에는 텍스트 기반 fixture를 우선하고, 바이너리 fixture 추가는 별도 검토를 거친다.
- 개발용 temp도 운영과 같은 무작위 파일명·cleanup을 사용한다.
- 실패 테스트 뒤 임시 root가 비었는지 fixture 단위로 검증한다.
- 테스트 출력과 snapshot에 원문이 들어가지 않도록 한다.
- 실제 외부 OCR 호출은 단위·통합 테스트에서 사용하지 않는다.

## 11. 삭제 검증 테스트 계획

| 시나리오 | 검증 |
|---|---|
| 정상 직접 추출 | 원본·파생 파일과 요청 디렉터리의 삭제 검증 후에만 `processed` 전환 |
| 정상 OCR | 각 페이지 파생 파일과 마지막 원본의 삭제 검증 후에만 분석 가능 전환 |
| 확장자·시그니처 거부 | parser 실행 없이 임시 파일 제거 |
| parser/decoder 예외 | 예외 후 handle 해제와 제거 |
| OCR timeout·worker 실패 | 부분 산출물 포함 제거, 분석 차단 |
| 사용자의 취소·연결 종료 | background cleanup 완료 |
| disk full·권한 오류 | 실패 분류, 격리, 운영 경보와 재삭제; 분석 진행 차단 |
| 프로세스 강제 종료 | 재시작 orphan sweeper가 TTL 이후 제거 |
| symlink·junction 공격 | 전용 root 밖 대상은 건드리지 않고 경보 |
| 중복 재시도 | 이전·새 요청 디렉터리가 독립적으로 제거 |
| backup 복원 | 임시 원본이 backup에 존재하지 않음 |
| 로그 검사 | 파일명·원문·로컬 경로가 기록되지 않음 |

삭제 테스트는 존재하지 않는 경로만 확인하지 않고, 합성 marker의 생성·처리·삭제와 전용 root 경계를 검증한다.

## 12. 미확정 사항

| 항목 | 필요한 정보 | 스파이크·검토 | 책임 범위 | 결정 시점 |
|---|---|---|---|---|
| 임시 root와 격리 방식 | Windows/Linux 경로, container volume | OS별 path·권한·재시작 시험 | Backend/운영 | Phase 2 전 |
| orphan TTL | 최대 처리 시간, 재시작 복구 | timeout·강제 종료 부하 시험 | 운영/보안 | Phase 4 전 |
| 사용자 확인 텍스트 보존 | 수정 UX, 재분석, 삭제 요구 | 데이터 흐름·DB 초안 검토 | 제품/보안 | Phase 5 전 |
| 운영 보존 기간 | 장애·감사·사용자 요구 | lifecycle·backup 복원 검토 | 보안/운영 | 배포 전 |
| 인증 시점 | 소유권과 공개 접근 위험 | 위협 모델·사용자 흐름 검토 | 제품/Backend | 공개 업로드 전 |
| 외부 Provider | 정책·계약·처리 지역 | 공급자 심사와 비식별 스파이크 계획 | 보안/제품 | 외부 전송 승인 전 |

## 13. 승인 조건

Phase 2 구현 전에 임시 root 경계, 정상·실패 cleanup, 원문 로그 차단과 기존 TXT 호환 테스트 계획을 검토한다. 실제 운영 입력을 허용하기 전에는 인증·인가, 외부 전송, 보존·backup과 삭제 복구 시험까지 완료해야 한다.
