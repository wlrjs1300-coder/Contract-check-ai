## 1. 문서 목적

이 문서는 ContractCheck AI v0.2.2 PR-6 범위에서 PR-1부터 PR-5까지 작성된 개인정보·보안 설계 문서를 통합 검토한 결과를 기록한다.

목적은 문서 간 상태값, 데이터 분류, 안전 경계가 서로 충돌하지 않는지 확인하고, 승인 가능 항목·보류 항목·금지 항목을 구분하며, 미결정 사항과 잔여 위험을 정리하여 [v0.2.2 최종 설계 결정](v0.2.2-final-design-decision.md) 문서의 근거로 제공하는 것이다.

이 문서는 PR-1~PR-5 원본 문서를 수정하지 않는다. 원본 문서에서 발견된 불일치는 이 문서에 발견 사항으로만 기록한다.

## 4. 문서 간 정합성 검증

### 4.1 상태값 정합성

PR-2~PR-5의 모든 문서 헤더는 다음 상태값을 동일하게 사용한다.

| 상태값 | 사용 문서 | 정합성 |
|---|---|---|
| `Status: Draft` | 전체 (PR-2~PR-5) | 일치 |
| `Approval status: Review required / Not approved` | 전체 (PR-2~PR-5) | 일치 |
| `Provider selection: Not selected` | 전체 | 일치 |
| `External AI use approval: Not granted` | 전체 | 일치 |
| `Real contract use: Not approved` | 전체 | 일치 |
| `Real personal data use: Not approved` | 전체 | 일치 |
| `Real external transfer: NOT EXECUTED` | PR-4, PR-5 | 일치 (PR-2/PR-3에는 아직 이 필드가 없으나 상충 아님 — PR-4에서 최초 도입) |
| `Production use: Not approved` | PR-3, PR-4, PR-5 | 일치 |
| `Security test execution: NOT EXECUTED` | PR-5 | PR-2~PR-4에는 해당 필드 없음 — 보안 테스트는 PR-5 범위이므로 상충 아님 |

실제로 실행되지 않은 항목이 완료·승인·실행된 것처럼 기록된 사례는 발견되지 않았다. `ALLOW`, `CONDITIONAL`, `APPROVED CANDIDATE` 등의 상태값은 모두 "후보" 또는 "조건부"로 명시되어 있으며, 무조건 승인으로 해석될 수 있는 표현은 없다.

### 4.2 데이터 분류 정합성

PR-2 `data-classification.md`가 정의한 D1(Restricted)/D2(Sensitive)/D3(Internal)/D4(Public) 체계를 PR-3 `storage-retention-deletion.md`가 저장 정책표에서, PR-4 문서들이 outbound allowlist와 금지 필드표에서 그대로 인용한다.

핵심 원칙 유지 확인:

- D1(원본 계약서, 추출 원문, 마스킹 전 조항, 탐지된 개인정보, raw provider request/response) 영구 저장 금지 — `data-classification.md` §6-7, `storage-retention-deletion.md` §7·§14에서 일관.
- D1 외부 전송 금지 — `data-classification.md` §7, `external-transfer-control.md` §17 Prohibited Transfer Fields에서 일관.
- D1 직접 조회 기본 금지 — `access-failure-output-safety.md` §13 Data Access Control에서 모든 역할(Administrator 포함)에 대해 `PROHIBITED`로 일관.
- 관리자도 D1 자동 접근 금지 — `access-failure-output-safety.md` §4, §11에서 "Administrator privileges must not imply unrestricted data access" 명시.
- 실제 계약서·실제 개인정보 사용 미승인 — 전체 문서 헤더 및 각 문서 §2 적용 범위에서 일관.

### 4.3 개인정보 흐름 정합성

`pii-processing-flow.md`의 21단계 흐름과 판정값(`ALLOW`/`BLOCK`/`REVIEW`/`NOT EXECUTED`)이 `external-transfer-control.md`(외부 전송 후보 단계), `access-failure-output-safety.md`(출력 안전 검증 단계)와 일관되게 이어진다.

- residual PII 발견 시 BLOCK — `pii-processing-flow.md` §6, `external-transfer-control.md` §10, §19, `security-test-scenarios.md` §9에서 일관.
- 개인정보 원문 로그 금지 — `pii-processing-flow.md` §10, `logging-audit.md` §8 금지 필드, `access-failure-output-safety.md` §14에서 일관.
- detected personal data 외부 전송 금지 — `pii-processing-flow.md` §12, `external-transfer-control.md` §17에서 일관.
- 개인정보 재생성 출력 차단 — `provider-boundary.md` §10 Response Boundary, `access-failure-output-safety.md` §22-23에서 일관 (`Detect personal data regeneration` → BLOCK).
- 실패 시 안전한 삭제 후보 — `pii-processing-flow.md` §9, `storage-retention-deletion.md` §9-11에서 일관.
- silent success 금지 — `storage-retention-deletion.md` §14, `logging-audit.md` §14·§16, `access-failure-output-safety.md` §19에서 일관 ("deletion failure -> silent success" 금지, "log failure must not ... silent success").

### 4.4 Trust Boundary 정합성

`trust-boundary.md`의 Zone A~G 체계를 `provider-boundary.md`(Zone C/D/E/G 재인용), `external-transfer-control.md`(Zone C→E 상태)가 동일하게 사용한다.

- 경계 통과 시 명시적 검증 — `trust-boundary.md` §5에서 경계별 필수 검증 정의, `external-transfer-control.md` §6 Pre-Transfer Safety Gates가 이를 구체화.
- 기본 DENY — `external-transfer-control.md` §4 "External transfer is default DENY", `access-failure-output-safety.md` §4 "Default access policy is DENY"에서 일관.
- 최소 권한 — `access-failure-output-safety.md` §4, §11에서 명시.
- 소유권 검증 — `access-failure-output-safety.md` §9 Authorization Boundary, §13 Data Access Control에서 명시.
- cross-user 접근 차단 — `access-failure-output-safety.md` §28 Prohibited Access Paths, `security-test-scenarios.md` §7에서 일관.
- 관리자 bypass 기본 금지 — `access-failure-output-safety.md` §9 "Administrator bypass is prohibited by default"에서 명시.
- 외부 provider는 신뢰 역할이 아님 — `provider-boundary.md` §4 "Zone E is not an internal trusted zone", `access-failure-output-safety.md` §5 역할표에서 External Provider를 `PROHIBITED`/내부 접근 없음으로 명시하여 일관.

**Minor 발견 사항**: `external-transfer-control.md` §17 Prohibited Transfer Fields 표에서 "raw operational log"와 "raw audit log"를 `D2` 등급으로 표기하나, `data-classification.md` §4·§5에서 "operational logs"와 "audit metadata"는 `D3 - Internal`로 분류되어 있다. 두 표현이 가리키는 대상이 완전히 동일한지(정상 로그 vs. 손상되어 원문이 유입된 로그) 원본 문서에서 명확히 구분되지 않는다. 이는 안전 방향(D2가 D3보다 보수적)으로 어긋난 것은 아니므로 Blocker/Major는 아니나, 후속 PR에서 등급 표기를 일치시킬 필요가 있다. (파일 수정은 본 PR-6 범위 밖이므로 원본 문서는 수정하지 않았다.)

### 4.5 저장·보존·삭제 정합성

- D1 영구 저장 금지 — `storage-retention-deletion.md` §7, §14에서 `PROHIBITED`/`IMMEDIATE DELETE`로 일관.
- 임시 저장 범위(S1) — §4 S1에서 정의, `data-classification.md` D1 예시와 일치.
- 삭제 트리거 — §9에서 18개 트리거 정의.
- 삭제 실패 상태 — §11에서 `BLOCK`/`REVIEW`/`RETRY CANDIDATE`/`ESCALATION REQUIRED`/`NOT IMPLEMENTED` 5개 값만 사용, `access-failure-output-safety.md` §15 Failure Status Model과 값 체계가 호환된다.
- 삭제 결과 검증 — §10 삭제 처리 흐름 12단계에서 확인.
- 백업 포함 여부 — §4 S5 "DESIGN ONLY, Not approved", §12에서 D1 백업 금지 명시. `external-transfer-control.md`에는 백업 관련 규정 없음(범위 밖이므로 상충 아님).
- 로그 보존, 감사 로그 보존 — §13, `logging-audit.md` §13에서 "미확정" 상태로 일관.
- 구체 retention 기간 미확정 — §8에서 상태값만 사용(`IMMEDIATE DELETE`, `SESSION ONLY`, `SHORT-LIVED — DURATION NOT APPROVED`, `RETENTION NOT APPROVED`, `NO PERSISTENCE`, `REVIEW REQUIRED`)하고 구체 일수를 하나도 확정하지 않음을 확인.
- 제품 또는 저장소 미선정 — §4 S5에서 "실제 백업 구성 없음", 본문 전체에서 특정 DB·스토리지 제품명이 등장하지 않음을 확인.

승인되지 않은 기간이나 제품이 확정된 것처럼 작성된 부분은 발견되지 않았다.

### 4.6 로그·감사 정합성

`logging-audit.md` §8 금지 필드 목록(계약서 본문, 개인정보, raw provider request/response, API Key, access token, refresh token, password, secret, authorization header, cookie value 등)이 `storage-retention-deletion.md` §14-15, `external-transfer-control.md` §15, `access-failure-output-safety.md` §14와 동일하게 반복된다. 세션 식별자 원문, 내부 보안 설정 관련 금지도 `access-failure-output-safety.md` §10(세션 원문 로그 금지), §28(내부 보안 설정 노출 경로 금지)에서 확인된다.

- 감사 이벤트는 실행 기록이 아니라 후보 — 모든 이벤트 표(로그·외부전송·PR-5 audit event candidate)가 `Decision status: PROPOSED`로 표기되어 있으며, "실제 성공 이벤트"(`EXTERNAL_AI_EXECUTED`, `PROVIDER_CALL_SUCCEEDED` 등)는 `external-transfer-control.md` §16에서 명시적으로 "not added as active events"로 배제됨을 확인.
- 감사 로그 조회 권한 별도 승인 필요 — `logging-audit.md` §11, `access-failure-output-safety.md` §14 "Audit access requires a separate permission"에서 일관.
- 로그 실패가 안전 경계를 우회하지 않음 — `logging-audit.md` §14 "로그 기록 실패 시 silent success 금지", `access-failure-output-safety.md` §19에서 로그·감사 실패를 BLOCK/REVIEW/ESCALATION으로 처리하도록 규정, 안전 경계 우회 사례 없음.

### 4.7 외부 전송 정합성

- 실제 외부 전송 NOT EXECUTED — `external-transfer-control.md` 헤더·§3·§5·§18에서 일관.
- 외부 AI 사용 미승인 — 전체 문서 헤더에서 일관.
- provider/endpoint/model 미선정 — `external-transfer-control.md` §3, `provider-boundary.md` §3에서 일관.
- allowlist 기반 전송 — `external-transfer-control.md` §7 Outbound Allowlist, 기본값 DENY.
- D1 외부 전송 금지, residual PII 외부 전송 금지 — §4, §17에서 일관.
- 사용자 승인 필요 후보 — §6 gate 27 "Final user approval obtained", §10 blocking condition에서 일관.
- 비용 승인 필요 후보 — §6 gate 24, §13(provider-boundary), §10에서 일관.
- timeout·retry 정책 미확정 — §14 "Timeout value is undecided", "Retry count is undecided"에서 명시.
- raw payload 저장 금지 — §15에서 명시.
- 미승인 fallback 금지 — `provider-boundary.md` §14 "must not automatically switch to another provider, endpoint, or model", §16 "provider failure to automatic unsafe fallback" 금지에서 명시.

### 4.8 Provider Boundary 정합성 (v0.2.1 연계)

v0.2.1 PR-6 최종 판단(`docs/checklists/v0.2/v0.2.1-core-validation.md` §, `spikes/v0.2.1-core-validation/reports/pr-6-provider-policy-final-decision.md`)은 다음과 같다.

- OpenAI API: `CONDITIONAL`
- Anthropic API: `CONDITIONAL`
- Google Gemini API: 유료 경로 `REVIEW`, 무료(Unpaid) 경로 `CONDITIONAL`이나 실질적으로 계약서 분석에 미사용
- 최종 판단: B. External AI use requires additional security and policy review before product use.
- Gemini API Unpaid Services는 계약서 분석에 사용하지 않음(차단 방향)
- 공급자 선정 없음, 실제 외부 AI 사용 없음, API Key 준비 없음, adapter 구현 없음

`v0.2.2-security-design-plan.md` §2가 이 결론을 표로 인용하고 있으며, `provider-boundary.md`와 `external-transfer-control.md`는 이 결론과 모순되지 않는 provider-agnostic 승인 게이트만 정의한다(특정 provider를 CONDITIONAL/REVIEW 판정과 다르게 재평가하거나 선정하지 않음). PR-6에서도 특정 provider, endpoint, model을 선정하지 않는다.

- 삭제 실패 silent success 금지 — `storage-retention-deletion.md` §14.

- 반복 실패 escalation 후보 — 전체 문서에서 `ESCALATION REQUIRED` 상태 공유.

### 4.12 Confidence 정합성

`access-failure-output-safety.md` §24가 유일한 상세 근거이며 `data-classification.md`(confidence score를 D2로 분류, "No user-facing risk score without later approval")와 상충하지 않는다.

- confidence는 법적 정확도 점수가 아님 — §24.
- confidence 단독 안전 판정 금지 — §24 "Confidence alone must not decide safety".
- 높은 confidence도 전문가 검토 필요성을 제거하지 않음 — §24, §25.
- 구체 threshold 미확정 — §24 "Specific threshold values are undecided".
- percentage 표시 여부 미확정 — §24 "Whether to display percentage values is undecided".
- 범위 오류는 INVALID OUTPUT 후보 — §24, §23.

### 4.13 Expert Review Recommendation 정합성

`access-failure-output-safety.md` §25가 유일한 상세 근거이다.

- 5개 상태(NOT REQUIRED / RECOMMENDED / STRONGLY RECOMMENDED / REQUIRED BEFORE ACTION / REVIEW STATUS NOT DECIDED) 정의.
- NOT REQUIRED는 안전 보장이 아님 — "does not mean the clause or contract is safe" 명시.
- REQUIRED BEFORE ACTION은 미승인 설계 후보 — "a design candidate, not an implemented enforcement rule" 명시.
- 고위험 트리거 명시 — §25 두 번째 표 (계약 체결, 해지, 배상, 위약금, 자동갱신 등).
- 구체 threshold 및 매핑 규칙 미확정 — §25 마지막 문단.
- 전문가 검토는 법률 자문 제공이 아님 — §25 첫 문단.
- 실제 전문가 연결 기능 미구현 — §25 "no expert connection feature is implemented in this draft".

### 4.14 사용자 고지 정합성

`access-failure-output-safety.md` §26과 `security-test-scenarios.md` §14가 거의 동일한 고지 항목 목록(참고 정보, 법률 자문 아님, 전문가 대체 아님, 누락·오판 가능, confidence 비보장, 전문가 검토 권고, 외부 AI 미실행, 실제 데이터 미승인)을 공유하며 상충하지 않는다. 구체 고지 문구, UI 위치는 §26 "deferred to later versions"로 명시되어 미확정 상태를 유지한다. UI는 미구현 상태(`security-test-scenarios.md` §14 "The current UI is not implemented").

### 4.15 보안 테스트 정합성

## 6. 상태값 검증 요약

다음 상태값이 문서 전반에서 발견되었으며, 모두 정의된 의미 범위 내에서만 사용되었다.

`Draft`, `Review required`, `Not approved`, `Not selected`, `Not started`, `Not implemented`, `NOT EXECUTED`, `BLOCK`, `DENY`, `REVIEW REQUIRED`, `SECURITY HOLD`, `PROHIBITED`, `CONDITIONAL`, `REVIEW`.

`Approved`로 표기된 항목은 PR-1 계획 문서의 5개 계획 수준 게이트(v0.2.2 범위 승인, 제외 범위 승인, 산출물 구조 승인, PR 분할 승인, 계획 문서 작성 승인)뿐이며, 이는 "계획 구조에 대한 승인"이지 "보안 설계 내용에 대한 승인"이 아니다. 이 구분을 [v0.2.2 최종 설계 결정](v0.2.2-final-design-decision.md) 문서에서 명확히 한다.

## 9. 잔여 위험

- 접근 통제·인증·인가·세션이 설계 후보 단계이며 실제 구현 전까지 위협 모델의 유효성이 코드 수준에서 검증되지 않는다.
- provider 미선정 상태이므로 실제 provider 정책 변경(예: 데이터 보존 정책 변경)이 재검토 없이 반영될 위험이 존재하지 않으나, 향후 provider 선정 시 §4.8의 v0.2.1 조건부 판정을 재확인해야 한다.
- 로그·감사 무결성 방식(hash chain, WORM, 서명)이 미확정이므로 실제 구현 시 감사 로그 위변조 방지 요구사항을 별도로 검증해야 한다.
- expert review recommendation과 confidence threshold의 구체 매핑이 없어 후속 PR에서 임계값을 정할 때 고위험 조항이 낮은 등급으로 잘못 분류될 위험이 있다. 이는 §5 M-1~M-3과 별개로, 설계 자체의 한계이며 후속 구현 설계 단계에서 명시적으로 다뤄야 한다.
