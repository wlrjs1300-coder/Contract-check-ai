# v0.7.2 데이터 분류표

## 문서 목적

v0.7.2 범위에서 계약 처리 데이터의 등급과 저장·전송·로그 정책을 단일 기준으로 통일한다.

## 현재 구현과 목표 상태

- 현재 구현
  - SQLite 기반 저장과 기본 분석 흐름이 동작한다.
  - 인증·권한·사용자별 ownership은 구현되지 않았다.
  - PII 마스킹 게이트와 Synthetic/ Fake Provider 경로는 존재한다.
  - 실제 외부 Provider 호출은 하지 않고, 실제 개인정보 계약서는 사용하지 않는다.
- 목표 상태
  - 일회성 분석과 보관형 모드별로 저장/삭제/전송 정책을 엄격히 분리한다.
  - 원본·추출본·token mapping·감사 데이터를 정합된 등급과 정책으로 관리한다.
  - 실제 파일럿 단계에서는 실무 보안 게이트를 충족한다.

## 데이터 등급 정의 D0~D5

- D0: 공개·비민감 데이터
- D1: credential·Secret·암호화 키
- D2: 운영 메타데이터와 감사 식별자
- D3: 마스킹·가명 데이터와 분석 결과
- D4: 개인정보가 포함될 수 있는 원문 데이터
- D5: 고유식별정보·민감정보·금융정보와 재식별 매핑

추가 규칙:

- 등급 숫자는 단순한 위험도 순위만을 뜻하지 않는다.
- D1은 계약서 본문 민감도와 비교되는 값이 아니라, credential/Secret 계열의 별도 비밀정보 통제군이다.
- 하나의 데이터가 여러 성격을 가지면 가장 강한 통제 정책을 적용한다.
- 마스킹 데이터도 재식별 가능성을 전제로 보호한다.

## 외부 전송 allowlist

- schema_version
- 불투명 request_id
- 불투명 document_id
- snapshot_version
- 마스킹된 clause text
- evidence candidate ID
- 분석에 필요한 최소 page·offset 메타데이터
- 데이터 크기 제한과 safety context

## 외부 전송 금지 데이터

- 원본 파일
- 원본 파일명
- 원본 bytes
- PDF·OCR 이미지
- 추출 원문
- 분리 조항 원문
- 실제 사용자 ID
- 이메일·전화번호·주소
- redaction token mapping
- DB 내부 PK
- 서버 경로
- raw prompt history
- 전체 confirmation snapshot
- 로그·예외·stack trace

## 저장·보관·삭제 원칙

- 저장은 목적 및 경로별 최소 데이터만 허용한다.
- 영구 보관은 보관형 모드에서만, 별도 승인된 조건에서 허용한다.
- 삭제는 논리 접근 차단을 먼저 적용하고, 물리 삭제 또는 cryptographic erasure를 병행한다.
- 삭제 실패는 감사 기록으로 남기고 재정리 대상으로 전환한다.

## 후속 구현 연결

- 보관형 모드 승인 정책은 `docs/checklists/v0.7/v0.7.2-data-lifecycle-security.md`와 정합성 확인.
- 외부 Provider 전송 정책은 ADR/Threat Model와 함께 검토한다.
- 실제 파일럿 게이트 전제 조건은 운영 보안 게이트 정책과 동기화한다.
