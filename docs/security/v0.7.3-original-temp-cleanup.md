# v0.7.3 원본·임시 데이터 파기 보장

## 개요
v0.7.3은 프로세스 비정상 종료나 재시작 뒤에도 임시 요청 디렉터리의 잔존 위험을 줄이기 위한
orphan cleanup 범위를 구현한다. 기존 request 단위의 동기 cleanup을 유지하면서,
재시작 후 잔존한 임시 디렉터리를 startup에서 1회 정리한다.

## 범위
- 추출 요청 단위 임시 디렉터리에 대한 marker 기반 상태 추적
- 활성(lease 갱신된) 요청과 orphan 후보의 분리
- 경로 경계/요청 경계/링크·reparse 차단
- bounded retry 기반 삭제 정리
- startup sweep 실패와 개별 항목 실패의 분리

## 범위 외
- 운영 알림 시스템, 정기 주기 스케줄러
- 사용자 삭제·탈퇴·backup 삭제 자동화의 완전 자동 연동
- 인증/ownership 기반 영속 데이터 삭제 정책
- 실제 external Provider 연동

## marker 정책
- 기록 항목: `schema_version`, `request_id`, `created_at_utc`, `lease_updated_at_utc`,
  `process_id`, `cleanup_state`
- 민감 항목(원본 파일명, 실제 경로, raw 업로드 텍스트, raw Provider payload)은 기록하지 않는다.
- marker 실패 또는 파싱 실패는 즉시 삭제 근거가 아니라 보수적 스킵 정책으로 처리한다.

## orphan sweep 정책
- 스캔 대상: temp root의 직접 자식 디렉터리 중 `request-*` 이름
- TTL: 기본 3600초, 60~86400 허용
- lease가 TTL 미만 남은 항목: `active`로 간주 후 보존
- lease가 TTL을 넘은 항목: 요청 디렉터리 삭제 시도
- 삭제 전 경로 검증 실패, symlink/reparse point, marker 부재/파싱 실패는 즉시 삭제 제외
- 삭제 실패는 항목 단위 실패로 기록하고 다른 항목 처리 계속 진행

## startup 동작
- FastAPI startup 시 1회 `sweep_orphan_request_directories` 수행
- 전역 설정 오류(잘못된 경로·구성)는 서비스 시작 실패 사유로 상향
- 단일 항목 삭제 실패는 앱 시작을 막지 않음

## 실패/오류 코드 연계
- 기존 `original_cleanup_failed`와 `temporary_storage_unavailable` 계약은 유지한다.
- 임시 marker 파싱 실패는 별도 `failed` 또는 `skipped` 상태로만 반영한다.
- 시작/설정 실패는 `OrphanCleanupError`로 구분해 관측한다.
