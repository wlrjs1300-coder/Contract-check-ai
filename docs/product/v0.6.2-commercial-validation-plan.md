# v0.6.2 상품성 검증 계획

## 1. 목적

고객이 PACTA의 근거 중심 사전점검에 실제 가치를 느끼는지, One과 Plus 중 어떤 구매 방식이 필요한지 검증한다. 품질·보안 검증 전에 결제를 활성화하지 않으며 가격과 제공량은 `검증 전 가설`로만 다룬다.

## 2. 고객 세그먼트

| 세그먼트 | 상황 | 현재 대안 | 검증할 문제 |
|---|---|---|---|
| 첫 근로계약을 검토하는 개인 | 계약 경험이 적고 질문할 항목을 모름 | 검색, 지인, 그대로 서명 | 근거·질문이 불안 감소와 확인 행동으로 이어지는가 |
| 임대차 계약을 확인하는 개인 | 보증금·관리비·수선·해지 조건을 비교 | 중개 설명, 특약 직접 읽기 | 누락 가능성·일정·금액 구조가 도움이 되는가 |
| 프리랜서·1인 사업자 | 업무위탁 계약을 반복 검토 | 템플릿, 수동 체크리스트 | 검수·지급·권리·책임 비교와 질문 관리 수요가 있는가 |
| 소규모 사업자 | 여러 거래처 계약과 마감 관리 | 폴더·스프레드시트 | 이력·비교·알림이 Plus 전환 가치를 만드는가 |
| 팀·조직 검토 담당자 | 담당·승인·감사 기록이 필요 | 이메일·문서 협업 | Business workflow가 필요한가; 실제 구현은 후속 |

## 3. 핵심 가설

1. 단순 라벨보다 원문 근거·이유·영향·확인사항·질문을 함께 제공할 때 신뢰와 행동 가능성이 높아진다.
2. 사용자는 누락 가능성과 일정·금액·의무 구조화를 “혼자 찾기 어려운 가치”로 인식한다.
3. 가끔 사용하는 개인은 구독보다 One을 선호한다.
4. 반복 사용자에게는 분석 정확도 차이가 아니라 이력·비교·알림이 Plus의 핵심 이유가 된다.
5. Free에서도 정해진 안전 하한선을 숨기지 않아야 신뢰와 자발적 전환이 높다.
6. 전문가 검토 권고는 구체적 이유와 질문이 있을 때만 유용하다.

## 4. 검증 방법

### 문제 인터뷰

- 최근 계약서를 확인할 때 가장 어려웠던 부분은 무엇이었나요?
- 중요한 조건을 놓쳤다고 나중에 알게 된 경험이 있나요?
- 날짜·금액·의무 중 따로 기록하는 것은 무엇인가요?
- 상대방에게 질문하고 싶었지만 무엇을 물어볼지 몰랐던 적이 있나요?
- 현재 어떤 사람·서비스·검색·체크리스트를 사용하나요?
- 계약 내용을 다시 확인하는 시점은 언제인가요?
- 수정본을 이전 버전과 비교하거나 마감을 알림받을 필요가 있나요?

유도 질문과 법률 조언 요청을 피하고 실제 과거 행동을 우선 묻는다. 실제 계약서나 개인정보를 수집하지 않는다.

### 결과 구조 인터뷰

합성 문구로 만든 결과 prototype만 사용한다.

- 이 결과를 보고 가장 먼저 확인할 항목은 무엇인가요?
- 원문 근거가 판단에 충분한가요, 너무 길거나 짧은가요?
- 이유와 가능한 영향이 구분되어 이해되나요?
- 제안 질문을 실제 상대방에게 사용할 수 있나요?
- `주의`, `추가 확인`, confidence를 어떻게 해석했나요?
- 무엇이 법률 판단처럼 오해될 수 있나요?

### 상품 인터뷰

- 한 건 전체 분석을 1회 구매하는 선택과 월 구독 중 어느 상황에서 각각 필요할까요?
- 이력, 비교, 알림, 리포트 중 반복 결제 이유가 되는 것은 무엇인가요?
- Free에서 반드시 확인되어야 신뢰할 수 있는 정보는 무엇인가요?
- 보관을 원하나요, 처리 후 삭제를 원하나요? 데이터 종류별 기대가 다른가요?
- 팀에서는 누가 업로드·검토·승인·삭제해야 하나요?

## 5. Free→유료 전환 가설

- Free는 탐지된 최우선 주의 항목, 해지·자동 갱신·위약금·금액·기간 관련 핵심 결과, 각 항목의 최소 근거와 즉시 확인 이유, 최소 1개의 실행 질문을 제공한다.
- 분석 실패·추출 실패·추가 확인 상태와 사용자의 의사결정에 중대한 영향을 줄 수 있는 탐지 결과, 안전성 경고와 근거 부족을 Free에서 의도적으로 숨기지 않는다.
- One 전환 이유 후보는 같은 문서의 전체 누락·질문·의무와 리포트다.
- Plus 전환 이유 후보는 월 사용량, 이력, 비교와 알림이다.
- 유료 범위는 전체 지원 category, 상세 누락 검사, 전체 질문 목록, 상세 일정·금액·의무 구조와 이력·비교·알림·리포트·협업 기능으로 확장한다.
- 핵심 위험을 흐리거나 공포를 자극하는 paywall은 실험하지 않는다.
- 전환 문구는 “더 정확한 분석”이 아니라 “전체 항목 보기”, “버전 비교”, “일정 관리”처럼 사실에 근거한다.

## 6. One vs Plus 수요 검증

| 관찰 | One 적합 신호 | Plus 적합 신호 |
|---|---|---|
| 사용 빈도 | 특정 사건에서 1건 | 월별·분기별 반복 |
| 사후 행동 | 한 번 확인·내보내기 | 이력 조회·재검토 |
| 수정본 | 단일 최종본 | 여러 차례 협상·비교 |
| 일정 | 일회성 기록 | 지속 알림 |
| 협업 | 개인 | 담당자·팀 |

선호 질문만으로 결론 내리지 않고 prototype 선택, 다시 사용 의향, 실제 반복 행동을 함께 본다.

## 7. 핵심 지표 후보

| 지표 | 의미 | 주의점 |
|---|---|---|
| 근거 확인률 | 결과에서 근거 유형에 맞는 인용·검색 범위·비교·계산 기준을 확인한 비율 | 높음이 곧 이해를 의미하지 않음 |
| 질문 사용 의향 | 제안 질문을 복사·수정·사용하겠다는 응답 | 실제 행동 후속 확인 필요 |
| 결과 이해도 | 라벨·이유·영향을 올바르게 설명하는 비율 | 법률 정답률과 다름 |
| 추가 확인 발견률 | 사용자가 몰랐던 확인사항을 찾았다고 응답 | 합성·자기보고 편향 기록 |
| 완료율 | 업로드부터 결과 검토까지 완료 | 품질 실패를 제외해 숨기지 않음 |
| 재사용 의향·재방문 | 다음 계약에도 사용할 의향·행동 | 가격 제시 전후 분리 |
| One 선택률 | 일회 구매 후보 선택 | 실제 결제 아님 |
| Plus 선택률 | 반복 기능 포함 구독 후보 선택 | 기능별 기여 분해 |
| 오류·추가 확인 비율 | 분석 불확실성과 실패 분포 | 낮추기 위해 오류를 숨기지 않음 |
| 전문가 rubric 통과율 | 근거·누락·질문 품질 | 결제 활성화 gate |

## 8. 가격 실험 원칙

- 가격은 실제 분석 품질과 단위 원가의 최소 검증 후 조사한다.
- 먼저 지불 의향 범위와 대안 비용을 인터뷰하고, 이후 명확한 기능 묶음으로 선택 실험한다.
- 가상의 숫자를 실제 확정 가격처럼 표시하지 않는다.
- 세금, 환불, 보관, 지원 범위를 함께 보여주지 않은 가격 비교는 의사결정 근거로 사용하지 않는다.
- 동일 기능을 임의로 다른 가격에 결제시키는 실험은 별도 윤리·정책 검토 없이 하지 않는다.

## 9. 품질 검증 전 결제 활성화 금지

다음 조건 전에는 실제 결제·카드 입력·구독을 활성화하지 않는다.

- 지원 계약 유형과 입력 형식이 명확함
- 원문 reference와 인용 검증 통과
- 날짜·금액 실패를 0으로 처리하지 않음
- 누락 가능성을 확정 누락으로 표현하지 않음
- 분석·추출 실패와 `위험 없음` 분리
- 법률 확정·보장 표현 차단
- 개인정보·삭제·소유권·접근 제어 승인
- 합성 평가셋과 전문가 rubric 목표 충족
- 실제 Provider 데이터 정책과 운영 장애 대응 승인

## 10. 개인정보와 연구 운영

- 인터뷰에서 실제 계약서 제출을 요구하지 않는다.
- 합성 scenario와 구조 prototype을 사용한다.
- 녹음·연락처·동의가 필요하면 저장 목적·기간·접근자를 별도 고지한다.
- analytics에 계약 내용, 파일명, 원문 인용과 당사자 정보를 보내지 않는다.
- 연구 참여와 향후 구매를 연결하거나 압박하지 않는다.

## 12. 산출물과 의사결정

인터뷰 결과는 개인정보가 제거된 집계·가설 상태로만 기록한다. 각 가설은 `지지`, `기각`, `추가 검증`으로 판정하고, 분석 품질과 보안 gate가 미달이면 상품·결제 구현을 진행하지 않는다.
