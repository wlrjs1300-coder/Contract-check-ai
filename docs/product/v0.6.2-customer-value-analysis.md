# v0.6.2 고객 가치 중심 분석 설계

## 1. 목적과 현재 경계

PACTA는 계약서를 대신 판단하는 서비스가 아니라, 사용자가 놓치기 쉬운 조건을 원문 근거와 함께 찾아주고 협상·확인에 필요한 다음 질문까지 제공하는 계약서 사전점검 서비스다.

고객이 받아야 할 핵심 가치는 다음 질문에 답하는 것이다.

1. 무엇을 주의해야 하는가
2. 왜 주의해야 하는가
3. 어느 문장이 근거인가
4. 사용자에게 어떤 영향이 있을 수 있는가
5. 무엇을 확인해야 하는가
6. 상대방이나 전문가에게 무엇을 물어봐야 하는가
7. 어떤 일정·금액·의무를 놓치면 안 되는가
8. 어떤 내용이 빠졌을 가능성이 있는가

현재 구현은 조항별 `display_label`, `summary`, `expert_review_recommended`를 합성 Provider로 반환한다. 이 문서는 목표 분석 구조와 검증 기준을 설계하며 실제 외부 분석, 법률 판단, OCR, 인증 또는 결제를 구현하지 않는다.

## 2. 결과 표현 원칙

- 모든 표시 결과는 검증 가능한 근거를 가지지만 모든 근거가 원문 인용일 필요는 없다.
- 근거 연결이 검증되지 않으면 결과를 표시하지 않거나 `추가 확인`으로 분리한다.
- `source_quote`는 `evidence_type=source_quote`인 결과에 필수이며 판단에 필요한 최소 구절만 사용한다. 전체 원문을 복제하지 않는다.
- 누락 가능성에는 존재하지 않는 문장을 가짜 인용으로 만들지 않고 문서 검토 범위와 검색 결과를 근거로 사용한다.
- `page_number`는 PDF·이미지 입력이 확인된 페이지 근거를 제공할 때만 채운다. 현재 TXT 분석에는 없을 수 있다.
- `severity`는 화면 정렬과 확인 우선순위이며 위법성, 무효 또는 법적 위험의 확정값이 아니다.
- `confidence`는 구조화 결과의 내부 신뢰 신호일 뿐 정확성 보장이나 고객용 확률이 아니다.
- `위험 없음`, 분석 실패, 추출 실패, 근거 부족과 `추가 확인`을 서로 다른 상태로 관리한다.
- 전문가 검토 미권고는 안전 보장이나 전문가 검토 불필요를 의미하지 않는다.
- 법률 자문, 계약 체결 여부, 적법성, 승소 가능성과 최종 판단을 제공하지 않는다.

## 3. 고객 가치 결과 유형

| 유형 | 고객이 얻는 답 | 필수 구성 |
|---|---|---|
| 주의 조항 | 무엇을 먼저 읽어야 하는가 | 제목, 심각도, 이유, 근거, 영향, 확인사항, 질문 |
| 누락 가능성 | 중요 항목이 문서에 보이지 않는가 | 계약 유형, 기대 항목, 검색 범위, 확인 질문 |
| 일정·날짜 | 언제 행동해야 하는가 | 날짜 종류, 원문 값, 계산 기준, 확정 여부 |
| 금액·지급 | 얼마를 언제 어떤 조건으로 지급하는가 | 금액 종류, 원문 값, 통화·세금·조건, 미확인 상태 |
| 권리·의무 | 누가 무엇을 언제까지 해야 하는가 | 주체, 행위, 기한, 조건, 위반 시 결과 |
| 책임 불균형 | 권리나 책임이 한쪽에 집중되는가 | 비교 대상, 근거 조항, 가능한 영향, 질문 |
| 질문 | 다음에 무엇을 물어봐야 하는가 | 대상, 목적, 근거, 실행 가능한 문장 |

## 4. 주의 조항 카탈로그

아래 항목은 탐지 후보이며 존재만으로 불리함이나 위법성을 확정하지 않는다.

| 항목 | 설명 | 가능한 사용자 영향 | 확인사항 | 질문 예시 | 전문가 검토 후보 기준 |
|---|---|---|---|---|---|
| 불리할 가능성이 있는 조항 | 의무·비용·권리 제한이 사용자에게 집중된 표현 | 예상하지 못한 부담 또는 권리 행사 제한 | 적용 주체, 예외, 상호성 | “이 의무가 양 당사자에게 동일하게 적용되나요?” | 금액이 크거나 핵심 권리 포기와 연결됨 |
| 모호한 표현 | 합리적, 상당한, 필요시처럼 판단 기준이 불명확함 | 이행 기준과 분쟁 대응 예측이 어려움 | 판단 주체, 객관 기준, 통지 절차 | “완료로 인정되는 객관적 기준을 적을 수 있나요?” | 핵심 의무·대금·해지 조건이 모호함 |
| 자동 갱신 | 별도 의사표시가 없으면 기간이 연장됨 | 원치 않는 계약·비용 지속 | 갱신 주기, 통보 방법, 마감일 | “갱신 거절은 언제까지 어떤 방식으로 해야 하나요?” | 장기·고액 계약 또는 해지 창구 불명확 |
| 일방 해지 | 한쪽만 넓은 해지권을 가짐 | 갑작스러운 종료와 회수 불가능한 비용 | 해지 사유, 사전 통지, 정산 | “동일한 해지권과 최소 통지 기간을 둘 수 있나요?” | 생계·주거·핵심 사업 지속에 영향 |
| 과도한 위약금 | 위반 정도와 무관하게 큰 금액을 부과할 수 있음 | 예상 손실보다 큰 금전 부담 | 산식, 상한, 감액·예외 | “위약금 산정 근거와 최대 한도는 무엇인가요?” | 고액·무제한 또는 중복 배상 가능성 |
| 책임 집중 | 손실과 대응 의무가 한쪽에 집중됨 | 통제할 수 없는 사유까지 부담 | 귀책 범위, 상호 책임, 면책 | “각 당사자의 귀책 범위에 맞게 책임을 나눌 수 있나요?” | 제3자 손해·광범위 indemnity 포함 |
| 무제한 손해배상 | 배상 범위나 상한이 없음 | 예측 불가능한 재정 노출 | 직접·간접 손해, 상한, 제외 | “배상 상한과 제외되는 간접 손해를 정할 수 있나요?” | 상한 없음, 영업손실 등 간접 손해 포함 |
| 과도한 비밀유지 | 대상·기간·예외가 지나치게 넓음 | 정상 업무·경력 활용 또는 신고 제한 | 공개 정보 예외, 기간, 허용 공개 | “이미 공개된 정보와 법적 신고는 예외인가요?” | 무기한·범위 무제한·과도한 제재 |
| 일방적 조건 변경 | 한쪽이 동의 없이 조건을 바꿀 수 있음 | 가격·업무·서비스 조건 예측 불가 | 변경 범위, 통지, 거부·해지권 | “중요 변경에는 사전 동의나 해지권이 있나요?” | 대금·핵심 의무를 일방 변경 가능 |
| 일방적 검수·지급 지연 | 검수 기준이나 기한 없이 지급을 미룰 수 있음 | 현금 흐름 악화와 무기한 재작업 | 검수 기준, 기한, 간주 승인 | “기한 내 의견이 없으면 승인된 것으로 볼 수 있나요?” | 지급 기한 없음 또는 반복 수정 무제한 |
| 과도한 원상복구 | 범위·노후도 구분 없이 복구 의무를 부과함 | 퇴거·종료 시 큰 비용 | 통상 손모, 사전 상태, 비용 산정 | “통상적인 사용 흔적은 복구 대상에서 제외되나요?” | 범위·상한 없음 또는 보증금과 직접 연계 |
| 비용 전가 가능성 | 세금·수수료·관리비 등 부담 주체가 포괄적임 | 숨은 비용 발생 | 항목별 부담자, 상한, 증빙 | “추가 비용의 항목과 최대 금액을 명시할 수 있나요?” | 포괄 문구로 제3자 비용까지 전가 가능 |

각 결과에는 위 일반 설명이 아니라 해당 결과 유형에 맞는 검증 가능한 근거, 그 문맥에서 가능한 영향과 구체적인 질문을 생성해야 한다.

### 근거 유형

| `evidence_type` | 의미 | 필수 근거 |
|---|---|---|
| `source_quote` | 특정 원문 구절을 직접 근거로 사용 | 최소 `source_quote`, 해당 `reference_id` |
| `document_scope_absence` | 검토한 문서 범위에서 기대 항목을 찾지 못함 | 검토 범위, 기대 항목, 검색 범위, 별지·첨부 미입력 가능성 |
| `cross_clause_comparison` | 둘 이상의 조항·문장을 비교해 차이 또는 불균형을 확인 | 비교한 두 개 이상의 reference |
| `derived_schedule` | 원문 날짜·기간으로 일정이나 마감을 계산 | 원문 reference, 계산 기준, 가정 |
| `derived_amount` | 원문 금액·비율로 금액을 계산 | 원문 reference, 계산 기준, 가정 |

`document_scope_absence`는 부재 사실의 확정이 아니라 확인된 범위에서 근거를 찾지 못했다는 의미다. `derived_schedule`과 `derived_amount`는 계산에 사용한 원문 값과 가정을 숨기지 않는다.

## 5. 계약 유형별 누락 가능성

누락 판정은 문서 전체와 분류 가능한 조항을 검색한 결과이며 `누락 확정`이 아니다. 계약 유형이 미확정이거나 추출 품질이 낮으면 누락 검사를 보류하고 사용자 검토 필요로 표시한다.

| 계약 유형 | 확인할 중요 항목 | 누락 가능성 표시 조건 | 사용자 질문 예시 |
|---|---|---|---|
| 근로계약서 | 근무 장소, 임금 지급일, 업무 범위, 휴일·휴가 | 확인된 전체 텍스트에서 해당 의미를 뒷받침하는 근거를 찾지 못함 | “근무 장소와 변경 가능 범위를 문서에 추가할 수 있나요?” |
| 주택 임대차 | 관리비 내역, 수선 책임, 보증금 반환, 중도 해지 | 유형 확인 후 관련 조항·표·별지에서 근거를 찾지 못함 | “관리비 항목과 정산 기준을 구체적으로 적을 수 있나요?” |
| 업무위탁 | 검수 기준, 수정 횟수, 결과물 권리, 지급 조건, 해지 정산 | 본문과 확인된 첨부 범위에 관련 근거가 없음 | “검수 기한과 수정 가능 횟수를 합의해 적을 수 있나요?” |

누락 가능성은 `evidence_type=document_scope_absence`를 사용한다. 검토한 문서·페이지·확인 텍스트 범위, `expected_term`, `searched_scope`와 별지·부속합의서 미입력 가능성을 남기며 가짜 `source_quote`를 생성하지 않는다.

## 6. 일정·날짜 구조

| 종류 | 구조화 값 | 계산·표시 원칙 |
|---|---|---|
| 계약 시작일·종료일 | 원문 날짜, 정규화 날짜 | 원문과 정규화 값을 함께 유지 |
| 자동 갱신일 | 종료일, 갱신 주기 | 계산식과 기준 timezone을 표시 |
| 해지 통보 마감일 | 기준일, 선행 기간 | “종료 30일 전” 등 원문 산식 보존 |
| 지급일·검수 기한 | 기준 사건, 기간, 영업일 여부 | 기준 사건이 불명확하면 날짜를 확정하지 않음 |
| 보증·비밀유지 기간 | 시작 사건, 기간, 종료 조건 | 계약 종료 후 기간인지 구분 |

일정 계산은 `evidence_type=derived_schedule`로 원문 reference, 기준 날짜, 산식과 가정을 연결한다. 상대 날짜의 기준이 없거나 날짜 해석이 여러 개면 `review_required`로 표시하며 임의 계산하지 않는다.

## 7. 금액·지급 구조

계약금, 중도금, 잔금, 월세, 보증금, 수수료, 위약금, 지연이자, 손해배상 상한과 세금 포함 여부를 후보로 추출한다.

- 원문 표기, 정규화 금액, 통화, 세금 포함 여부, 지급 조건과 기한을 분리한다.
- 퍼센트는 기준 금액과 산식을 함께 가져야 한다.
- 범위·상한·하한을 단일 금액으로 축약하지 않는다.
- 숫자 추출 실패, 통화 불명, 기준 금액 불명은 `null`과 경고로 표시하며 `0`으로 처리하지 않는다.
- 서로 다른 조항의 금액이 충돌하면 둘 다 근거와 함께 표시하고 사용자 확인을 요구한다.
- 계산 금액은 `evidence_type=derived_amount`로 원문 reference, 계산 기준과 가정을 보존한다.

## 8. 당사자별 권리·의무와 책임 불균형

각 의무는 다음 질문으로 구조화한다.

```text
누가 → 무엇을 → 언제까지 → 어떤 선행 조건에서 → 위반 시 어떤 결과를 부담하는가
```

주체가 “갑”, “을”, 임대인, 임차인 등으로만 표시되면 당사자 정의 조항과 연결한다. 주체·기한·조건 중 하나가 불명확하면 추정하지 않고 확인사항으로 남긴다.

책임 불균형은 다음 비교 신호를 사용한다.

- 같은 사건에 대해 한쪽만 해지·변경·검수 권한을 가짐
- 한쪽만 손해배상·면책·비용 부담을 가짐
- 의무 위반 결과가 당사자별로 현저히 다름
- 권리 행사 기간이나 통지 조건이 한쪽에만 불리함

불균형 표시는 협상·확인 우선순위이며 부당성이나 위법성의 확정 판단이 아니다.

둘 이상의 조항을 비교한 불균형 결과는 `evidence_type=cross_clause_comparison`으로 표시하고 비교한 reference를 두 개 이상 보존한다.

## 9. Free 안전 하한선

Free에서도 다음 결과는 반드시 표시한다.

- 탐지된 최우선 주의 항목
- 해지·자동 갱신·위약금·금액·기간 관련 핵심 결과
- 각 항목의 최소 근거
- 즉시 확인이 필요한 이유
- 최소 1개의 실행 질문
- 분석 실패·추출 실패·추가 확인 상태

사용자의 의사결정에 중대한 영향을 줄 수 있는 탐지 결과, 안전성 경고, 근거 부족과 추출 실패를 Free에서 의도적으로 숨기지 않는다. 유료 상품은 전체 지원 category, 상세 누락 검사, 전체 질문 목록, 상세 일정·금액·의무 구조와 이력·비교·알림·리포트·협업 기능을 확장한다. 유료 전환이 분석 정확도나 핵심 안전 정보의 잠금을 의미하지 않는다.

## 10. 질문 생성

질문은 결과마다 검증 가능한 근거와 목적을 가져야 하며 다음 세 유형으로 구분한다.

- 상대방 질문: 사실·기준·일정·비용을 명확히 확인
- 수정 협의 질문: 상호성, 상한, 기한, 예외를 문구로 합의
- 전문가 질문: 고액, 핵심 권리, 광범위 책임 등 전문 검토가 필요한 쟁점 확인

좋은 질문은 “이 조항이 위험한가요?”가 아니라 “제8조의 손해배상에 최대 한도와 간접 손해 제외를 둘 수 있나요?”처럼 대상과 원하는 확인 내용을 포함한다.

## 11. 계약 버전 비교 후보

향후 Plus·Business 후보 기능이며 현재 구현하지 않는다.

- 동일 계약 계보 안에서 버전과 비교 기준 버전을 지정한다.
- 조항 추가·삭제·수정과 금액·기간·책임 변경을 별도 change type으로 기록한다.
- 자동 갱신·해지 조건 변경과 중요도 변화를 원문 근거 쌍으로 보여준다.
- OCR·사용자 수정 차이는 계약 협상 변경과 구분한다.
- 비교 결과도 원문 양쪽 reference를 가져야 하며 법적 영향 확정값으로 표현하지 않는다.

## 12. 일정 알림 후보

향후 Plus·Business 후보 기능이며 인증, 소유권, timezone과 사용자 확인이 선행되어야 한다.

- 자동 갱신 거절 마감
- 해지 통보 마감
- 지급일
- 계약 종료일
- 보증 기간 만료

알림은 사용자가 확인한 일정만 활성화하고 기준 timezone, 발송 시점, 채널, 수신자와 상태를 기록한다. 추출값만으로 자동 발송하지 않으며 알림 실패가 법적 통지 이행을 의미하지 않음을 고지한다.

## 13. 품질 게이트

- 근거 reference·범위 검증 실패 결과 차단
- 전체 원문과 과도한 인용 차단
- 날짜·금액 파싱 실패를 0이나 임의 값으로 대체하지 않음
- 분석·추출 실패를 `위험 없음`으로 변환하지 않음
- 계약 유형 미확정 시 유형별 누락 단정 금지
- 금지된 법률 확정·보장 표현 차단
- 합성 평가셋과 전문가 rubric 검증 전 실제 분석 또는 결제 활성화 금지

## 14. 제외 범위

실제 Provider, 실제 계약서, OCR 구현, 사용자 계정, 결제, 알림 발송, 버전 비교 실행, 전문가 연결과 법률 자문은 포함하지 않는다.
