# v0.6 문서 입력·OCR 로드맵

## 1. 목적과 운영 원칙

PDF·이미지 입력을 안전하게 도입하고, 확인된 텍스트를 기존 개인정보 마스킹·조항 분리·분석 흐름에 연결하기 위한 단계별 계획이다. 각 Phase는 이전 단계의 보안·계약 게이트를 통과한 뒤 진행한다. 미구현 항목을 지원 완료로 표시하지 않는다.

공통 원칙:

- PDF·JPG·JPEG·PNG와 OCR은 P0, TXT는 테스트·보조 입력으로 유지한다.
- 텍스트 PDF는 직접 추출하고 스캔 PDF·이미지는 OCR 추출한다.
- 페이지 순서와 원문 위치를 보존한다.
- OCR 실패 또는 사용자 미확인 상태에서 분석하지 않는다.
- 원본 파일·파생 파일은 성공·실패 모두 삭제한다.
- 삭제 성공이 검증되기 전에는 `processed` 또는 분석 가능 상태로 전환하지 않으며, 실패 항목은 격리·운영 경보·재삭제 대상으로 둔다.
- 실제 데이터 대신 합성 fixture만 사용한다.
- 외부 OCR·분석 전송은 별도 고지·동의와 보안 승인이 필요하다.

### v0.6.2 고객 가치·상품 권한 병행 설계

v0.6.2는 OCR Phase 순서를 변경하지 않는 문서·도메인 설계 작업이다. Phase 6~7에서 구현할 근거 기반 분석의 목표 구조와 향후 상품 권한을 미리 정의하되 현재 지원 기능으로 표시하지 않는다.

- `source_quote`, `document_scope_absence`, `cross_clause_comparison`, `derived_schedule`, `derived_amount`를 근거 유형 후보로 구분하고, 이유·가능한 영향·확인사항과 질문을 목표 분석 결과의 필수 가치로 둔다.
- 누락 가능성, 일정·금액·의무와 책임 불균형은 사용자 확인이 필요한 구조화 결과로 설계한다.
- Free·One·Plus·Business의 기본 분석 품질은 동일하며 차이는 범위·사용량·보관·비교·알림·리포트·협업이다.
- entitlement, usage ledger, 결제·구독·소유권은 설계만 하며 인증·결제·webhook을 구현하지 않는다.
- Phase 3~6의 OCR·사용자 확인·페이지 근거 연결을 건너뛰어 실제 분석이나 상용 결제를 활성화하지 않는다.
- Phase 7 실제 Provider 전에는 합성 평가셋, 전문가 rubric, 개인정보·소유권·삭제와 출력 안전성 gate를 통과해야 한다.

가격, 무료 제공량, 상품별 보존 기간, 결제 Provider와 환불 정책은 검증 전 미결정 사항이다.

## 2. Phase 1 — 입력·보안·기술 결정

### 목표

지원 파일 계약, OCR 선택 기준, 원본 생명주기, 추출 결과 schema, 위협 모델과 후속 구현 순서를 검토 가능한 문서로 만든다.

### 선행 조건

- 기존 TXT API·frontend·테스트와 저장소 규칙 확인
- 현재 원본·조항·분석 데이터 경계 확인

### 구현 범위

- 문서 작성과 `docs/05-folder-structure.md` 갱신만 수행
- 수치의 초기 제안값과 최종 결정값 구분
- OCR engine·Provider를 확정하지 않고 스파이크 계획 정의

### 테스트 범위

- 문서 간 용어·상태·경로 일치 검토
- UTF-8, Markdown, trailing whitespace와 저장소 위생 검사
- 허용된 문서 외 변경 없음 확인

### 완료 기준

- A~G 문서와 폴더 구조가 작성됨
- 보안·호환성·미결정 사항과 책임·결정 시점이 명시됨
- 현재 단계가 `검토 필요`로 남음

### 제외 범위

- parser, decoder, OCR, API, DB, UI와 dependency 구현
- 실제 외부 호출과 실제 문서 사용

### 주요 위험

- 구현 검증 전 수치·상태가 확정 동작으로 오해될 수 있음
- 장기 기획 문서와 현재 SQLite·무인증 구현 간 차이

## 3. Phase 2 — 텍스트 PDF 직접 추출

- 상태: 구현 및 자동 검증 완료, 검토 필요

### 목표

텍스트 레이어 PDF를 안전하게 검증·직접 추출하고 페이지 순서와 원문 위치를 공통 결과 모델로 반환한다.

### 선행 조건

- 입력 계약과 원본 생명주기 승인
- parser 후보의 라이선스·보안·Windows/Linux 호환 스파이크
- API resource·동기/비동기 경계 결정

### 구현 범위

- PDF signature, 암호화·손상·페이지·객체·자원 제한
- 페이지별 텍스트 직접 추출; block/bbox는 현재 최소 계약에서 제외하고 Phase 5 전 재검토
- 텍스트 품질 판별과 OCR 필요 페이지 표시
- 전용 temp·무작위 파일명·정상/실패 cleanup
- 기존 TXT 경로 호환 유지

### 테스트 범위

- 합성 텍스트 PDF, 빈·암호·손상·0페이지·혼합 글꼴
- 페이지 순서, 한글·조항 번호·표, 빈 추출과 깨진 문자
- size/page/text 경계, parser 구조 오류와 cleanup 실패
- parser 예외·취소·삭제 실패와 로그 유출
- 기존 backend·frontend TXT 회귀

### 완료 기준

- 유효 텍스트 PDF가 페이지별 공통 schema로 직접 추출됨
- 추출 품질 부족 페이지가 성공으로 숨겨지지 않음
- 원본·파생 파일 삭제 검증 통과 후에만 `processed` 및 분석 가능 상태로 전환
- 기존 TXT API 계약 회귀 없음

v0.6.1 구현 결과는 pypdf 6.14.2, 별도 `/extractions` resource, 20 MiB·100페이지·추출 문자 제한과 요청 내 cleanup 경계를 사용한다. Backend 전체 테스트는 83 passed, 1 skipped이고 PDF extraction 관련 테스트는 30 passed, 1 skipped다. Ruff와 pip check도 통과했다. skip 1건은 Windows symlink 생성 권한 제약에 따른 조건부 테스트다. parser process 격리, 강제 timeout, orphan sweeper, bbox/layout과 실제 다양한 font corpus 평가는 후속 보안·품질 작업으로 남는다.

### 제외 범위

- 이미지 OCR, 스캔 PDF OCR, 사용자 편집, 실제 분석 Provider

### 주요 위험

- PDF parser 취약점과 압축 폭주
- 시각적 읽기 순서와 텍스트 레이어 순서 불일치

## 4. Phase 3 — JPG·JPEG·PNG OCR

- 상태: 안전 기반 구현 완료, 로컬 OCR 경로 설정형 연동, 검토 필요

### 목표

단일·다중 JPEG/PNG의 방향과 순서를 정규화하고 OCR 결과·경고를 공통 모델로 반환한다.

### 선행 조건

- ADR 스파이크로 1차 OCR 구현 후보 결정
- image decoder와 OCR worker 격리·자원 예산 결정
- 다중 이미지 순서 확인 UX/API 계약 결정

### 구현 범위

- extension·MIME·signature·decoder 검증
- EXIF 회전, 제한된 기울기·이미지 전처리
- 실제 픽셀·크기·장수·총량 제한
- 이미지별 문서 페이지 번호와 OCR block·경고
- 내부 OCR adapter와 timeout·cleanup

### 테스트 범위

- 합성 한글 이미지, JPG/JPEG alias, PNG alpha·회전·기울기
- 낮은 해상도·흐림·과다 노출·빈 이미지
- pixel bomb, 손상·trailing payload와 decoder crash 격리
- 여러 이미지 순서·중복·누락
- 한국어, 조항 번호와 표 인식 평가

### 완료 기준

- 지원 이미지가 순서 보존된 페이지 결과로 변환됨
- 낮은 품질·빈 OCR은 사용자 검토 필요 또는 분석 차단으로 표시됨
- OCR 엔진 세부가 공통 API에 누출되지 않음
- 모든 성공·실패 경로에서 원본·파생 파일 삭제가 검증되고, 삭제 실패 시 격리·운영 경보·재삭제 및 분석 차단이 수행됨

### 제외 범위

- 스캔 PDF page rendering, 외부 OCR 기본 사용, 사용자 편집

### 주요 위험

- 촬영 품질 편차와 잘못된 EXIF
- native OCR dependency의 OS별 결과 차이

v0.6.3은 `POST /extractions/images`, Pillow 12.3.0 decoder·정규화, engine 독립 adapter, 입력 순서와 identity 기반 cleanup을 구현했다.
v0.6.6에서는 `OCR_ADAPTER=local_korean`(또는 `local`, `tesseract`)일 때 로컬 OCR 경로로 전환되며, 엔진 미설치 시 fail-closed로 명확한 오류를 반환한다. 실제 OCR 품질, process sandbox, 강제 timeout, 안정적 기울기 보정과 Linux 배포 재현은 후속 작업이다.

## 5. Phase 4 — 스캔 PDF·다중 페이지 OCR

### 목표

PDF 페이지별로 직접 추출과 OCR을 안전하게 혼합하고, 다중 페이지의 순서·부분 실패·자원을 관리한다.

### 선행 조건

- Phase 2 PDF 검증과 Phase 3 OCR adapter 안정화
- worker queue·동시성·timeout·재시작 복구 결정
- 최종 크기·페이지·장수 제한을 위한 부하 결과

### 구현 범위

- 스캔 페이지의 제한된 해상도 렌더링
- 페이지별 direct/OCR 판별과 `mixed` 문서
- 페이지 순서를 보존하는 병렬 처리와 결과 조립
- 부분 실패를 문서 성공으로 숨기지 않는 상태
- 자원 제한, 취소, 재시도와 orphan cleanup

### 테스트 범위

- 텍스트·스캔 혼합 PDF, 1·최대·초과 페이지
- 병렬 완료 역순, 일부 OCR timeout, worker 재시작
- CPU·메모리·임시 디스크·처리 시간 부하
- 페이지 누락·중복·렌더링 실패와 cleanup

### 완료 기준

- 모든 페이지 method·순서·경고가 보존됨
- 한 페이지 실패도 사용자에게 드러나며 분석이 차단됨
- 부하 결과로 지원 제한을 확정하고 문서·frontend에 일치시킴
- orphan 정리와 재시작 검증 통과
- 원본·파생 파일 삭제 성공이 검증된 문서만 `processed` 또는 분석 가능 상태로 전환됨

### 제외 범위

- 사용자 텍스트 수정, 기존 분석 연결, 실제 외부 Provider

### 주요 위험

- 긴 처리 시간과 동기 HTTP timeout
- 병렬화로 인한 순서·중복·자원 폭주

## 6. Phase 5 — OCR 결과 확인·수정

### 목표

사용자가 페이지별 추출 텍스트와 경고를 확인하고 수정본을 명시적으로 확정한다.

### 선행 조건

- 공통 extraction schema와 revision API 승인
- 수정본 보존·삭제·소유권 정책 결정
- 사용자 확인 주체에 필요한 인증 시점 결정

### 구현 범위

- 페이지 이미지/텍스트 대응과 선택적 bbox 표시
- 원본 추출본 읽기 전용, 사용자 수정본 별도 revision
- 페이지 순서 확인과 품질 경고 표시
- stale revision·빈 수정본·문자 수 검증
- 전체 확인 전 분석 차단

### 테스트 범위

- keyboard·screen reader와 모바일 확인 흐름
- 수정·취소·재추출·stale revision·페이지 이동
- confidence 미제공·낮음과 경고의 비보장 표현
- 다른 문서 revision 혼합·무단 조회 방지
- 원문이 로그·analytics에 남지 않음

### 완료 기준

- 원본 추출본과 사용자 확인 텍스트가 분리됨
- 사용자가 모든 페이지와 순서를 확인해야 확정 가능
- 분석에는 최신 확인 텍스트만 전달됨
- OCR confidence를 정확성 보장으로 표시하지 않음

### 제외 범위

- 실제 위험 분석 품질, 법률 판단, 장기 원본 보관

### 주요 위험

- 사용자가 긴 문서를 형식적으로 확인할 가능성
- 수정본 저장으로 개인정보 보존 범위 증가

## 7. Phase 6 — 기존 마스킹·조항 분리·분석 연결

### 목표

사용자 확인 텍스트를 기존 개인정보 탐지·마스킹, 조항 분리와 합성 분석 pipeline에 연결하고 페이지 근거를 유지한다.

### 선행 조건

- Phase 5 확인 완료 계약
- 페이지→문자 범위→조항 reference 모델 결정
- 기존 DB schema 변경·migration과 rollback 계획

### 구현 범위

- 확인 텍스트 정규화와 페이지 경계 보존
- 마스킹 전 로그 차단과 기존 PII 검사 재사용
- 조항에 source page·block/bbox 후보 연결
- 기존 `clause_id`, `reference_id`, `source_hash` 검증 유지
- 합성 Provider로 end-to-end 흐름 검증

### 테스트 범위

- 페이지 경계에 걸친 조항, 표·번호·미분류 영역
- OCR 수정 전후 source mapping
- 개인정보 마스킹과 잔여 검사
- 결과와 다른 문서·페이지·조항 연결 차단
- OCR 실패·미확인 상태에서 분석 요청 차단

### 완료 기준

- 확인된 페이지 텍스트만 기존 pipeline에 진입
- 조항별 원문 페이지 근거를 추적 가능
- 기존 안전성·rollback·결과 연결 테스트 유지
- `위험 없음`, 분석 실패와 추출 실패를 구분

### 제외 범위

- 실제 외부 분석 품질, 법률적 최종 판단

### 주요 위험

- OCR 수정으로 offset·bbox 연결이 어긋남
- 기존 조항 DB에 마스킹 전 원문이 저장되는 범위

## 8. Phase 7 — 실제 Provider 및 근거 기반 결과

- 상태: 진행 중 (v0.6.7 사진 품질 보강은 API 경고 체계 정합성 선행 단계로 연결)

### 목표

승인된 실제 분석 Provider를 최소화된 확인·마스킹 텍스트에 연결하고 사용자에게 근거 중심의 결과를 제공한다.

### 선행 조건

- Provider 데이터 정책·보존·학습·처리 지역 심사
- 사용자 고지·동의와 인증·접근 제어
- 전문가 평가 rubric과 합성 평가셋
- timeout·재시도·비용·장애·교체 정책

### 구현 범위

- provider adapter와 구조화된 입력·출력
- 각 결과 유형에 맞는 검증 가능한 근거, 이유, 확인사항, 권장 질문
- reference·schema·개인정보 재생성·금지 표현 검증
- 근거 없는 결과와 문서 지시문 영향을 받은 출력 차단
- 합성 결과와 실제 분석 결과의 UI·상태 구분

### 테스트 범위

- 합성 평가셋의 근거 연결·누락·일관성
- 문서 내 지시문, schema 위반, 다른 조항 인용
- timeout·rate limit·인증·부분 실패·중복 요청
- 개인정보 전송 최소화와 출력 재생성 차단
- 전문가 검토의 정성·정량 평가

### 완료 기준

- 안전 검증을 통과한 근거 기반 결과만 저장·표시
- 결과 유형에 맞는 검증 가능한 근거·이유·확인사항·권장 질문이 모두 연결됨
- 실패를 위험 없음으로 표시하지 않음
- 사용자에게 법률 자문·최종 판단·안전 보장으로 표현하지 않음

### 제외 범위

- 모든 계약 유형·위험·개인정보 탐지 보장
- 자동 계약 체결 판단과 법률 자문

### 주요 위험

- Provider 환각·정책 변경·개인정보 유출
- 전문가 평가셋이 실제 사용 다양성을 대표하지 못함

## 9. Phase 8 — 품질·보안·삭제 검증

### 목표

공개 운영 후보 흐름을 공격 입력, 부하, 장애, 접근 제어, 삭제와 사용자 품질 관점에서 통합 검증한다.

### 선행 조건

- Phase 2~7 구현과 단계별 테스트 통과
- 운영 배포·DB·worker·backup·관측 후보 확정
- 공개 범위와 incident 대응 책임 확정

### 구현 범위

- 미비한 보안·관측·삭제·접근 통제 보완
- 지원 제한과 사용자 안내 최종화
- 운영 runbook, feature disable과 Provider 교체 절차
- 데이터 보존·사용자 삭제·backup 복구 정책 구현

### 테스트 범위

- parser/decoder fuzz와 adversarial corpus
- 최대치·동시성·장시간 soak·비용 제한
- worker kill, OS 재시작, network 장애와 orphan 정리
- temp·DB·backup·로그·APM의 민감정보 검사
- 권한 우회·문서 소유권·rate limit·abuse
- 접근성·모바일·다중 페이지 사용자 확인
- 전문가 rubric 기반 품질 검증

### 완료 기준

- 위협 모델의 high-impact 통제와 실패 처리가 검증됨
- 원본·파생 파일 삭제와 backup 제외가 장애 상황에서도 검증됨
- 공개 지원 형식·한도·한계가 코드·API·UI·문서에서 일치
- 미해결 고위험 이슈 없이 배포 승인 또는 명시적 보류 결정

### 제외 범위

- 검증되지 않은 새 파일 형식과 계약 유형의 동시 추가
- 무제한 보존·무제한 처리·정확성 보장

### 주요 위험

- 운영 환경과 테스트 환경 차이
- 장기 보존·지원 절차가 새로운 민감정보 사본을 만들 가능성
